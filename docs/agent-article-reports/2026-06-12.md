## Code Agent

### 1. Getting Better at Working With You: Compiling User Corrections into Runtime Enforcement for Coding Agents

- 链接：[arXiv](https://arxiv.org/abs/2606.13174) | [HTML](https://arxiv.org/html/2606.13174v1) | [PDF](https://arxiv.org/pdf/2606.13174)
- 证据来源：arXiv HTML / paper page
- 问题背景：交互式 Coding Agent 已经能编辑仓库、运行命令、检查失败，但它们并不会自然地“越用越顺手”。用户在一次会话里纠正过的偏好或约束，下一次仍可能被违反。论文把这个问题定义为 preference access 和 preference compliance 的缺口：记住了偏好，不等于执行前真的检查偏好。
- 核心方法：TRACE 将用户聊天中的纠正挖掘出来，改写为原子规则，再编译成 agent runtime 必须通过的检查项。直观地说，它不是把“以后记得清理临时文件”塞回 memory，而是把这句话变成任务结束前会自动执行的 verifier。TRACE 包含规则抽取、规则生命周期管理、编译执行、失败后重试等环节，因此用户偏好从软提示变成了硬约束。
- 创新点：现有 memory 系统通常停在“可检索偏好”，TRACE 则把偏好落到 runtime enforcement。对 Code Agent 特别关键，因为很多纠正都能被执行环境验证，例如文件是否残留、命令是否运行、是否在修改前询问、是否满足 workspace 终止条件。
- 实验与 benchmark：作者使用真实用户摩擦案例派生的任务做 access-compliance 诊断，并在 ClawArena coding-agent tasks 与 MemoryArena-derived memory-intensive tasks 上做模拟用户实验。摘要报告 Mem0 仍有 57.5% applicable preference checks 被违反；TRACE 在 ClawArena 上将 held-out preference violation 从 100.0% 降到 37.6%（ID），OOD 降到 2.0%；在 MemoryArena-derived tasks 上 ID violation 降到 60.5%，同时 task pass 与最强 memory baseline 持平或更高。
- 可转化价值：非常适合做 Code Agent 的长期偏好训练数据：把“用户纠正 -> 原子规则 -> verifier -> 失败重试轨迹”作为监督或 RL 样本。也可迁移到 Codex/IDE 场景，把项目规则、用户禁忌、提交前检查都编译成可审计 runtime checks。
- 局限：规则抽取和编译本身会出错；如果偏好不可验证或依赖隐性语境，硬约束可能过窄。需要冲突规则检测和用户确认机制。
- 阅读建议：今天 Code Agent 第一优先级。

### 4. IterCAD: An Iterative Multimodal Agent for Visually-Grounded CAD Generation and Editing

- 链接：[arXiv](https://arxiv.org/abs/2606.13368) | [HTML](https://arxiv.org/html/2606.13368v1) | [PDF](https://arxiv.org/pdf/2606.13368)
- 证据来源：arXiv HTML
- 问题背景：CAD 生成是一个典型的“代码生成 + 可执行环境 + 视觉/几何反馈”任务。已有方法多是一轮生成完整 CadQuery / build123d 程序，缺少人类工程师式的 generate-verify-refine 闭环，所以小的几何错误、拓扑错误或运行时错误很难被定位和修复。
- 核心方法：IterCAD 把任务建模为多轮 multimodal agent 与可执行 CAD sandbox 的交互，覆盖 Drawing-to-Code、Text-to-Code、Interactive Editing。系统先合成包含工业特征的多视图工程图、复杂代码编辑任务和交互轨迹，再用 progressive SFT 冷启动，随后进行 geometry-aware RL，并用 viable-prefix masking 提高代码可执行性和几何保真度。
- 创新点：它不是只让模型“看图写代码”，而是把 CAD 生成转成可执行沙箱中的闭环修复：生成程序、运行、观察错误和几何偏差、再编辑。CD-TR 曲线和 AUC-TR 指标也试图同时衡量 code validity 与 geometric precision，避免只统计幸存的可运行样本。
- 实验与 benchmark：论文提出 IterCAD-Bench，并在多个 CAD generation / editing benchmark 上比较可执行率、几何精度和闭环迭代能力。摘要声称在 code executability 和 geometric precision 上显著优于既有方法。
- 可转化价值：对 Code Agent 很有启发：很多代码任务也可以把“编译/测试失败 + 局部误差定位 + 可行前缀约束”转为 RL 信号，而不是只用最终 pass/fail。
- 局限：CAD 领域的几何反馈比通用软件测试更结构化，迁移到普通仓库修复时需要额外的错误定位器和测试覆盖建模。
- 阅读建议：适合关注 executable sandbox 与 closed-loop editing 的读者。

### 10. Structured Testbench Generation for LLM-Driven HDL Design and Verification-Oriented Data Curation

- 链接：[arXiv](https://arxiv.org/abs/2606.12983) | [HTML](https://arxiv.org/html/2606.12983v1) | [PDF](https://arxiv.org/pdf/2606.12983)
- 证据来源：arXiv HTML
- 问题背景：LLM 驱动 RTL/HDL 设计时，验证用 testbench 变成瓶颈。如果仍把 testbench 当成自由形式代码生成，输出随机、token 成本高、覆盖不足，导致错误设计被误判为通过。
- 核心方法：STG 利用 HDL 模块结构确定性生成 testbench：解析模块接口和状态结构，直接生成实例化、输出检查、报告框架，再针对组合逻辑、一般时序设计和 FSM-guided 设计生成高覆盖刺激。它把验证问题从“再让 LLM 写一段测试代码”改成“用结构分析生成稳定 verifier”。
- 创新点：论文把 verifier 作为 LLM 代码工作流的数据筛选器、直接验证工具和 test-time scaling oracle，而不是只作为后处理。这个思路对 Code Agent 很实用：当验证器可由程序结构导出时，优先用确定性工具，而不是让 LLM 反复猜测试。
- 实验与 benchmark：摘要报告 STG 作为直接验证工具比迭代式 LLM testbench flow 快 720x，编译成功率更高、覆盖更好、错误 DUT 的 false-pass 更少；作为数据清洗引擎，比 LLM-based filtering 快 11x、能耗低 127x；作为 test-time scaling oracle 可减少 14-47% 节点数。
- 可转化价值：适合迁移到一般 Code Agent 的“结构化 verifier 生成”：从接口、类型、状态机、配置文件中导出测试骨架，再让 LLM 只补充难以程序化的语义场景。
- 局限：HDL 的模块边界和模拟语义比较规整；通用软件仓库里副作用、外部服务和集成测试更复杂。
- 阅读建议：适合加入 code execution feedback / verification-oriented data curation 的 related work。

### 2. EurekAgent: Agent Environment Engineering is All You Need for Autonomous Scientific Discovery

- 链接：[arXiv](https://arxiv.org/abs/2606.13662) | [HTML](https://arxiv.org/html/2606.13662) | [PDF](https://arxiv.org/pdf/2606.13662)
- 证据来源：arXiv HTML
- 问题背景：自主科研 Agent 的能力越来越依赖可优化指标和执行环境。作者认为瓶颈正在从“设计更复杂的 agent workflow”转向“设计能让 agent 有效探索、记录、协作且不越界的环境”。
- 核心方法：EurekAgent 围绕 environment engineering 设计系统，而不是只改 agent prompt。四个维度是 permissions engineering（限制可执行权限与隔离评估）、artifact engineering（用文件系统和 Git 组织候选方案、日志、结果）、budget engineering（让探索感知成本边界）、human-in-the-loop engineering（让人类能低摩擦监督和介入）。系统流程包括 prepare、propose、implement，让多个 CLI agent 在受控环境中做度量驱动探索。
- 创新点：它把“环境”提升为 Agent 训练与部署的一等对象。对后训练而言，环境不仅提供工具，还定义奖励、证据、状态持久化、失败边界和协作协议。
- 实验与 benchmark：论文在数学、kernel engineering 和 machine learning engineering 任务上实验。HTML 摘要表显示 EurekAgent 在 26-circle packing、Erdos minimum overlap、first autocorrelation inequality、TriMul kernel 和 MLE-Bench 子集上取得新 SOTA 或第一名；其中 26-circle packing 新结果总 API 成本低于 11 美元，MLE-Bench 子集为 85.71%。
- 可转化价值：可以直接迁移为 Agent RL 环境设计原则：权限、artifact、预算、人类介入都应进入轨迹数据结构，而不是作为外部运维细节。
- 局限：强依赖任务可度量、可执行、可验证；开放式研究问题的奖励定义仍是难点。
- 阅读建议：今天 General Agent 第一优先级。

### 3. AgentBeats: Agentifying Agent Assessment for Openness, Standardization, and Reproducibility

- 链接：[arXiv](https://arxiv.org/abs/2606.13608) | [HTML](https://arxiv.org/html/2606.13608v1) | [PDF](https://arxiv.org/pdf/2606.13608)
- 证据来源：arXiv HTML
- 问题背景：Agent 评测目前碎片化严重：每个 benchmark 都有自己的 harness，每个 agent 也有自己的生产接口，导致评测和真实使用不一致，复现和横向比较困难。
- 核心方法：论文提出 Agentified Agent Assessment（AAA）：让 judge agents 通过标准协议评测 subject agents，任务管理用 A2A，工具访问用 MCP。传统 benchmark 需要 benchmark interface 和 agent interface 两套适配，AAA 试图只保留统一 agent 交互接口。AgentBeats 是 AAA 的实现，包含五种 operation modes，以适配开放性、隐私、复现和生产约束。
- 创新点：它把评测从固定 LLM-centric harness 改成 agent-to-agent assessment。这样评测逻辑和被测 agent 实现解耦，也更接近生产系统中的工具访问、任务交接和多 Agent 协作。
- 实验与 benchmark：论文包含两项研究：五个月开放竞赛，覆盖 12 类任务、298 个 judge agents 和 467 个 subject agents；以及 coding agents case study，显示 agentified evaluation 可以保持与公开记录的 fidelity，并暴露缺失的 head-to-head 结果。附录还讨论 OSWorld、MLE-bench、CyberGym、BrowseComp-Plus 的 agentification。
- 可转化价值：适合构造通用 Agent 评测平台：把 benchmark 包装成 judge agent，把被测系统保留为真实 agent runtime，而不是强制迁移到离线脚本。
- 局限：协议生态成熟度、judge agent 可信度和成本控制会影响落地；judge 本身也需要审计。
- 阅读建议：适合评测、benchmark infra、MCP/A2A 标准化方向。

### 5. Reward Modeling for Multi-Agent Orchestration

- 链接：[arXiv](https://arxiv.org/abs/2606.13598) | [PDF](https://arxiv.org/pdf/2606.13598)
- 证据来源：PDF / paper page
- 问题背景：多 Agent 系统真正难的是 orchestration：何时调用哪个 agent、如何组合中间产物、如何避免昂贵 rollout。现有 orchestrator 训练缺少监督，且常依赖完整子 agent 执行，成本高。
- 核心方法：OrchRM 使用多 Agent 执行中的 intermediate artifacts 构造 win-lose pair，用 Bradley-Terry 训练 reward model。它把奖励建模放在 orchestration level，而不是每次都重新跑子 agent，因此可以更便宜地指导 orchestrator 训练和 test-time scaling。
- 创新点：奖励不再只来自最终答案，也不需要人工标注，而是从编排过程里的中间证据、候选路径和执行产物中自监督构造偏好。这直接对应“如何训练 agent coordinator”的核心问题。
- 实验与 benchmark：摘要报告 token 使用效率最高提升 10x，MAS test-time scaling accuracy 最高提升 8%，并在数学推理、web-based QA、多跳推理多个领域迁移。
- 可转化价值：适合做多 Agent 后训练数据：保存中间 artifact、构造 orchestration preference、训练 coordinator reward model，然后在推理时用 reward-guided search 选择编排路径。
- 局限：PDF/摘要未在本次阅读中完整核验全部 benchmark 名称和数值表；需要进一步检查 pair 构造是否会偏向浅层 artifact 特征。
- 阅读建议：适合和“多 Agent 优势幻觉”一起读，一个提供训练方法，一个提醒成本和架构退化。

### 6. Agents-K1: Towards Agent-native Knowledge Orchestration

- 链接：[arXiv](https://arxiv.org/abs/2606.13669) | [HTML](https://arxiv.org/html/2606.13669v1) | [PDF](https://arxiv.org/pdf/2606.13669)
- 证据来源：arXiv HTML
- 问题背景：研究型 Agent 往往只把论文当成摘要、关键词和引用边，缺少 claims、evidence、mechanisms、method lineages 等结构。这样 agent 能检索到论文，却很难做可追溯的科学推理。
- 核心方法：Agents-K1 构建 agent-native scientific knowledge graph：多模态 parser 抽取实体、证据、引用和关系；4B 信息抽取模型用 GRPO 和 rule-based reward 训练；graphanything CLI 将 web search、多模态图检索和跨文档遍历统一成 agent 可调用工具。系统还支持 citation lineage、baseline retrieval、gap detection、idea grounding、novelty judging 等图操作。
- 创新点：它把知识组织从 RAG chunk 提升到可遍历、可验证、可组合的图接口，并显式面向研究 Agent 的 workflow，而不是面向静态搜索页面。
- 实验与 benchmark：论文处理 246 万篇六大学科论文构建 Scholar-KG，并释放百万论文子集；实验覆盖 scientific information extraction、KG construction、multi-hop scientific reasoning。HTML 中还说明 4B IE backbone 在部分任务上缩小到 Qwen3-32B 的约 8 倍规模差距，并在 NER 上超过它。
- 可转化价值：适合做长期 research memory：把论文证据、实验、方法谱系和引用关系变成 tool-callable graph，而不是把阅读记录压成一段摘要。
- 局限：系统很大，数据构建成本和抽取误差会影响可信度；知识图谱更新、冲突处理和版本追踪仍需进一步工程化。
- 阅读建议：适合科学 Agent、related work mining、idea grounding 方向。

### 7. TerraBench: Can Agents Reason Over Heterogeneous Earth-System Data?

- 链接：[arXiv](https://arxiv.org/abs/2606.13148) | [HTML](https://arxiv.org/html/2606.13148v1) | [PDF](https://arxiv.org/pdf/2606.13148)
- 证据来源：arXiv HTML
- 问题背景：地球系统分析需要同时处理栅格物理数据、卫星图像、GIS、模拟器输出和文档证据。天气/气候 foundation models 会预测但不会语言交互，LLM 会推理但不能直接操作高维科学数据。
- 核心方法：TerraBench 基于 TerraAgent，一个 ReAct-style executable framework，在 reasoning、tool calls 和 observations 之间交替，将 LLM 规划与环境检索、地理处理、模拟、artifact-backed computation 连接起来。任务覆盖 Fundamentals、Simulator-Grounded、Document-Grounded Verification 三个 track。
- 创新点：它把过程级 tool-use metrics 和 tolerance-aware numeric scoring 结合起来，不只看最终文字答案，还检查工具选择、参数、执行顺序和数值容差。
- 实验与 benchmark：TerraBench 包含 403 个 agentic tasks、8 个应用域、约 24,500 个 verified execution steps。HTML 摘要报告仅 50.9% proposed samples 通过人工审查，74.4% 需要多次 execution pass。top frontier model Claude Sonnet 4.6 只达到 59.2 ToolUseScore 和 22.9 Hit@tol，最强 open-weight Qwen3.5-35B 为 40.0 和 5.9；超过 84% 数值答案超出可接受误差。
- 可转化价值：虽然领域是 Earth-system，但评测设计可迁移：复杂 Agent benchmark 应同时记录工具调用过程、参数正确性、artifact provenance 和数值容差。
- 局限：领域特化明显；工具和模拟器质量决定任务代表性。
- 阅读建议：适合 benchmark 设计和工具调用过程评分。

### 8. The Illusion of Multi-Agent Advantage

- 链接：[arXiv](https://arxiv.org/abs/2606.13003) | [HTML](https://arxiv.org/html/2606.13003) | [PDF](https://arxiv.org/pdf/2606.13003)
- 证据来源：arXiv HTML
- 问题背景：多 Agent 系统常被假设天然优于单 Agent，因为它们可以分工、并行、隔离上下文和互相验证。但很多评测没有控制推理预算，MAS 看起来更强可能只是因为调用更多次模型。
- 核心方法：论文系统重评 automatic MAS，对照强单 Agent baseline CoT-SC，并引入 SMFR diagnostic benchmark，专门测试任务分解、上下文分离、并行潜力等 MAS 理论优势。作者还拆解 DyLAN、AFlow、MAS-Zero、ADAS、MaAS、MAS-Orchestra 等自动 MAS 架构的失效模式。
- 创新点：它不是提出新的 MAS，而是给出成本归一化的反证框架：多 Agent 是否真的带来功能性 utility，而不是架构膨胀。
- 实验与 benchmark：摘要报告 automatic MAS 在传统推理和 BrowseComp-Plus 等交互式多步 workflow 中持续弱于 CoT-SC，且成本最高可达 10x；expert-designed MAS 在合成诊断任务上明显优于自动生成架构。
- 可转化价值：训练多 Agent orchestrator 时应显式记录 marginal utility 和 cost。若一个子 agent 只复制主 agent 的思路，就应被 reward model 惩罚。
- 局限：聚焦 automatic MAS 和中心化协调范式；去中心化长期协作、工具专用 agent、生产约束下的收益仍需单独评估。
- 阅读建议：与 OrchRM 搭配读，避免把“更多 agent”误当成方法贡献。

### 9. ReSum: Synergizing LLM Reasoning and Summarization with Reinforcement Learning

- 链接：[arXiv](https://arxiv.org/abs/2606.13316) | [HTML](https://arxiv.org/html/2606.13316v1) | [PDF](https://arxiv.org/pdf/2606.13316)
- 证据来源：arXiv HTML
- 问题背景：RLVR 提升长程推理，但容易奖励过长 rollout，导致模型过度思考、重复验证、耗尽上下文。已有长上下文组织常依赖外部总结器或额外 workflow，模型自身不会主动管理推理轨迹。
- 核心方法：ReSum 让模型在 RL 过程中学习自我总结。若模型自发触发 summarization，训练时构造一个屏蔽 summarization phrase 的对比分支；若没有触发，则随机注入匹配分支。再用 summarization-aware advantage 比较两条轨迹，让模型学会何时总结会改善后续推理。
- 创新点：它把“压缩上下文”从外部工具变成模型策略的一部分，并把总结行为纳入 RL credit assignment。
- 实验与 benchmark：摘要报告平均性能提升 4%，rollout 长度降低 18.6%；论文实验覆盖不同 backbones、与现有 RL 方法组合、多模态场景，以及消融和效率分析。摘要未列出具体全部 benchmark 名称。
- 可转化价值：对长程 Agent 很直接：浏览器 Agent、Search Agent、Code Agent 都需要在轨迹中定期压缩状态，但压缩应由奖励学习控制，而不是固定步数触发。
- 局限：本次阅读未完整核验所有 benchmark 表格；总结行为也可能丢失低频但关键证据，需要 provenance 约束。
- 阅读建议：适合 long-context agent post-training。

## Search Agent

### 候补：From Verdict to Process: Agentic Reinforcement Learning for Multi-Stage Fact Verification

- 链接：[arXiv](https://arxiv.org/abs/2606.13262) | [HTML](https://arxiv.org/html/2606.13262v1) | [PDF](https://arxiv.org/pdf/2606.13262)
- 证据来源：arXiv HTML；未进入 Top 10
- 核心方法：ProFact 把事实核验建模为长程决策过程，统一策略负责 claim decomposition、evidence seeking、answer generation、verdict prediction，并用 process-aware rewards 给中间阶段提供学习信号。
- 可转化价值：适合 Search Agent：搜索不是单次 query，而是“分解要查什么 -> 检索证据 -> 生成中间答案 -> 汇总裁决”的轨迹优化问题。过程奖励可用于训练检索动作和证据组织。
- 局限：摘要未给出具体 benchmark 名称和关键数值，需进一步读实验表后再作为强 related work 引用。

### 候补：Rethinking RAG in Long Videos: What to Retrieve and How to Use It?

- 链接：[arXiv](https://arxiv.org/abs/2606.13141) | [PDF](https://arxiv.org/pdf/2606.13141)
- 证据来源：paper page / abstract；未进入 Top 10
- 核心方法：V-RAGBench 用 query、evidence chunk、answer 三元组把 retrieval 和 generation 解耦评估；CARVE 并行运行多种 modality-granularity retriever，并对每个 chunk 做自适应 reranking，让 chunk 级配置贯穿检索和生成。
- 可转化价值：Search Agent 的检索粒度不应只在 query 级决定；网页、视频、长文档中不同证据块可能需要不同视角和粒度。
- 局限：主要是长视频 RAG，迁移到网页搜索和文本证据需要重新定义 chunk 证据边界。

## 今日重点趋势

1. Agent 后训练越来越像“运行时系统训练”。TRACE、STG、IterCAD 都把 verifier、sandbox、runtime rule、执行反馈作为核心，不再只依赖语言监督。
2. 多 Agent 研究开始从“角色越多越好”转向“编排是否有边际收益”。OrchRM 试图训练 orchestration reward，而 Illusion of Multi-Agent Advantage 提醒必须控制成本和架构膨胀。
3. 评测基础设施正在标准化和过程化。AgentBeats 用 A2A/MCP 统一评测接口，TerraBench 记录 tool-use process metrics，二者都强调真实工具使用过程比最终文本答案更重要。
4. 长程 Agent 的核心资产是可追溯知识和可压缩轨迹。Agents-K1 关注论文知识图谱，ReSum 关注推理轨迹自总结，两者分别对应外部知识和内部状态。

## 潜在研究 Idea

1. 做一个“用户纠正到 verifier”的 Code Agent 数据集：收集真实交互中的纠正，自动提炼规则，生成终止前检查器，用执行结果作为偏好/奖励信号。
2. 将 EurekAgent 的 environment engineering 形式化为 Agent RL 环境 schema：permissions、artifacts、budget、human intervention 都进入 trajectory log。
3. 训练 orchestration reward model 时加入 cost-normalized marginal utility：每个子 agent 的输出必须相对单 Agent baseline 提供新增证据或行动价值。
4. 为 Search Agent 设计 process-aware reward：claim decomposition、query generation、evidence selection、citation provenance、final answer 每一阶段都有可验证反馈。
5. 把 Agents-K1 式知识图谱和 ReSum 式轨迹压缩结合：外部文献用 graph provenance，内部执行轨迹用 learned summary，并在最终答案中强制引用证据节点。
