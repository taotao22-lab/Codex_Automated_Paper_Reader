## General Agent

### 1. Agentic Reasoning for Large Language Models: A Survey

- 方向：General Agent
- 评分：4.395 / 5
- 来源：OpenReview, Under review for TMLR
- 链接：https://openreview.net/forum?id=2HIxxXIq2u
- 证据来源：PDF，阅读 abstract、introduction、taxonomy/table of contents、benchmark/future agenda 相关部分。

**问题背景**：这篇综述试图把“会推理的 LLM”和“能行动的 Agent”之间的断层系统化。论文指出，数学、代码等封闭环境中的推理能力不能直接等价于开放环境中的规划、工具调用、搜索、记忆和多轮反馈能力；Agent 的关键不只是多想几步，而是把思考、动作、观测和修正组织成可持续的交互循环。

**核心方法**：文章把 agentic reasoning 分成三层：foundational agentic reasoning 覆盖单 Agent 的 planning、tool use、search；self-evolving agentic reasoning 关注反馈、记忆、自适应和经验累积；collective multi-agent reasoning 关注角色协作、共享知识和多 Agent 协调。它还把优化方式拆成 in-context reasoning 和 post-training reasoning：前者是在推理时用 workflow、搜索、编排和工具接口放大能力；后者是用 SFT/RL 把成功的规划、工具调用和搜索策略内化到模型参数里。

**创新点**：它的价值不是提出新算法，而是给 Agent 后训练提供了清晰坐标系：哪些能力只是 prompt/workflow 层面的临时编排，哪些能力应该通过轨迹数据、奖励和偏好反馈进入模型训练。对你的研究来说，这个框架适合作为 related work 的总入口，尤其是区分 tool-use optimization、agentic search、memory control、self-evolving planning 和 multi-agent post-training。

**实验与 benchmark**：作为 survey，论文没有自己的实验；它综述了 tool use、search、memory/planning、多 Agent，以及科学、机器人、医疗、autonomous research、数学等应用 benchmark。需要注意它覆盖面很广，直接可复用的实验协议需要再回到被综述的原论文。

**可迁移价值**：可以用它设计一套 Agent 训练数据 taxonomy：把每条轨迹标成 planning、tool-use、search、memory、verification、collaboration 等模块，再决定哪些轨迹适合 SFT，哪些适合 RL 或偏好优化。

**局限**：综述很长且范围宽，工程细节和数值实验不是重点；如果直接照搬 taxonomy，可能会稀释 Search Agent 和 Code Agent 的具体闭环差异。

**阅读建议**：优先读 introduction、Section 3 的 planning/tool/search、Section 4 的 self-evolving，以及 benchmark/future agenda；不要从头线性读 120 多页。

### 2. Runtime Compliance Verification for AI Agents

- 方向：General Agent / Tool-use Safety
- 评分：4.365 / 5
- 来源：arXiv, cs.SE
- 链接：https://arxiv.org/abs/2606.19242
- 证据来源：PDF，阅读 abstract、introduction、framework、monitor/attack driver、evaluation。

**问题背景**：真实 Agent 会通过工具调用处理个人数据、执行订单、查询客户信息或输出敏感内容。传统红队或静态 prompt 审查只能在上线前发现部分问题，不能保证运行时每一次工具调用和模型输出都符合规则。论文把这个问题转化为“对 Agent 执行轨迹做在线合规验证”。

**核心方法**：C-Trace 把自然语言对话、工具参数、用户同意、删除请求等事件转成 typed event trace，再把 GDPR 中 consent、purpose limitation、data minimization、right to erasure 四类原则写成一阶谓词。运行时 monitor 夹在 agent loop 和模型/工具 API 中间，每一步都检查当前 trace prefix；一旦发现某个工具调用或输出会违反策略，就拒绝该动作。它还用独立 attack driver 生成攻击对话，避免用同一套 predicate 同时当防御器和评估器。

**创新点**：这篇文章的直觉很有用：合规、隐私、权限不是 prompt 里的一段提醒，而是可验证 runtime 上的 trace predicate。它把工具调用从“模型说了算”搬到“事件流可审计、策略可执行”的控制平面里，这对 Browser/Code Agent 的文件权限、网络权限、凭据访问同样可迁移。

**实验与 benchmark**：论文在四个 GDPR-reframed case studies 上评估，覆盖 retail、customer support、healthcare、banking/KYC 风格工具流程。攻击包括 DSPy 生成提示和 HarmBench、AdvBench、DAN 等红队语料的 verbatim prompt。关键结果包括：在 10% extractor noise 下 attack success rate 不超过 12%，false positive 不超过 16%；完美抽取时 ASR 为 0%。作者还报告 Python/MonPoly 在 400 条 trace、Rego/OPA 在 720 条 trace 上与策略库一致。

**可迁移价值**：可把 Agent 训练环境的奖励拆成两层：任务成功奖励和 runtime policy 约束。对于 Code Agent，可以把文件写入范围、命令执行、依赖安装、网络访问也表示成 trace predicate，用 monitor 的拒绝动作生成负样本或 RL penalty。

**局限**：GDPR predicate 是人工建模的子集，extractor 噪声会影响实际部署；论文没有证明复杂政策都能低成本形式化。

**阅读建议**：如果关注 Agent 安全和工具权限，重点读 event model、predicate design、monitor loop；法规背景可快速扫过。

## Search Agent

### 3. HANSEL: Extracting Breadcrumbs from Web Agent Trajectories for Interactive Verification

- 方向：Web Agent / Search Agent
- 评分：4.715 / 5
- 来源：arXiv, cs.HC
- 链接：https://arxiv.org/abs/2606.18671
- 证据来源：PDF，阅读 abstract、introduction、system design、technical evaluation、user study。

**问题背景**：Web Agent 可以搜索商品、比较选项、下单或完成长程网页任务，但用户很难验证它到底有没有做对。完整轨迹太长，截图和链接太散，LLM 生成摘要又可能不忠实。HANSEL 的切入点是：验证不应该是“读日志”，而应该是“带状态回到关键网页证据上交互检查”。

**核心方法**：HANSEL 接收 Web Agent 的动作轨迹，抽取直接支撑最终答案的 evidence pages 和 snippets，并把它们展示成可导航、可交互的页面视图。关键是保留页面状态，例如搜索词、过滤器、排序方式、滚动位置和点击后的页面状态；如果最终答案无法追溯到任何访问过的页面，系统会明确标出证据缺口。直观说，它把 agent trace 压缩成一组“可复现的网页证据断点”，用户可以直接回到这些断点验证结果。

**创新点**：相比轨迹日志、source link 或自然语言解释，HANSEL 把 evidence 做成可操作对象。这个方向很适合作为 Browser Agent 训练的 reward/evaluation 设计：不仅要求答案正确，还要求答案能映射回最小充分证据页面，并且页面状态可恢复。

**实验与 benchmark**：技术评估使用 45 个来自 AssistantBench 和 Online-Mind2Web 的任务，轨迹来自 HAL/brower-use 等 Web Agent 运行。HANSEL 识别 evidence pages 的 precision 为 83.7%、recall 为 88.8%，同时减少 61.6% 的轨迹体量。用户研究包含 14 名参与者，相比标准 Agent 界面，HANSEL 显著降低任务完成时间和感知努力，并提高可用性、验证便利性和错误识别评分。

**可迁移价值**：可以在 Search Agent 训练数据中加入 `evidence_page`, `state_snapshot`, `support_span`, `unsupported_answer_flag` 字段，让模型学习“答复必须能落回可检查证据”。这比只训练 citation 更接近真实网页操作。

**局限**：论文主要研究 post-hoc verification，不直接提升 Agent 自身成功率；复杂动态网页的状态恢复成本和隐私边界还需要更多工程验证。

**阅读建议**：重点读 design goals、evidence extraction、evaluation setup；用户研究可作为交互评测参考。

### 4. Decoupling Search from Reasoning: A Vendor-Agnostic Grounding Architecture for LLM Agents

- 方向：Search Agent / Grounding Runtime
- 评分：4.635 / 5
- 来源：arXiv, cs.AI / cs.CL / cs.IR / cs.MA
- 链接：https://arxiv.org/abs/2606.18947
- 证据来源：PDF，阅读 abstract、introduction、architecture、evaluation。

**问题背景**：生产 Agent 越来越依赖实时搜索，但许多模型原生搜索把检索策略、供应商选择、证据注入、成本、延迟和生成风格全部封装在一个 provider 边界里。开发者很难检查检索深度、缓存、fallback、证据渲染和输出格式是否稳定。

**核心方法**：论文提出 Decoupled Search Grounding (DSG)，把 search grounding 从 reasoning model 中拆出来，放到 MCP-compatible gateway 后面。这个 gateway 负责 provider routing、source-aware context rendering、fallback、retrieval-depth control、exact/semantic caching。LLM 只看到结构化工具结果，而不是被 provider-native search 隐式改变输出行为。直观说，它把搜索从“模型的隐藏能力”变成“可配置、可缓存、可审计的工具边界”。

**创新点**：DSG 的新意在接口边界，而不是某个检索模型。它强调 Search Agent 的可控性：同一个 reasoning model 可以接不同 search backend，同一套 search layer 可以服务不同模型，并且成本、延迟、缓存命中、输出格式都可测。

**实验与 benchmark**：作者在五个 frontier models 上比较 native search、DSG 和 no-search，使用 SimpleQA、FreshQA、HotpotQA，以及 DoorDash 风格的 e-commerce Query Intent Understanding workload。关键结果：SimpleQA 上 DSG 接近 native accuracy，86.1% vs 87.7%，但搜索成本低 91%；warm-cache hit rate 达 99.4%，延迟降低 68%；生产 QIU 工作负载中搜索成本降低超过 98%。FreshQA 上 native search 仍在 recency-sensitive 问题上有优势。

**可迁移价值**：对你的 Search Agent 系统设计很直接：把检索轨迹、检索深度、provider、缓存命中、证据渲染作为训练和评测字段，而不是只把搜索结果拼进 prompt。也可以把“Search-Induced Verbosity”作为输出契约违反的负样本。

**局限**：部分生产数据不可公开复现；native search 的能力会随 provider 变化，结论需要持续重测。

**阅读建议**：优先读 architecture figure、cache-gated provider execution、SimpleQA/FreshQA/HotpotQA 结果表。

### 5. EpiLoop: A Diagnostic Protocol for Measuring Epistemic Lock-in in Iterative Search Agents

- 方向：Search Agent Evaluation
- 评分：4.565 / 5
- 来源：OpenReview, Under review for TMLR
- 链接：https://openreview.net/forum?id=hMptycsA60
- 证据来源：PDF，阅读 abstract、definition、metrics、probe、evaluation。

**问题背景**：迭代式搜索 Agent 会提出假设、检索证据、更新上下文，再决定是否继续搜索。常规评测看最终准确率、召回率或 citation 质量，但忽略了搜索过程中的信念状态是否被早期假设锁死。EpiLoop 定义的 epistemic lock-in 是一种路径依赖失败：早期假设塑造后续查询、证据选择、置信度和停止决策，导致更多检索反而强化错误。

**核心方法**：论文提出一套诊断协议，用可观察 trajectory proxy 近似 agent 的 epistemic state，包括 leading hypothesis、confidence、source coverage、closure judgment、refutation intent。核心指标是 Hypothesis Change Rate (HCR)，即相邻步骤中 leading hypothesis 是否发生变化。EpiLoop 本身不是新模型或新 retriever，而是轻量 inference-time probe/controller：追踪五类状态，并据此选择下一轮 search intent、审计停止条件。

**创新点**：它把 Search Agent 可靠性从“是否搜到答案”推进到“搜索循环是否真的支持修正”。这对构造 RL reward 很关键：可以奖励 source independence、contradiction integration、合理 closure，惩罚 redundancy support 和 premature closure。

**实验与 benchmark**：主实验是 40-task HoVer-derived study。EpiLoop 与 equal-budget forced extra search 的最终准确率相同，75.0% vs 75.0%，但 HCR 更低，0.083 vs 0.121。作者还做了 cross-model validation 和 FEVER/SciFact/VitaminC 外部检查，发现 HCR 下降较一致，但准确率提升依赖数据集和 parser。

**可迁移价值**：可把 EpiLoop 的状态字段加到 Search Agent trajectory schema：`hypothesis`, `confidence`, `source_groups`, `contradiction_seen`, `closure_reason`。训练时不只优化答案，还优化“能否在反证出现时改变问题表述和答案”。

**局限**：论文明确说 EpiLoop 不证明比额外搜索更准确，它的主要价值是诊断；HCR 也不是越低越好，需要结合证据覆盖和最终正确性解释。

**阅读建议**：重点读 epistemic lock-in definition、Table 1、metrics 和实验 setup；这篇适合做评测协议，不适合作为单独 Agent 架构照搬。

### 6. MCompassRAG: Topic Metadata as a Semantic Compass for Paragraph-Level Retrieval

- 方向：Search Agent / RAG
- 评分：4.105 / 5
- 来源：arXiv, cs.CL / cs.IR
- 链接：https://arxiv.org/abs/2606.18508
- 证据来源：PDF，阅读 abstract、introduction、method、benchmark summary。

**问题背景**：RAG 系统在 chunk 粒度上常有两难：细粒度 chunk 精准但搜索空间大、延迟高；粗粒度 chunk 数量少但 embedding 混合多个主题，语义噪声更大。对 deep research agent 来说，这个问题更明显，因为它要在大规模异构语料中反复检索。

**核心方法**：MCompassRAG 不继续把 chunk 切得更细，也不在检索后用昂贵 LLM 过滤，而是给粗粒度 chunk 加 topic metadata。它用 topic modeling encoder 把文档、chunk 和主题信号映射到同一语义空间，再用 LLM-teacher distillation 训练轻量 retriever，使 query 可以先找到相关 topic，再用 topic-aware representation 检索大 chunk。直观说，它给每个大段落挂上“语义指南针”，让粗 chunk 仍然可精准搜索。

**创新点**：它强调把 expensive teacher 的检索判断蒸馏到 inference-time 无 LLM 调用的 retriever 中。对 Search Agent 来说，这种“离线用强模型标注主题，在线用轻模型快速检索”的设计适合降低多轮检索成本。

**实验与 benchmark**：论文报告六个 complex retrieval benchmarks，图中明确包含 HotpotQA 和 DRBench；整体 information efficiency 平均提升 8.24%，相比强 LLM-based RAG baseline 延迟降低超过 5 倍。benchmark 测的是多跳/复杂问答下 evidence retrieval 的效率和答案 F1/IE 权衡。

**可迁移价值**：可以把 query planning 与 corpus topic metadata 结合：Search Agent 先预测需要的主题方向，再选择检索子空间，而不是每一步都从全库 dense search 开始。

**局限**：效果依赖 topic metadata 质量和 teacher distillation；对动态网页、代码仓库这类结构化语料还需要重新验证。

**阅读建议**：重点读 metadata selection/abstraction 和 distillation 部分，和 SproutRAG 一起对比“粗 chunk 加语义指南针”与“细 chunk 建层次树”。

### 7. SproutRAG: Attention-Guided Tree Search with Progressive Embeddings for Long-Document RAG

- 方向：Search Agent / Long-document RAG
- 评分：4.015 / 5
- 来源：arXiv, cs.CL / cs.IR
- 链接：https://arxiv.org/abs/2606.18381
- 证据来源：PDF，阅读 abstract、introduction、method、retrieval algorithm、evaluation summary。

**问题背景**：长文档 RAG 既要细粒度命中事实，又要保留跨句、跨段的上下文。固定 chunk、单层 expansion 或摘要式层级索引都会在成本、上下文连贯性和信息损失之间取舍。

**核心方法**：SproutRAG 把文档切成句子级 chunk，用小语言模型的 inter-sentence attention 学习哪些句子语义相关，再自底向上构建二叉 attention tree。内部节点使用 progressive embeddings 表示更大语义单元。检索时用 hierarchical beam search 从树上收集不同层级的候选，再 rerank 选择 top-k 证据。直观说，它不是让每个 chunk 孤立竞争，而是让检索沿着一棵可学习的文档结构树探索。

**创新点**：相对 RAPTOR、LLM-guided chunking 或 summary hierarchy，SproutRAG 的结构来自模型 attention，并且在线检索不需要额外 LLM 调用或摘要压缩。它适合需要多粒度 evidence 的 research/search agent。

**实验与 benchmark**：作者报告四个 benchmark，覆盖 scientific、legal、open-domain settings，平均 information efficiency 比最强 baseline 提升 6.1%。论文强调 dispersed evidence 场景收益更明显，benchmark 测的是长文档中跨句/跨段证据召回与上下文效率。

**可迁移价值**：可以把 Search Agent 的文档记忆从 flat vector store 升级成多粒度树结构；对于论文阅读、法律文档、长网页，Agent 可以在叶子节点和内部节点之间切换检索粒度。

**局限**：需要额外索引构建和训练；attention 是否稳定表示语义结构，在不同 backbone 和代码/网页语料上还需验证。

**阅读建议**：重点读 Figure 2、Algorithm 1、attention-guided indexing；如果只做短网页检索，优先级低于 DSG/HANSEL/EpiLoop。

## Code Agent

### 8. SWE-Future: Forecast-Conditioned Data Synthesis for Future-Oriented Software Engineering Agents

- 方向：Code Agent
- 评分：4.765 / 5
- 来源：arXiv, cs.SE / cs.AI
- 链接：https://arxiv.org/abs/2606.18733
- 证据来源：PDF，阅读 abstract、introduction、method stages、related work、evaluation summary。

**问题背景**：很多 coding-agent benchmark 直接复刻公开 GitHub issue 和 PR，真实但容易被预训练、微调、合成数据或 leaderboard tuning 污染。纯合成任务能避开历史复刻，却容易脱离真实仓库需求。SWE-Future 试图走中间路线：不复刻未来 PR，而是用过去仓库证据预测未来会出现的任务家族，再用这些预测来合成任务。

**核心方法**：方法分四步。第一，在时间点 T0 为每个仓库构建 evidence bundle，只包含 T0 前的 issue、PR、label 和文本。第二，预测未来 feature/enhancement、bugfix、refactor 任务家族。第三，在 retrospective study 中冻结预测，再用 T0 后真实 PR 元数据验证这些预测是否命中未来工作。第四，在任务生成快照 Tgen 上，用经过验证的 forecast family 作为 conditioning signal 合成 coding-agent 任务，而不直接 replay 后来的 PR。直观说，它让任务生成器先学习“这个仓库接下来可能往哪里演化”，再生成可执行任务。

**创新点**：它把 Code Agent 数据构造的时间边界讲清楚了：未来 PR 只用于验证 forecast，不用于直接生成任务内容。这比历史 issue replay 更抗污染，也比完全 synthetic 更贴近仓库真实压力。

**实验与 benchmark**：80 个仓库的 retrospective forecasting study 中，forecaster 在主 semantic matching metric 下达到 58.1% future-work relevance。随后作者用 validated forecast families 合成 200 个 coding-agent tasks，覆盖 61 个仓库，并强调任务来自 task-generation snapshot，不复刻验证用的未来 PR。论文也对比 SWE-Bench、SWE-Bench++、FEA-Bench、SWE-Factory 等历史 GitHub 任务构造方法。

**可迁移价值**：非常适合作为 Code Agent 后训练数据管线参考。可以把仓库演化预测器作为任务采样器：从活跃仓库的历史证据预测未来变更族，再生成 issue、测试和 fail-to-pass patch，从而降低 benchmark leakage。

**局限**：58.1% future-work relevance 说明预测仍有相当噪声；生成任务的真实可维护性和人类接受度还需要更大规模验证。

**阅读建议**：优先读 method 的 information boundary 和 validation procedure；这是今天最值得加入 related work 的 Code Agent 数据构造论文。

### 9. Below the Reliability Floor: Recovering True Success from Judge-Gated Loops

- 方向：Code Agent / Agent Evaluation
- 评分：4.56 / 5
- 来源：OpenReview, Under review for TMLR
- 链接：https://openreview.net/forum?id=J2Yg9vJcYb
- 证据来源：PDF，阅读 abstract、introduction、measurement model、empirical results。

**问题背景**：很多 Agent 闭环把 LLM judge 放在控制回路里：Agent 生成答案，judge 判 PASS/FAIL，FAIL 就重试，直到 PASS 或预算耗尽。表面上这像自我修正，实际上如果 judge 有噪声，retry-until-PASS 会不断抽样直到 judge 漏判，导致报告的 pass rate 系统性高估真实成功率。

**核心方法**：论文把 cap-K retry gate 建模成二分类器，定义 sensitivity、specificity 和 Youden index J。报告成功率 `R(K)` 不是真实成功率 `π`，而是由 gate 的分类误差混合出来的 biased estimator。作者提出用小规模 gold-labeled calibration set 估计 gate 的 class-conditional rates，并用 PPI++ 恢复真实成功率，同时建议报告 J 作为 gate informativeness。

**创新点**：它把 Agent 自评闭环从“多试几次更可靠”反过来解释为“更多 retry 可能让测量更不可靠”。对 Code Agent 特别重要，因为 LLM judge 不能运行候选代码时，最容易把错误 patch 放行。

**实验与 benchmark**：作者在 GSM8K、MATH 和 code generation 的 44 个 capable-agent loops 上做实验，观察到系统性膨胀：median slip +0.16；代码任务最严重，真实 0.74 被报告为 0.98。closed-form law 对 slip 的预测 pooled r=0.95。恢复真实成功率时，PPI++ 的 mean recovery MAE 为 0.050，优于 Rogan-Gladen 的 0.149；gold 稀缺时 0.081 vs 0.241。非可验证安全/摘要质量任务上，PPI++ 也把 MAE 从 naive 0.165 降到 0.043。

**可迁移价值**：Code Agent 评测不能只看 judge-gated pass rate。应在闭环中保留 K=1 原始测量、少量可执行/人工 gold labels、judge instability、多次判定概率，并用校准估计真实成功率。

**局限**：校准依赖 gold labels 和 judge 分布稳定；PPI++ 解决的是测量偏差，不直接提升 Agent 输出质量。

**阅读建议**：重点读 Eq. 3、Youden index、Section 5 的 code 结果；这篇适合作为所有 judge-in-the-loop Agent 论文的评测警示引用。

### 10. From Specification to Execution: AI Assisted Scientific Workflow Management

- 方向：Code Agent / Tool-use Workflow
- 评分：3.985 / 5
- 来源：arXiv, cs.SE / cs.AI / cs.DC
- 链接：https://arxiv.org/abs/2606.18425
- 证据来源：PDF，阅读 abstract、introduction、workflow generation、debugging agent、execution integration。

**问题背景**：科学 workflow 通常是带依赖关系的复杂 DAG，需要可复现执行、分布式调度、数据管理和故障恢复。LLM 可以从自然语言生成 workflow，但直接生成代码容易不透明、难验证，也难接入成熟 workflow management systems。

**核心方法**：论文提出 specification-driven workflow generation：先把用户意图、数据描述、设计和实现分离成结构化 specification，再生成可执行 workflow。它还加入 LLM-based debugging agent，通过日志分析定位多层失败并提出修复；执行层集成 Pegasus WMS，并通过 MCP layer 提供 workflow submission、monitoring、control 的统一接口。直观说，它把“让 LLM 写脚本”升级为“LLM 先写可检查规格，再连接 workflow runtime 执行和调试”。

**创新点**：对 Code Agent 的启发是 specification-first 和 runtime integration。模型不应直接吐出一堆不可控脚本，而应先生成可验证 workflow spec，再由执行系统负责调度、 provenance、监控和失败恢复。

**实验与 benchmark**：论文用 medical imaging 的 federated learning workflow 做 case study，原因是该流程包含 parallel、iterative、dependency-intensive 结构。作者报告系统能生成和执行 thousands of jobs 的大规模 workflow，降低 debugging effort，并让非专家构建带专家设计模式的 workflow。没有像 SWE-bench 那样的公开通用 benchmark，更多是系统/案例验证。

**可迁移价值**：可把 Code Agent 任务拆成 `spec -> plan/DAG -> implementation -> execution -> logs -> repair`，并把 Pegasus/MCP 类 runtime 抽象为工具层。对构造代码执行闭环和多步骤工程任务数据很有参考价值。

**局限**：评估集中在一个科学 workflow case study，领域迁移到普通软件仓库、CLI 工具链和测试修复还需要重做。

**阅读建议**：读 specification phase 和 debugging agent；医学应用细节不是重点。

## 今日重点趋势

1. **Agent 评测正在从 final answer 转向 trajectory measurement**。HANSEL 关注可交互证据页，EpiLoop 关注搜索过程中的信念修正，Reliability Floor 关注 judge-gated pass rate 的统计偏差。这三篇合起来说明：Agent 能力评测必须保留轨迹、证据、状态和 judge calibration。
2. **Search runtime 正在工程化为可控边界**。DSG 把 search grounding 移到 MCP-compatible gateway；MCompassRAG 和 SproutRAG 分别从 topic metadata 和 attention tree 改造检索结构。这说明 Search Agent 的核心不只是“接一个搜索 API”，而是 query planning、provider control、evidence rendering、cache 和多粒度索引。
3. **Code Agent 数据构造开始处理时间污染问题**。SWE-Future 用 forecast-conditioned synthesis 避免直接 replay 未来 PR，是比传统 SWE-bench-style 数据更适合后训练的新方向。
4. **运行时约束会成为 Agent 后训练的重要信号**。C-Trace 说明工具调用、数据访问和策略合规可以转成 trace predicate；这些 predicate 可以作为环境拒绝、奖励扣分或训练样本标签。

## 潜在研究 idea

- **Forecast-conditioned SWE 数据管线**：复现 SWE-Future 的时间边界设计，但加入可执行测试生成、仓库环境构建和 fail-to-pass patch 验证，把 forecast family 转成可训练的 Code Agent trajectory。
- **Search Agent trajectory schema**：综合 HANSEL、EpiLoop、DSG，定义统一字段：query intent、provider、retrieval depth、source group、evidence page state、hypothesis、confidence、closure reason、unsupported flag。
- **Judge-gated loop 校准层**：在所有 self-correction / verifier / critic Agent benchmark 中报告 K=1、K>1、judge instability、Youden J、PPI++ recovered success，避免把 judge slip 当作能力提升。
- **Runtime predicate reward**：把 C-Trace 的 trace predicate 思路迁移到 Code Agent 文件系统权限、命令执行权限、网络访问和测试执行，构造“任务成功但策略违规”的 hard negatives。
- **多粒度 RAG for paper/code reading**：把 SproutRAG 的树结构和 MCompassRAG 的 topic metadata 结合到论文阅读或仓库阅读 Agent：先按主题缩小检索，再在文档树上找细粒度证据。
