## General Agent

### 1. ACRL: Adaptive Control of Training-Inference Discrepancy for Stable Reinforcement Learning
- 链接：[OpenReview](https://openreview.net/forum?id=8sBZSgAkAH) | [PDF](https://openreview.net/pdf/7da20f22f50e7e2cc577af346ad2abdfce36600e.pdf)
- 证据来源：PDF
- 核心方法：论文把 RL 后训练里经常被工程化掩盖的问题单独拿出来研究，即训练引擎和推理引擎分离、推理侧低精度量化带来的 policy mismatch。作者提出 ACRL，用序列级统计量持续监控并自适应控制 training-inference discrepancy，而不是只靠重要性采样做事后校正。
- 为什么重要：这比“换一个 RL 算法”更贴近真实大模型训练系统，因为问题直接来自 `vLLM/SGLang + FSDP/Megatron` 这类分离式栈。对 Agent post-training 来说，这条线直接关系到 rollout 稳定性、探索质量和训练吞吐。
- 实验信息：文中在 GSM8K 和数学数据上报告，FP8 推理下普通方案会失败，而 ACRL 能稳定训练；在给出的表格里，Qwen2.5-3B 的结果接近或略超 BF16 基线，也优于文中的 IS 修复。
- 可转化实验：可以把“训练-推理差异监控”接到 tool-use / browser / code agent 的 rollout pipeline 上，把差异值与 reward 波动、trajectory 退化、测试通过率下降联动记录。
- 局限：论文主要验证的是 RL 稳定性，不是直接面向复杂 Agent 环境；任务也更偏数学推理，还没证明它会自动改善长链工具使用。
- 阅读建议：优先看 introduction、ACRL 的 discrepancy control 定义、以及 5.1/5.4 对训练崩溃和效率的分析。

### 2. LossVal: Data Valuation using Weighted Loss Functions
- 链接：[OpenReview](https://openreview.net/forum?id=TIGUqDjVMZ) | [PDF](https://openreview.net/pdf/df5a74da8d5a5dfa1415b090a4207aedfff8020a.pdf)
- 证据来源：PDF
- 核心方法：在标准 loss 里加入可学习权重，让模型在训练中同时学任务和样本重要性，避免反复重训式 data valuation 的高成本。
- 为什么重要：你关心 Agent 训练数据构造，这篇虽然不是 Agent 论文，但它提供了一个更轻量的“样本值估计”思路，可以迁移到 trajectory 清洗、bad trace 移除、reward model 训练集筛噪。
- 实验信息：作者在 OpenDataVal 基准、CIFAR-10 和 BERT/ResNet-50 场景里验证，结论是能在 noisy-label / mixed-noise 识别上取得有竞争力的结果，同时计算成本低于重训型方案。
- 可转化实验：给 code-agent 轨迹、search trace、tool-call 序列加 LossVal 风格权重头，筛掉“高损失但低价值”的坏样本和“低损失但无增益”的重复样本。
- 局限：没有直接处理序列决策、长期 credit assignment 或多轮交互数据。

### 3. PULSE: Projection-based Unlearning via Linear Speedy Entropy Maximization
- 链接：[OpenReview](https://openreview.net/forum?id=38wtCnBTyv) | [PDF](https://openreview.net/pdf/6b4142b92f840f771f7f71e6be0391ac31f4658a.pdf)
- 证据来源：PDF
- 核心方法：只通过一个轻量 projection matrix 做遗忘，保持 backbone 和 classifier 冻结；目标是在 retain-data-free 条件下提升 forget set 熵，同时尽量保住 retain utility。
- 为什么重要：如果以后做 Agent 轨迹删除、版权/隐私数据回撤、在线数据回收，这类“局部、快速、黑盒兼容”的遗忘机制有工程价值。
- 实验信息：文中在 CIFAR-10/100、CIFARSuper20、ImageNet-1k 上报告，forget accuracy 可压到接近 0，且速度通常比强基线快 2 到 10 倍。
- 局限：当前验证对象是图像分类，不是语言模型或 Agent 记忆模块；迁移到长上下文或外部记忆系统还需要新设计。

## Search Agent

### 4. Mental Damage: Caption Poisoning Attacks on Retrieval-Augmented Text-to-Music Generation
- 链接：[arXiv](https://arxiv.org/abs/2605.30365) | [PDF](https://arxiv.org/pdf/2605.30365)
- 证据来源：PDF
- 核心方法：论文不改用户查询、不改 retriever、不改生成器，而是污染 RAG 知识库里的 captions，让系统在 prompt augmentation 阶段检索到“看似相关、实则带毒”的描述，再把生成结果悄悄往攻击者目标意图上拉。
- 为什么重要：虽然场景是 text-to-music，但威胁模型非常适合 Search Agent。它直接揭示了“检索命中不等于证据可信”，而且 poisoning 可以同时保持高层语义锚点和底层操控信号。
- 实验信息：作者在 MusicCaps + CLAP retriever + MusicGen pipeline 上验证，攻击后生成结果明显更靠近攻击者目标，同时表面上仍与原查询保持一定对齐。
- 可转化实验：对 Search Agent 的检索缓存、候选证据库、网页摘要库做 poisoning stress test；单独评估 anchor preservation、retrieval plausibility、steering strength 三个维度。
- 局限：不是网页搜索，也不是文本问答；更多提供的是 RAG 完整性安全视角，而不是查询规划方法。

### 5. Diversity Matters: Revisiting Test-Time Compute in Vision-Language Models
- 链接：[arXiv](https://arxiv.org/abs/2605.30713) | [PDF](https://arxiv.org/pdf/2605.30713)
- 证据来源：PDF
- 核心方法：系统研究 test-time compute 在 VLM 上何时有效。结论不是“多采样一定更好”，而是要看预测之间有没有足够 diversity；在单模型场景里，feature heuristic 效果差，多数投票提升有限；多模型场景下作者提出 ETTC，用 predictive entropy 选更可信答案。
- 为什么重要：Search Agent 做多候选答案验证、证据重排或多模型协同时，也会遇到“样本多但彼此高度相关”的问题。这篇给了一个判断 TTC 是否值得上的视角：先看依赖性，再谈投票。
- 实验信息：论文覆盖 7 个 VLM、6 个 benchmark，并证明 ETTC 在多模型情形下通常优于 majority voting，部分情况下还能超过最强单模型。
- 局限：任务是视觉推理，不是搜索；迁移价值主要在 ensemble 和验证策略，而不是直接方法复用。

### 6. Reproducibility study of FACTER: Fairness-Aware Conformal Thresholding and Prompt Engineering
- 链接：[OpenReview](https://openreview.net/forum?id=vX5mPaLdeQ) | [PDF](https://openreview.net/pdf/f33bcf868401db9c33284336c664581124b5cd59.pdf)
- 证据来源：PDF
- 核心方法：复现实验表明，FACTER 这类 “conformal thresholding + iterative prompt engineering” 的后处理框架确实能压低 fairness violation，但 utility 会明显下降，而且开源实现和论文表述之间存在若干不一致。
- 为什么重要：Search Agent/推荐式检索系统以后如果也做后处理约束，这篇提醒你必须把 `validity`、映射误差和评测假设写清楚，否则表面指标可能掩盖真实退化。
- 局限：推荐系统取向很重，和 Agent 主线距离较远；更多是“复现与评测卫生”启发。

## Code Agent

### 7. Kernel Foundry: A Diagnosis-driven Evolutionary Kernel Optimizer with Multi-Experts
- 链接：[arXiv](https://arxiv.org/abs/2605.30359) | [PDF](https://arxiv.org/pdf/2605.30359)
- 证据来源：PDF
- 核心方法：用 expert-guided initialization + retrieval-augmented hints 启动候选代码，再进入 multi-island evolutionary search。每轮不是只看 pass/fail，而是抽取编译错误、输出不匹配、memory-bound / latency-bound / instruction-bound 等诊断信号，喂回经验库继续修。
- 为什么重要：这非常像高质量 Code Agent 的理想闭环，不是一次性生成补丁，而是“生成-运行-诊断-检索经验-再生成”的持续优化循环。
- 实验信息：论文在 KernelBench Level 1/2 上评估，声称正确率最高可到 100%，并在多个类别上优于直接 LLM 生成和自动 Triton 转换基线。
- 可转化实验：把经验库存成 patch 模式、失败模式和修复 hint；把 multi-island search 改成多候选 patch 并行分支；把 cheating validator 思路迁移到测试规避检测。
- 局限：目标是 GPU kernel 优化，任务结构比仓库级 issue resolution 更窄。

### 8. Neuroforger: certified violation witnesses for smart contracts verification via LLMs
- 链接：[arXiv](https://arxiv.org/abs/2605.31389) | [PDF](https://arxiv.org/pdf/2605.31389)
- 证据来源：PDF
- 核心方法：先把自然语言属性翻译成扩展 Solidity 的形式化规格，再让 LLM 为抽象类型变量找具体赋值，最后用类型检查和具体执行验证生成出的 violation witness / PoC 是否真的成立。
- 为什么重要：它把 LLM 放在“搜索候选反例”的位置，而不是“直接给最终判断”的位置，特别适合 Code Agent 的 verifier 设计。也就是先让模型提候选，再让程序执行去判真伪。
- 实验信息：作者在一个 17 个 Bank 合约版本、27 条属性的验证数据集上评估，展示了可以生成经执行验证的 counterexample。
- 可转化实验：把自然语言 issue 描述转为可执行测试模板；让模型填充触发条件，再用沙箱执行器验证失败轨迹是否真实存在。
- 局限：目前仍有人工校验步骤，且实验规模不算大。

### 9. Practical Algebraic Stepping with Scoped Filters
- 链接：[arXiv](https://arxiv.org/abs/2605.31517) | [PDF](https://arxiv.org/pdf/2605.31517)
- 证据来源：PDF
- 核心方法：为 stepper 增加带词法作用域的 pattern-based filters，让用户控制哪些归约步骤显示、隐藏、只停一次或全程展开；并在 Hazel 里实现，还做了 Agda 机械化证明。
- 为什么重要：Code Agent 现在常见的问题不是“没有 trace”，而是 trace 太长、信噪比太差。这篇的过滤式 step trace 对调试代理、教学式解释器、程序执行可视化很有启发。
- 实验信息：作者把系统部署到大学 PL 课程，观察到学生会自然使用 stepper，但复杂 filter 仍需要教学支撑。
- 局限：更偏程序理解与教学工具，不直接解决仓库级修复。

### 10. A Virtual Processor brings back the Free Lunch
- 链接：[arXiv](https://arxiv.org/abs/2605.30507) | [PDF](https://arxiv.org/pdf/2605.30507)
- 证据来源：PDF
- 核心方法：把异构硬件抽象成一个 self-optimizing virtual processor，用分布式、局部决策的 execution segments 在运行时决定 task placement、kernel preparation 和 data movement，而不是依赖中心化调度器。
- 为什么重要：这篇不是 Code Agent 论文，但它把“agent-like runtime mechanism”落实到一个真实执行系统里，对你思考 agent runtime、调度分解和局部反馈组织方式有参考价值。
- 局限：更偏数值数组程序和运行时系统，和 LLM agent 训练的连接需要你自己补系统层映射。

## 今日重点趋势

1. 真正高价值的条目都在强调“执行反馈结构化”，而不是单次生成能力本身。`ACRL` 控训练/推理差异，`Kernel Foundry` 做诊断驱动演化，`Neuroforger` 用类型检查和具体执行验证反例。
2. Search/RAG 方向今天缺少正面构建论文，反而是 `Mental Damage` 这种攻击论文更有启发，因为它逼着你把证据完整性、检索污染和 prompt augmentation 风险单独建模。
3. 数据治理和后训练基础设施是第二条暗线。`LossVal` 讲样本价值，`PULSE` 讲快速遗忘，二者都不是 Agent 论文，但都能直接服务 Agent 训练数据生命周期。

## 潜在研究 Idea

1. 把 `ACRL` 的 discrepancy control 扩展到 tool-use RL：不仅监控 token policy 偏差，还监控工具调用分布、环境状态分布和测试通过率分布。
2. 把 `Kernel Foundry` 的经验库做成 Code Agent 的失败模式记忆：编译失败、单测失败、回归失败、性能退化分别维护可检索修复 hint。
3. 把 `Neuroforger` 的“自然语言属性 -> 可执行验证模板 -> LLM 搜反例 -> 执行器验真”范式迁移到 issue reproduction，自动产出最小失败脚本。
4. 给 Search Agent 构建 `poisoned evidence benchmark`：分开测试检索命中率、证据可信度、答案稳健性和引用可验证性。
