### 1. AgentLens: Production-Assessed Trajectory Reviews for Coding Agent Evaluation

- 类别：Code Agent / Trajectory Evaluation
- 链接：[2607.06624](https://arxiv.org/abs/2607.06624)
- 综合评分：4.9

**问题背景与核心方法**：这篇论文解决的是 coding agent 在生产使用中“只看最终是否通过”不够的问题。真实用户看到的是完整轨迹：Agent 是否理解指令、如何调用工具、是否验证自己的改动、失败后如何恢复、与用户沟通是否清楚。AgentLens 把一次交互轨迹作为评测对象，结合可形式化检查的 verifier、LLM 写出的 trajectory review 和 side-by-side 比较，让每次运行不仅有分数，还有解释为什么得这个分数的证据。

**创新点**：它不是再造一个单点 pass/fail benchmark，而是把生产 coding agent 的交互质量变成可夜间回归的评测管线。这个思路对后训练很重要：轨迹 review 可以提供“工具使用差、验证不足、恢复失败、沟通含糊”等细粒度标签，比最终测试结果更适合训练过程奖励。

**实验与 benchmark**：PDF 正文显示 AgentLens 围绕生产式交互 coding workflows 构造任务和模拟用户；每条轨迹有 formal verification、五类 LLM judge 指标和 aggregate quality index。论文还描述 nightly evaluation pipeline，用于比较 agent 版本、检测产品回归，并讨论与 SWE-bench、Terminal-Bench、tau-bench、LLM-as-judge benchmark 的关系。

**可迁移价值**：适合借鉴为 Code Agent 轨迹数据清洗器：把原始日志转成用户请求、工具调用、文件编辑、验证尝试、恢复行为和最终状态，再由 judge 生成带证据指针的 review。局限是 LLM judge 本身需要校准，且生产任务 fold 的代表性决定了最终分数是否可信。

**阅读建议**：优先读 task set、trajectory review protocol、metric design、side-by-side review 和 nightly pipeline。证据来源：PDF 正文抽取。

### 2. Mining Workflow Graphs for Black-Box Boundary Testing of Conversational LLM Agents

- 类别：General Agent / Black-box Agent Testing
- 链接：[2607.06873](https://arxiv.org/abs/2607.06873)
- 综合评分：4.77

**问题背景与核心方法**：很多对话 Agent 的危险失败发生在状态边界，例如未确认就取消订单、未完成身份校验就泄露信息。普通提示测试很难到达这些边界，因为边界藏在多轮流程后面。AgentEval 先通过黑盒交互挖掘 conversational workflow graph，再用图结构枚举 guard、prerequisite 和 boundary target，重放到目标状态后施加扰动，并只基于对话回合判断测试是否通过。

**创新点**：它把 Agent 测试从“随机问一些坏问题”升级为“先建行为图，再按结构覆盖边界”。这对 tool-use 和 browser agent 很关键，因为真实风险通常不是单轮 jailbreak，而是多步流程中状态机约束被绕过。

**实验与 benchmark**：论文在四个 `tau^3-bench` agents 上评测，并与有源码访问权限的 privileged white-box auditor 对比。PDF 摘要和正文显示 AgentEval 每个 agent 覆盖 23 到 38 个 distinct boundaries；在 airline agent 上 graph-guided generation 生成 23 个 distinct boundaries，而 prompt-only baseline 为 12，并把 duplicate rate 从 0.56 降到 0.26，false-alarm 近零。

**可迁移价值**：可以迁移为 General Agent 安全评测模块：把成功和失败轨迹都抽成流程图，围绕每个状态边界生成可执行测试，再把“是否绕过边界”作为奖励或安全过滤信号。局限是图挖掘质量受探索预算影响，且黑盒 judge 对复杂业务语义可能仍会误判。

**阅读建议**：重点读 workflow graph construction、boundary target selection、privileged auditor benchmark 和 ablation。证据来源：PDF 正文抽取。

### 3. Mitigating Taint-Style Vulnerabilities in MCP Servers via Security-Aware Tool Descriptions

- 类别：General Agent / MCP Tool Security
- 链接：[2607.07461](https://arxiv.org/abs/2607.07461)
- 综合评分：4.61

**问题背景与核心方法**：MCP 让 LLM Agent 能动态调用外部工具，但也把工具描述、参数和服务器实现连接成新的攻击面。论文先系统分析 MCP server 漏洞，指出 taint-style vulnerabilities 占很大比例，传统修复往往需要理解服务端代码并做较大改动。SPELLSMITH 的思路是不用只依赖代码 patch，而是分析 MCP server 暴露的高风险能力、tool description 和参数语义，生成 tool-level risk profile，再通过 Description Enhancement Module 把行为约束写入 MCP description，并用 Self-Reflection Module 迭代检查输出。

**创新点**：它把工具安全的一部分控制面从服务端代码移到 Agent 可见的工具元数据上。直观说，它不是等 Agent 把危险参数传给工具后再拦截，而是在工具选择和参数构造前就让模型看到风险边界，并用自反思检查是否越界。

**实验与 benchmark**：PDF 正文显示论文分析了 NVD 中已披露的 MCP server 漏洞，报告 taint-style 漏洞占 81.13%。实验构造了 792 个恶意 prompt，用于利用 taint-style vulnerabilities；SPELLSMITH 将攻击成功率降到 0.13%，并声称相比 code-level mitigations 具有更低修复成本和更强泛化性。

**可迁移价值**：适合借鉴到工具路由和 MCP 安全训练：把工具描述改写、风险 profile、参数级 taint 标签和自反思结果做成训练样本。局限是 text-based mitigation 更像 shield，不等同于根除漏洞；如果模型忽略 description 或工具实现本身无校验，仍需要运行时 sandbox 和服务端修复。

**阅读建议**：重点读漏洞分类、risk profile 构造、Description Enhancement、Self-Reflection 和 792 prompt benchmark。证据来源：PDF 正文抽取。

### 4. What Makes a Good Bug Report for an AI Agent?

- 类别：Code Agent / Issue Resolution Input Design
- 链接：[2607.07593](https://arxiv.org/abs/2607.07593)
- 综合评分：4.74

**问题背景与核心方法**：自动程序修复 Agent 仍然从人类写的 bug report 开始，但“对人类有用”的报告特征不一定对 Agent 有用。论文做了两组分析：第一，在 433 个 SWE-bench Verified issue 和 87 个 repair agent 的结果上，用统计模型分析 27 类 bug-report feature 与修复成功的关系；第二，在 SWE-bench Pro 上对 2 个模型做 17 种 problem-statement mutation，控制内容变化来测 solve rate。

**创新点**：这篇论文把 issue text 当成 Code Agent 的输入分布来研究，而不是只把模型当变量。它说明可执行、可定位、结构化的信息比自然语言“写得详细”更重要，这直接影响后训练数据应该如何写任务说明。

**实验与 benchmark**：PDF 显示 433 个 SWE-bench Verified 实例对应 87 个 agent，共 37,671 个 instance-agent outcomes，整体 solve rate 为 47.7%。显著正相关特征包括 fix suggestion、repository source code、reproduction script 和 localization info；其中 fix suggestion odds ratio 为 3.61，repository source code 为 2.82，reproduction script 为 2.52。SWE-bench Pro ablation 显示 Qwen 与 Gemma 对定位线索和 expected behavior 都敏感，结构变化本身也会降低 solve rate。

**可迁移价值**：可直接用于构造 Code Agent 任务说明模板：优先提供可运行复现脚本、相关源码片段、定位线索和期望行为；避免只堆长文本。局限是 SWE-bench 任务可能存在公开 leaderboard 和训练污染风险，统计相关性也不等同于因果；论文用第二组 ablation 部分缓解了这个问题。

**阅读建议**：重点读 feature taxonomy、mixed-effects regression、SWE-bench Pro mutation 设计和 Qwen/Gemma 错误分析。证据来源：PDF 正文抽取。

### 5. Specification Grounding Drives Test Effectiveness for LLM Code

- 类别：Code Agent / Test Generation
- 链接：[2607.06636](https://arxiv.org/abs/2607.06636)
- 综合评分：4.59

**问题背景与核心方法**：LLM 写代码常在边界条件和非法输入上失败。常见做法是让模型自己写测试再修复，但论文追问：收益到底来自“测试数量”，还是来自“测试被外部规格约束”？作者固定 tester、test budget 和 repair loop，只改一行提示：tester 是否拿到 spec checklist。结果显示，spec grounding 才是主要驱动。

**创新点**：论文把“生成更多测试”与“测试从哪里获得期望值”清晰分开。它证明不是让模型多想几个 edge case 就够了，关键是把外部规格条款转成测试预期，避免模型自己发明需求。

**实验与 benchmark**：PDF 显示主 benchmark 包含 26 个小型 Python 任务，其中 18 个是 specification-completeness tasks。SPEC 在 Claude Haiku 4.5、Sonnet 4.6、Opus 4.8 上相对强 baseline 都提升 +38 percentage points，在 held-out set 上提升 +36 points。消融中，tester 拿到 plain paragraph spec 能抓到 27/30 个 bug；只让它做无 spec 的 coverage plan 只抓到 2/30。跨供应商复现中 GPT-5.3-codex 提升 +28，Gemini 3.5 Flash 提升 +19；false-alarm rate 从 33% 降到 0%。

**可迁移价值**：可用于 Code Agent 训练闭环：把 issue 里的隐式需求先抽成规格条款，再用条款驱动测试生成和修复。局限是论文主要覆盖 specification-completeness defects，对复杂算法逻辑、系统状态和并发错误的泛化还需要额外验证。

**阅读建议**：重点读 SPEC/FREE+ 对照、spec content vs enumeration 消融、property-based baseline 和 AlphaCodium-style flow 对比。证据来源：PDF 正文抽取。

### 6. Rethinking Code Performance Benchmarks for LLMs

- 类别：Code Agent / Performance Benchmark Reliability
- 链接：[2607.07619](https://arxiv.org/abs/2607.07619)
- 综合评分：4.46

**问题背景与核心方法**：很多 LLM code performance benchmark 声称能评估“模型是否写出更高效代码”，但原测试集未必能暴露运行时差异。论文重审 EffiBench、Enamel、EvalPerf、Mercury 四个 benchmark，对 1,538 个任务进行更严格测量：每个任务运行 30 次，并用统计检验比较 canonical solution 和 benchmark-provided performant implementation。

**创新点**：它把性能 benchmark 的问题定位到测试充分性，而不是只怪评测统计方法。随后提出一个三 Agent 框架：生成 performance-oriented tests、诊断失败原因、修复仍保持功能正确且能暴露性能差异的 deterministic tests。

**实验与 benchmark**：PDF 显示 1,538 个任务中，原 benchmark-provided tests 下只有 94 个任务，即 6.11%，能证明 performant implementation 显著更快。四个 benchmark 的任务数分别是 EffiBench 1,000、Enamel 164、EvalPerf 118、Mercury 256。对 1,345 个原测试未显示显著差异的任务，DeepSeek-V3.1 和 GPT-4o 生成的测试分别让 323 个任务（24.01%）和 342 个任务（25.43%）暴露出显著性能改进。

**可迁移价值**：适合转化为 Code Agent 的“性能修复验证器”：修复不是写出看似更快的代码，而是自动生成能稳定暴露差异的测试输入。局限是函数级 Python 性能任务与仓库级性能优化还有距离，且性能测试生成本身可能过拟合特定输入分布。

**阅读建议**：重点读四个 benchmark 的重测协议、多 Agent test generation/diagnosis/repair loop 和 RQ3 结果。证据来源：PDF 正文抽取。

### 7. Thinking More, Harnessing Better: State Machine Guided Harness Automatic Generation with Project Digestion and Workflow Decomposition

- 类别：Code Agent / Fuzz Harness Generation
- 链接：[2607.07007](https://arxiv.org/abs/2607.07007)
- 综合评分：4.25

**问题背景与核心方法**：灰盒 fuzzing 需要高质量 harness，但 LLM 一次性生成 harness 容易幻觉 API、遗漏初始化或覆盖很浅。SynapseFlow 先分析 C 项目源码，构造 Structural Flow Graph，抽取 coherent Function Triplets；再用四阶段生成流程和 staged rollback algorithm 逐步生成、验证、回滚和修复 harness。

**创新点**：它把“让 LLM 写 harness”拆成可验证工作流：先做 dataflow-aware function aggregation，再让每个阶段通过轻量语法/语义检查，失败时回滚到合适阶段，而不是让模型重写一大段代码。这与 Code Agent 的仓库级任务分解和执行反馈闭环高度一致。

**实验与 benchmark**：PDF 显示 SynapseFlow 在 25 个真实开源 C 项目上评测，对比 OSS-Fuzz-Gen、CKGFuzzer、PromeFuzz。它在 branch coverage 上分别达到 3.07x、1.71x、4.26x 提升；bug detection rate 分别达到 1.77x、1.51x、1.36x 提升。论文还报告发现 7 个此前未报告 bug，摘要写 5 个分配 CVE，正文局部写 4 个分配 CVE；日报不强行统一该冲突，只记录“7 个新 bug，CVE 数在抽取片段中不一致”。

**可迁移价值**：可迁移为 Code Agent 的 staged execution policy：把复杂代码生成任务拆成结构分析、局部生成、编译验证、回滚修复。局限是当前实现偏 C fuzz harness，迁移到 Python/TypeScript 仓库修复需要重新定义结构图和验证器。

**阅读建议**：重点读 Function Triplets、Structural Flow Graph、staged rollback algorithm 和 RQ1/RQ2/RQ3 实验。证据来源：PDF 正文抽取。

### 8. Progressive Crystallization: Turning Agent Exploration into Deterministic, Lower-Cost Workflows in Production

- 类别：General Agent / Production Workflow Distillation
- 链接：[2607.07052](https://arxiv.org/abs/2607.07052)
- 综合评分：4.24

**问题背景与核心方法**：生产 AIOps Agent 每次处理重复问题都调用完整 LLM 推理，会形成持续成本和不可复现风险。Progressive Crystallization 把 Agent 探索视为发现机制，而不是永久执行方式：新问题先由 Type 3 agent-orchestrated flow 处理；反复成功后提升为 Type 2 hybrid workflow；证据足够后再提升为 Type 1 deterministic playbook；如果回归则自动 demote。

**创新点**：它把 agent trace 当成可蒸馏的工作流资产。直观说，Agent 不必永远以高成本“思考”同一类问题，成功轨迹可以转成更便宜、更可审计、更可复现的确定性流程。

**实验与 benchmark**：PDF 显示该方法部署在生产 cloud network operations AIOps 系统，处理每月数万 incident。八个月内 Type 1 deterministic executions 从 0% 提升到约 45%，另有约 30% hybrid、25% agent-orchestrated；per-incident agent cost 降低超过 70%。论文还给出 promotion/demotion 条件示例，例如多次成功运行、LLM classification consistency 和 deterministic rule review。

**可迁移价值**：适合用于长期运行的 Agent 产品：把高频成功任务抽成 workflow，再保留 Agent 处理新问题。局限是论文更像生产经验/系统论文，缺少开放 benchmark；promotion 依赖高质量 trace、acceptance tests 和人工/规则审核。

**阅读建议**：重点读 execution-type taxonomy、promotion/demotion lifecycle、trace extraction 和 cost model。证据来源：PDF 正文抽取。

### 9. Interpretable Uncertainty for Adaptive Retrieval and Reasoning in Question Answering

- 类别：Search Agent / Adaptive RAG Decision
- 链接：[2607.07380](https://arxiv.org/abs/2607.07380)
- 综合评分：4.03

**问题背景与核心方法**：RAG 系统常需要决定何时检索、何时继续推理。固定检索或多轮 prompt 策略要么低效，要么不透明。论文提出从 LLM hidden states 中估计两类信号：knowledge insufficiency 和 knowledge ambiguity/conflict。前者高时触发 RAG 补知识，后者高时追加推理或更高 test-time compute。

**创新点**：它把“检索”与“推理”触发条件拆成两个可解释不确定性维度。对 Search Agent 来说，这提供了一个简洁 action policy：缺知识就查，知识冲突就推理/澄清，而不是把所有不确定性都交给同一个阈值。

**实验与 benchmark**：PDF 是短论文/初步结果。正文显示实验在 factoid QA 的 NQ dataset 上，使用 Llama-2-7b-chat，并结合 SE 与 WEPR 选择性触发 RAG。结果显示相对 LLM-only baseline 分别提升 +5.9% 和 +4.7%，相对 always-on RAG baseline 分别提升 +3.3% 和 +2.1%。

**可迁移价值**：可作为 Search Agent 后训练的状态特征：把每步的不确定性拆成缺知识、冲突/歧义、证据充分性，再监督或强化学习选择 retrieve、reason、clarify、abstain。局限是目前结果初步，且 hidden-state probe 在闭源模型或工具 API 中未必可用。

**阅读建议**：重点读 uncertainty decomposition、hidden-state probe、RAG trigger policy 和 NQ preliminary results。证据来源：PDF 正文抽取；论文篇幅较短。

### 10. Reliable and Developer-Aligned Evaluation of Agents for Software Engineering

- 类别：Code Agent / Evaluation Methodology
- 链接：[2607.06713](https://arxiv.org/abs/2607.06713)
- 综合评分：3.87

**问题背景与核心方法**：这篇短文关注 software engineering agents 的评测方法，主张现有评测过于碎片化，并且容易由假设性语法场景扭曲真实能力。作者提出评测应关注 contamination-awareness、in-the-wild agentic behavior、trajectory-aware benchmarks 和 human-aligned failure modes。

**实验与 benchmark**：PDF 抽取显示该文为 preliminary methodology paper，没有稳定抽取到具体 benchmark 表格或数值结果。因此本日报不编造任务数、模型名或性能数字。

## 今日重点趋势

1. Code Agent 评测正在从最终 patch 成败转向轨迹、输入质量和测试充分性。AgentLens 看完整交互轨迹，bug report 论文看问题陈述，spec grounding 和 performance benchmark 论文看测试是否真的有外部依据。
2. Agent 安全开始进入工具协议层。SPELLSMITH 说明 MCP tool description、参数语义和自反思可以成为安全缓解机制，但也暴露出必须配合服务端校验和 sandbox。
3. 生产 Agent 的长期成本需要“轨迹蒸馏”。Progressive Crystallization 提醒我们，Agent 探索可以生成确定性 workflow，而不是永远依赖 LLM 重跑。
4. Search Agent 的检索决策更细粒度。Interpretable Uncertainty 将“缺知识”和“知识冲突”分开，为 retrieve/reason/clarify 的动作空间提供了状态信号。

## 潜在研究 idea

- 用 AgentLens 风格的 trajectory review 给 Code Agent 轨迹打细粒度标签，再把 spec-grounded tests 作为验证信号，训练“写测试前先抽规格”的修复策略。
- 把 AgentEval 的 workflow graph 与 SPELLSMITH 的 tool-level risk profile 结合：先挖 Agent 流程边界，再为高风险 MCP 工具自动生成安全描述和边界测试。
- 为 Search Agent 设计一个 uncertainty-state policy：hidden-state 或 logprob probe 判断缺知识/冲突，动作包括 retrieve、rewrite query、reason more、ask clarification、abstain。
- 在生产 Agent 中做 crystallization replay buffer：将反复成功的轨迹转成 deterministic workflow，并把 demotion 案例作为负样本训练异常检测器。
