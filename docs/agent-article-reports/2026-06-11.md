## Code Agent

### 1. CORE-Bench: A Comprehensive Benchmark for Code Retrieval in the Era of Agentic Coding

- 链接：[arXiv](https://arxiv.org/abs/2606.11864) | [PDF](https://arxiv.org/pdf/2606.11864)
- 证据来源：PDF
- 核心方法：论文把 Code Agent 的检索问题重新定义为 requirement-driven repository search，而不是传统 docstring-to-function 或 snippet retrieval。CORE-Bench 分三层：基础代码理解、issue-to-edit localization、broader context retrieval；后两层保留具体仓库状态、真实修改请求、多位置相关文件和局部干扰项。
- 创新点：它直接补了 coding agent 当前最薄的基础设施：给定 issue 后，模型不只是找“语义相似代码”，而是要找需要编辑的文件、函数、配置、测试和辅助上下文。数据规模也足够大：Level-1 约 172K queries，Level-2 覆盖 632 个仓库和 5,061 个查询，Level-3 覆盖 97 个仓库和 2,580 个查询， broader-context relevance labels 超过 106K。
- 可转化模块：适合拿来训练或评估 Code Agent 的检索器、上下文构造器和 repository navigator。尤其可以把 Level-2/3 拆成“先定位可编辑位置，再找验证/依赖上下文”的两阶段奖励。
- 局限：它主要评估 retrieval，不直接验证 patch 是否最终通过测试；自动构造 broader context label 的质量仍需要人工审计样本校准。
- 阅读建议：今天 Code Agent 方向第一优先级。

### 2. Exploration Structure in LLM Agents for Multi-File Change Localization

- 链接：[arXiv](https://arxiv.org/abs/2606.11976) | [PDF](https://arxiv.org/pdf/2606.11976)
- 证据来源：PDF
- 核心方法：针对多文件变更定位，比较线性单 Agent 遍历、带 REPL 的单 Agent、按仓库子系统划分的 domain-scoped parallel agents、BM25 与 Codex CLI baseline。作者在 ansible 仓库上构建 persistent-session 评估，使不同方法面对相同 base commit 和探索成本。
- 创新点：关键不是“多 Agent 一定更强”，而是 decomposition axis 要对齐仓库结构。论文明确区分 role-based multi-agent 与 subsystem-based multi-agent，并发现强制多 Agent 咨询会增加噪声和 token 成本；按子系统并行探索更适合多文件修改。
- 关键结果：domain-scoped agents 在小模型组里 micro-F1 最强，并且在扩展到 2025/2026 PR 的新基准上仅次于更大的 Codex 5.5 High；同时作者指出文档共演化是所有方法都没解决的隐性依赖。
- 可转化模块：可以迁移到 repo-level Code Agent 的 exploration policy：先建立稳定的 subsystem registry，再让局部 agent 产出候选文件和证据，由 coordinator 汇总，而不是让一个 agent 在线性路径里赌第一个目录。
- 局限：主要在 ansible 上验证，仓库边界清晰度会影响方法收益；论文评估的是 localization，不包含完整 patch-repair-test 闭环。
- 阅读建议：与 CORE-Bench 搭配读，前者定义评测，本文定义探索结构。

### 3. Rule Taxonomy and Evolution in AI IDEs: A Mining and Survey Study

- 链接：[arXiv](https://arxiv.org/abs/2606.12231) | [PDF](https://arxiv.org/pdf/2606.12231)
- 证据来源：PDF
- 核心方法：挖掘 83 个 AI IDE 开源项目中的 7,310 条 rule，构建 5 大类、25 小类 taxonomy，并结合 99 份开发者问卷与 1,540 次 rule evolution 事件分析规则如何演化。
- 创新点：把 `.cursor/rules` 这类“上下文配置”当成软件工程 artifact 研究，而不是 prompt 小技巧。论文还做了 artifact compliance assessment：160 个规则演化事件显示，规则更新后平均 compliance 从 49.14% 提升到 72.13%。
- 可转化模块：对 Code Agent 的长期项目记忆很有价值。可以把规则文件视为 agent context 的可版本化、可审计、可冲突检测资产，并训练 agent 在修复错误后主动建议规则增补。
- 局限：规则质量和任务成功率之间仍不是严格因果；样本来自已公开项目，可能偏向愿意暴露 AI IDE 配置的开发者。
- 阅读建议：适合放入 AI IDE / coding-agent context management 的 related work。

### 4. Enhancing LLM-Based Code Translation with Verified Multi-Semantic Representations

- 链接：[arXiv](https://arxiv.org/abs/2606.11863) | [PDF](https://arxiv.org/pdf/2606.11863)
- 证据来源：PDF；未进入 Top 10，作为 Code Agent 候补保留
- 核心方法：Multisage 从源代码中抽取 data-flow、type constraints、external API 等结构化语义，再生成 summaries、function-level tests、API-oriented descriptions/tests，并通过语义保持 mutation 与 cross-semantic consistency verification 过滤不可靠语义。
- 可转化价值：虽然任务是代码翻译，不是通用 repo patching，但“从代码自动构造多视角语义证据，再校准后喂给 LLM”的思路可迁移到 Code Agent 的验证器和 patch critic。
- 局限：迁移到真实 issue resolution 时，还要处理跨文件依赖、测试运行和增量编辑。

## General Agent

### 5. SkillJuror: Measuring How Agent Skill Organization Changes Runtime Behavior

- 链接：[arXiv](https://arxiv.org/abs/2606.11543) | [PDF](https://arxiv.org/pdf/2606.11543)
- 证据来源：PDF
- 核心方法：研究 Skill 的组织方式本身是否改变 agent 行为。作者对比 normalized flat baseline 与 Progressive Disclosure：根 `SKILL.md` 只给简洁入口，并按需指向支持资源；评估时保持任务知识等价，只改变组织结构。
- 创新点：论文把“Skill 写法”从主观风格变成可控变量，用 semantically controlled variants、matched multi-trial evaluations 和 trajectory evidence 分析运行行为。结果显示 Progressive Disclosure 先改变资源使用轨迹，再带来有限但可见的 outcome 收益。
- 关键结果：82 个 SkillsBench 任务中，Progressive Disclosure 让每条轨迹触达的 distinct skill resources 从 1.18 升到 3.85，effective uptake events 从 1.33 升到 3.92，并比 flat baseline 多 17 个 verifier-passing trials / 410 matched trials。
- 可转化模块：非常适合 Agent post-training 和 skill library 设计：训练目标不能只看最终成功，还要看 agent 是否按需打开资源、是否把支持材料用于检查和修复。
- 局限：收益 task-dependent；对严格输出格式、数值阈值或长 artifact 生成任务帮助较弱。
- 阅读建议：今天最贴近“Agent 能否稳定使用外部程序化知识”的论文。

### 6. Agents All the Way Down; A Methodology for Building Custom AI Agents from Substrate to Production

- 链接：[arXiv](https://arxiv.org/abs/2606.11869) | [PDF](https://arxiv.org/pdf/2606.11869)
- 证据来源：PDF
- 核心方法：提出构建 custom AI agent 的五阶段方法论：P1 substrate，把 LLM 当软件组件；P2 building blocks，包括 function calling、MCP、CLI orchestration、liteshell、agent loop、skills、hooks；P3 用通用 agent prototype；P4 harvest/fold/ship as CLI；P5 agent-tests-agent。
- 创新点：它把很多工程实践串成可执行循环，尤其是 Turtle pattern：把 agent 作为 CLI 发布后，多 Agent 编排就退化为 CLI composition。P5 也有启发：让通用 agent 驱动 custom agent 走行为场景，作为经典测试之外的行为评估层。
- 可转化模块：适合做真实生产 Agent 的工程 checklist：安全边界、私有工具、审计轨迹、成本曲线和 agent-tests-agent 都可以直接变成评测维度。
- 局限：方法论论文，实证强度不如 benchmark/实验论文；很多结论来自案例经验，不应当当作算法 SOTA。
- 阅读建议：适合工程落地和系统论文写作，不适合作为“新模型方法”引用。

### 7. A Five-Plane Reference Architecture for Runtime Governance of Production AI Agents

- 链接：[arXiv](https://arxiv.org/abs/2606.12320) | [PDF](https://arxiv.org/pdf/2606.12320)
- 证据来源：PDF
- 核心方法：为生产 Agent 运行期治理提出 five-plane decomposition：reasoning plane 负责意图裁决，network / identity / endpoint / data 四个 enforcement planes 执行策略；配套 stop-anywhere mediation、composite principals with capability attenuation、structured audit substrate。
- 创新点：把风险从“单次 API 请求是否允许”推进到“授权动作序列是否构成未授权业务流程”。这对工具型 Agent 很关键，因为每一步都可能被允许，但组合起来可能越权。
- 关键结果：论文给出 7 类生产 Agent 威胁和 5 个 workflow 演示；参考实现中 policy-engine core 的 attenuation correctness 和 evidence reconstructability 在 trial 中成立，adjudication 为 single-digit microseconds 级。
- 可转化模块：可直接迁移到 tool-use RL / deployment harness：reward 不只看任务完成，还应记录 delegation chain、capability attenuation、stop point 和 audit evidence。
- 局限：作者也明确说它治理 delegated action，不治理模型行为；还缺少 live agent benchmark 的完整系统评估。
- 阅读建议：适合生产 Agent 安全、审计与权限模型方向。

### 8. Knowing When to Ask: Self-Gated Clarification for Hierarchical Language Agents

- 链接：[arXiv](https://arxiv.org/abs/2606.11349) | [PDF](https://arxiv.org/pdf/2606.11349)
- 证据来源：PDF
- 核心方法：ACTION-RATING 把 clarification 放进 agent action space，用同一 0-100 ordinal scale 同时评分“继续导航”和“请求澄清”，让 ask 与 act 在每个中间决策点直接竞争。
- 创新点：论文区分 mandatory clarification 与 opportunistic clarification，并用 Information-Seeking Effectiveness 衡量“求助后下一步导航是否正确”，把“在哪里求助”和“收到的答案质量”分开。
- 关键结果：在 Harmonized Tariff Schedule 分类任务上，ISE 从 50% 升到 74%，10-digit accuracy 从 50.8% 到 67.0%；当答案质量降级时，accuracy 下降 18.8%，但信息寻求模式仍保持，说明求助定位是可单独分析的行为属性。
- 可转化模块：适合迁移到 Search Agent / Code Agent 的交互式执行：当检索、工具调用或文件定位不确定时，模型应把“问澄清问题/请求额外证据”作为正式动作，而不是事后 confidence threshold。
- 局限：任务是层级分类，不是开放 Web 或代码仓库；真实环境中 clarification 的成本和用户耐心还要建模。

### 9. Doc-to-Atom: Learning to Compile and Compose Memory Atoms

- 链接：[arXiv](https://arxiv.org/abs/2606.12400) | [PDF](https://arxiv.org/pdf/2606.12400)
- 证据来源：PDF
- 核心方法：Doc2Atom 将长文档拆成 semantically typed knowledge atoms，每个 atom 编译成独立 micro-LoRA adapter 和 provenance retrieval key；推理时 query router 只选择相关 atoms，组合成 query-specific adapter 注入 frozen base model。
- 创新点：相比 Doc-to-LoRA 这类单文档单 adapter，Doc2Atom 试图解决 irrelevant-query interference、compositional recall 和长文档扩展性问题。它把“记忆”做成可检索、可组合、带 provenance 的参数化模块。
- 可转化模块：适合启发 Agent 长期记忆：不要把完整历史压成单块摘要或单个 adapter，而是维护 atom-level provenance、router 和组合机制。
- 局限：目前更偏 parametric memory / QA，不是工具执行轨迹记忆；对可更新性、冲突事实和时间版本的处理还不够接近生产 Agent。

## Search Agent

### 10. When More Documents Hurt RAG: Mitigating Vector Search Dilution with Domain-Scoped, Model-Agnostic Retrieval

- 链接：[arXiv](https://arxiv.org/abs/2606.11350) | [PDF](https://arxiv.org/pdf/2606.11350)
- 证据来源：PDF
- 核心方法：提出 vector search dilution：大规模异构语料会让 dense/hybrid retrieval 返回语义相近但作用域错误的 chunk。MASDR-RAG 先用组织元数据做 domain scoping，再在作用域内检索和生成。
- 创新点：论文最有价值的结论是“scope first, then synthesize”。作者在 WYDOT 真实语料中观察到，文档从 54 扩到 1,128、chunk 到 88,907 后，准确率从 75% 掉到 40% 以下；domain scoping 将 P@10 从 0.77 提到 0.86。
- 关键结果：作者还发现 multi-agent orchestration 具有强配置依赖，在某个 Gemini production stack 上会把 RAGAS faithfulness 从 0.61 降到 0.35，形成 precision-faithfulness paradox。
- 可转化模块：对 Search Agent 很直接：先让 agent 学会识别检索作用域或证据域，再调用 scoped retriever；多 Agent 编排只应在真正多域且工具调用能力稳定时开启。
- 局限：主要基于企业文档和若干 QA 语料；domain metadata 质量决定上限。
- 阅读建议：今天 Search Agent 方向第一优先级。

### 11. Tail-Aware Adaptive-k: Query-Adaptive Context Selection for Retrieval-Augmented Generation

- 链接：[arXiv](https://arxiv.org/abs/2606.11907) | [PDF](https://arxiv.org/pdf/2606.11907)
- 证据来源：PDF
- 核心方法：TAA-k 用 ranked similarity curve 的 steep-flat-steep 几何结构先做 knee detection，定位候选窗口，再在窗口内用 EVT goodness-of-fit 检验识别 noise-dominated tail 的起点，从而为每个 query 选择自适应 k。
- 创新点：它不是训练新 reranker，而是给 RAG context selection 一个 training-free 的统计截断器，避免固定 Top-K 在 query 难度和相似度分布差异下失效。
- 关键结果：摘要和正文显示，TAA-k 在 WebQuestions、2WikiMultiHopQA、MuSiQue 上接近 oracle retrieval quality，F1 距 oracle 约 2-3%，复杂度从全局 EVT 的 O(N^2M) 降到 O(sqrt(N log N) * M) 量级。
- 可转化模块：适合做 Search Agent 的 context budget controller：把“要读多少证据”交给 query-adaptive cutoff，而不是固定 Top-5/Top-10。
- 局限：依赖相似度曲线形态假设；对开放网页搜索中多查询、多跳、去重后的证据集合还需要额外验证。

### 12. uva-irlab-conv at SemEval-2026 Task 8: Multi-Turn RAG with Learned Sparse Retrieval and Listwise Reranking

- 链接：[arXiv](https://arxiv.org/abs/2606.11945) | [PDF](https://arxiv.org/pdf/2606.11945)
- 证据来源：PDF；未进入 Top 10，作为 Search Agent 候补保留
- 核心方法：多轮 RAG pipeline：conversation query rewriting、learned sparse retrieval、pointwise reranking、LLM listwise reranking、最终生成，且全流程利用对话历史。
- 关键结果：SemEval-2026 Task 8 Task A retrieval 中 nDCG@5 为 0.5475，排名 2/38；但 generation 排名较低，主要因为没有显式建模 unanswerable scenarios。
- 可转化价值：适合借鉴“稀疏召回 + 对话上下文 listwise rerank”的工程组合，但对于可靠 Search Agent 还要补 unanswerable detection。

## 今日重点趋势

1. Code Agent 的瓶颈正在从“模型能不能写代码”前移到“能不能找到正确仓库上下文”。CORE-Bench 和 Multi-File Change Localization 都把 repository search、file localization、broader context retrieval 作为核心对象。
2. Agent 知识组织开始被实证化。SkillJuror 研究 Skill 的组织结构，AI IDE Rules 研究持久规则文件，两者都说明 context/skill/rule 不是静态提示，而是会改变运行轨迹的软件资产。
3. Search Agent 不应盲目扩大上下文。MASDR-RAG 和 TAA-k 都强调先控制作用域或自适应截断，再交给 LLM 综合。
4. 生产 Agent 的研究开始从 benchmark success 转向 governance、audit、capability attenuation、runtime mediation。这类工作会成为真实工具 Agent 的必要配套。

## 潜在研究 Idea

1. 构建一个 `repository search -> edit localization -> broader context retrieval -> patch/test` 的分阶段 Code Agent 训练环境，把 CORE-Bench Level-2/3 与真实测试执行连接起来。
2. 为 Skill/Rule/Memory 设计轨迹级 reward：不仅奖励最终成功，也奖励按需打开资源、正确引用规则、避免过度上下文注入。
3. 在 Search Agent 中加入 domain-scope router + adaptive-k controller：先判断证据域，再动态决定每个域读多少证据。
4. 把 runtime governance 变成 tool-use evaluator：每次工具调用都记录 principal、delegation chain、capability attenuation 和 audit evidence，作为训练数据字段。
