## 方向索引

### General Agent

`Compile, Then Page` 是今天最值得优先读的 General Agent 工作。它把企业 SOP 从长文本规则编译成可执行伪代码，并用 capability-gated runtime 只把当前 active frame 分页给模型。`Motif` 则把 Web agent 从“用户说我要自动化什么”推进到“系统被动观察浏览器行为并发现可自动化 workflow”。`Robo-Reporters` 和 `Gauntlet` 都强调 agent architecture 本身会显著改变输出质量、透明度和审计性。

### Search Agent

`Tool-Adaptive LLM Reranker` 最贴近可训练 Search Agent：它把 pointwise reranking 形式化为 agentic MDP，用 cost-aware GRPO 学会在不确定时调用搜索工具，在简单样本上绕过工具。`Boolean queries are all you need?` 的价值在于提醒我们，agentic search 未必需要复杂 dense retriever；如果 LLM 能生成和迭代布尔查询，传统精确匹配也可能成为很强的交互式检索底座。`NGM-RAG`、`SVD-RAG` 和 `FLARE` 则分别对应 graph matching、低成本树组织检索、反馈驱动 RAG 自改进。

### Code Agent

`Mizzle` 是今天最重要的 Code Agent 信号。它不直接提升 LLM 找 bug 的召回率，而是要求每个 bug report 附带 Rocq/Iris 中可检查的 incorrectness proof，从而把“模型声称这里有 bug”变成“存在一条可达错误执行”。这对减少 agentic bug finding 的 false alarms 很关键。

## Top 10

### 1. Compile, Then Page: Executable SOP Programs and a Capability-Gated Runtime for Procedural LLM Agents

- 类别：General Agent / SOP Runtime / Tool-use Procedure Following
- 链接：[2607.11346](https://arxiv.org/abs/2607.11346)
- 综合评分：4.78

**问题背景与核心方法**：企业 agent 常常需要遵守长程、条件分支很多、且带拒绝规则的 SOP。把整份 SOP 直接放进 prompt 会让模型跳过检查、过早执行动作或错误拒绝。论文把机器可读 SOP 约束离线编译成两层伪代码：每个用户目标对应 process function，每条规则对应会返回证据的 rule subroutine。在线时，一个 program-guided stack machine 维护 stack、cursor、variables 和 audit trail，只把当前 active frame 分页给 LLM；LLM 负责语义判断、工具调用和证据解释。这个 runtime 是软约束：它不硬拦模型动作，但能让跳过检查、无证据 return、越界调用等行为在轨迹里可见。

**创新点**：核心贡献不是“更长上下文”，而是把流程状态从 prompt 文本转移到可执行 runtime。它把 SOP 的 obligations、verification recipes、evidence-bearing returns 组织成 goal-scoped executable flow，使 agent 在每一步只面对当前要验证的 frame。更重要的是作者明确给出 capability gate：强模型能从 active-frame paging 受益，弱模型可能被 runtime 伤害。

**实验与 benchmark**：论文在 SOPBench 上做三臂实验：official prose、compiled text、compiled program plus PG runtime。Bank clean subset 共 125 个任务，DeepSeek-V4-Flash 从 70.4% 提升到 86.4%，再到 92.8%；refusal correctness 达到 100%。七个 domain 的 runtime 对比中，两个强模型分别得到 58:19 和 75:31 的正向 discordant pairs；弱模型在 Bank 上反而下降 14-26 分。full-program cursor ablation 显示，把 active frame 放在前面且保留完整程序，能恢复大部分拒绝收益，selective visibility 只带来较小增量。

**可迁移价值与限制**：这篇非常适合迁移到真实 tool-use agent：把任务规范、工具前置条件、拒绝条件和审计证据编译成 runtime，而不是塞进系统 prompt。限制是软约束仍依赖模型自律，论文也提示不可逆动作仍需要工具层硬 guard；此外 SOPBench 与真实企业系统仍有 simulator 和任务清洗差异。

### 2. Tool-Adaptive LLM Reranker

- 类别：Search Agent / Tool-use RL / Reranking
- 链接：[2607.10555](https://arxiv.org/abs/2607.10555)
- 综合评分：4.62

**问题背景与核心方法**：LLM reranker 对复杂或超出参数知识边界的 query-document pair 容易幻觉，但如果对每个候选都调用搜索工具，延迟会无法接受。TALRanker 把 pointwise relevance scoring 写成 agentic Markov decision process：模型先生成 reasoning trajectory；如果内部知识足够，就直接输出 yes/no 相关性；如果发现知识缺口，就暂停生成、调用外部搜索工具、把 tool response 拼回状态，再继续判断。训练分两阶段：先用 language-preserving hybrid loss 建立打分能力并防止生成/工具能力塌缩，再用 cost-aware GRPO 学会成本敏感工具调用。

**创新点**：这篇把 Search Agent 的工具调用从“是否可用工具”推进到“是否值得为这个样本付出工具成本”。reward 设计是不对称的：不必要工具调用要付延迟惩罚，但不查导致严重幻觉也会被罚。它适合启发 agent post-training 中的 tool-use gating 数据构造。

**实验与 benchmark**：论文报告 TALRanker 在 BEIR 和 BRIGHT 上同时提升准确率和效率。工具调用边际贡献实验显示，TALRanker-4B 在 BEIR Avg. 从 zero-hop 42.9 到 with-tools 46.0，BRIGHT Avg. 从 26.1 到 28.2；8B 分别从 43.5 到 46.3、26.8 到 29.7。消融显示直接 RL 会让 0.6B 模型结构性崩溃，NDCG@10 下降 30.33 点；4B/8B 才能稳定从 tool-adaptive learning 中获益。附录还用 NQ、TriviaQA、PopQA、HotpotQA、2WikiMultiHopQA、Musique、Bamboogle、Facts、SimpleQA 检查训练前后 agentic search 能力。

**可迁移价值与限制**：可以直接用于训练 Search Agent 的“查询前置信心判断”和“工具调用预算控制”。限制是该方法仍依赖 base LLM 原生 reasoning/tool-use 能力，小模型会崩；论文也说明测试集中强制每个 query-document 都调用工具成本不可行，因此很多结果围绕 adaptive 策略而非全量工具检索上限。

### 3. Mizzle: A Complete Concurrent Incorrectness Logic for Preventing False Alarms in Agentic Bug Finding

- 类别：Code Agent / Bug Finding / Machine-Checked Evidence
- 链接：[2607.11611](https://arxiv.org/abs/2607.11611)
- 综合评分：4.45

**问题背景与核心方法**：LLM 被越来越多地用于真实项目 bug finding，但误报会消耗开发者大量时间。Mizzle 的思路不是再让另一个 LLM 判断 bug report，而是要求模型为每个 bug report 构造一个机器可检查的 incorrectness proof：证明存在一条可达执行会触发 stuckness、non-linearizability 或 race。逻辑面向 OCaml 5 的实质子集，基于 ZooLang 操作语义，用 Iris 框架在 Rocq 中机械化，并证明 soundness 和 completeness。

**创新点**：这篇把 Code Agent 的“我发现 bug”变成“我能给出可检查证据证明这个错误行为可达”。Incorrectness logic 是欠近似的，它证明的是某条坏执行确实存在，因此天然适合防止 false alarms。对 agentic bug finding 来说，这比自然语言解释或复现测试更强，尤其适合并发 race 这类不稳定复现的 bug。

**实验与 benchmark**：论文展示 Claude Sonnet 5 使用 Mizzle 证明 race 的 proof-of-concept。测试来自 DataRaceBench 的 OCaml 翻译子集，样例 4-18 行，模型输出 32-223 行 proof。论文报告每例成本约 0.90-7.10 美元、平均 2.70 美元，平均时间约 7 分钟，Rocq MCP failure 平均 31.6%，最高 outlier 70.0%。这些数字说明方向可行，但工程摩擦仍然很高。

**可迁移价值与限制**：适合做 Code Agent 的 verifier layer：让 agent 报 bug 时同时产出 proof、trace、test 或可检查证据。限制是当前范围集中在 OCaml 5、Rocq/Iris 和并发逻辑，距离通用多语言仓库级 agent 还有很远；作者也明确说目标不是提升找 bug 能力，而是证明报告不是 false alarm。

### 4. Boolean queries are all you need?

- 类别：Search Agent / Agentic Search / Boolean Retrieval
- 链接：[2607.11362](https://arxiv.org/abs/2607.11362)
- 综合评分：4.40

**问题背景与核心方法**：论文重新审视布尔检索：人类不擅长写布尔查询，但 LLM 可以把自然语言 topic 改写成合理的布尔表达式，并在交互中迭代。作者实现了 Vole search agent，LLM 每轮可以提交 Boolean query、请求下一个结果或读取全文；检索底层使用 shortest substring ranking，把语料看成连续字符串，按满足布尔表达式的最短匹配片段密度来排名，不需要监督训练、全局统计或 term weights。

**创新点**：它挑战了 agentic search 默认依赖 dense retrieval 或复杂 reranker 的假设。对于 Search Agent，关键不是检索器越神经越好，而是 agent 能否根据已见证据调整可解释查询，并把局部精确匹配当作可控工具。论文还指出 Cottontail 这类 annotative indexing 可以承载 raw text、结构、annotations 和模型判断，作为 agent memory/retrieval substrate。

**实验与 benchmark**：实验使用 TREC 2024 RAG Track 的 MS MARCO V2.1 deduped segment collection，标准 86 topics，每个 topic 最多 100 次模型调用、最多 50 个 distinct relevance judgments。Vole 达到 NDCG@10=0.6863；若不对未判文档做 UMBRELA hole-filling，则为 0.5340。论文指出该成绩高于许多 dense、sparse 和 learned-sparse first-stage retrievers，但低于很多 LLM reranking run。

**可迁移价值与限制**：适合作为 Search Agent 的轻量交互式检索工具，尤其适合需要可解释 query trace 的场景。限制是实验只在一个公开 test collection 上完成，作者也承认模型可能在训练中见过该集合；结果更像探索性证据，而不是通用结论。

### 5. Motif: Discovering and Automating Personal Web Workflows

- 类别：General Agent / Browser Workflow Automation / End-user Programming
- 链接：[2607.10531](https://arxiv.org/abs/2607.10531)
- 综合评分：4.26

**问题背景与核心方法**：现有 vibe coding 或 programming by demonstration 假设用户知道什么任务值得自动化，但很多重复浏览器 workflow 已经习惯化，用户不会把它们识别成“可编程任务”。Motif 被动记录日常浏览器行为，收集点击、复制粘贴、tab/URL 切换、周围文本和 UI 标签，再用 LLM 做 pattern mining、ROI 评估、program design 和 Tampermonkey 脚本生成。用户可以 review、refine、deploy，最终得到本地执行的确定性程序，而不是每次都跑 LLM agent。

**创新点**：它把 Web Agent 的重心从“执行用户明确指令”移到“发现用户没有意识到的自动化机会”。这对真实 browser agent 很重要：很多高价值 agent 任务并不是一次性 goal，而是重复 workflow 的抽象、压缩和程序化。

**实验与 benchmark**：用户研究包含 8 名参与者，平均 5.5 天真实浏览器使用。系统识别 175 个 patterns，平均每人 22 个；研究中参与者审查 40 个 pattern，其中 34 个（85%）被认为对应日常 routine，24 个（60%）成功部署并按预期运行。参与者在使用工具前只自行识别出 3 个（7.5%）为可自动化。2-4 天后 follow-up 中，保留的 18 个程序里 30% 被经常使用，40% 被使用过几次。

**可迁移价值与限制**：可作为 browser/computer-use agent 的 workflow mining 前端，把重复人工轨迹转成确定性技能。限制是复杂 DOM、Google Docs、canvas/game 等页面仍容易失败；用户也常觉得修改 program description 负担高，需要更高层的 refinement interface。

### 6. Can LLMs Perform Deep Technical Comprehension of Computer Architecture Papers?

- 类别：General/Research Agent / Multi-agent Reading / Technical Review
- 链接：[2607.11859](https://arxiv.org/abs/2607.11859)
- 综合评分：4.24

**问题背景与核心方法**：论文研究 LLM 是否能做“深技术理解”，不是摘要，而是说清核心机制、隐含假设、评估问题和跨工作联系。Gauntlet pipeline 用五个独立 expert-persona reviewer 阅读整篇论文：microarchitecture、workload/evaluation、simulation-tools auditor，以及从约 90 个 persona 库里按子主题匹配的两个专家。之后 synthesis agent 保留分歧，而不是把意见平均掉。

**创新点**：它强调 multi-agent 结构的价值来自独立视角和 disagreement-preserving synthesis。对科研阅读 agent 来说，这比单个“请深入分析”prompt 更接近真实组会：不同专家抓住不同风险，最后由综合者组织冲突。

**实验与 benchmark**：20 篇 ISCA 2025 和 HPCA 2026 论文上，10 名研究者各写人工分析，再对非自己论文比较人类分析和 Gauntlet。20 次比较中 Gauntlet 赢 15 次，人类赢 4 次，1 次平；paired Wilcoxon 在 per-analyst totals 上 p<0.01。98 篇自动消融显示 pipeline 相对强 single-persona agent 在 96% 论文上胜出，说明收益主要来自多视角结构和 synthesis pass。

**可迁移价值与限制**：可迁移到论文阅读 agent 的 deep-read 模板：独立 reviewer、评估审计、机制重构、综合分歧。限制是实验 open-label，且全部使用 Claude Opus 4.5；人类胜出的案例往往源自 Gauntlet 的 confident wrong claim 或未能把机制讲成可教学形式，说明 calibration 仍是短板。

### 7. Robo-Reporters: Evaluating Autonomous AI Agents as Algorithmic Gatekeepers in Computational Journalism

- 类别：General Agent / Architecture Evaluation / Transparency
- 链接：[2607.10736](https://arxiv.org/abs/2607.10736)
- 综合评分：3.99

**问题背景与核心方法**：论文把新闻任务当成 agent architecture 对信息流的 gatekeeping 实验。四种架构在相同底层模型和工具下比较：monolithic Claude、chain-based LangChain、multi-agent CrewAI、autonomous iterative AutoGPT。任务覆盖 50 个新闻工作任务，五个难度等级，包括事实核验、来源分析、背景综合和调查任务。

**创新点**：它不是只问哪个 agent 准，而是拆分 duration、steps、accuracy、source consultation/citation、transparency 等维度，证明架构本身会改变信息选择、拒绝、引用和可审计性。这个视角可迁移到 Search Agent 和 Research Agent 的 harness 设计。

**实验与 benchmark**：200 次受控实验全部完成。CrewAI 准确率最高 84.7%，但平均耗时 60.5 秒，约为最快 LangChain 30.0 秒的两倍。架构对 task duration 影响显著，F(3,196)=24.54，p<.001，eta-squared=.27；对 computational steps 的影响更强，F(3,196)=305.63，eta-squared=.82。Claude 可完整追踪 source consultation，平均咨询 15.0 个 source、最终引用 4.2 个，source rejection rate 71.7%；框架式系统则内部检索不透明，只能看到最终引用。

**可迁移价值与限制**：适合提醒我们：agent harness 的日志、工具封装和分工会影响评估可解释性。限制是新闻任务和自动准确率指标不能完全代表真实 newsroom 质量，且框架不透明既是发现也是测量限制。

### 8. Enhancing LLMs through human feedback: a journey towards self-improvement

- 类别：Search Agent / Feedback RAG / Human-in-the-loop
- 链接：[2607.11267](https://arxiv.org/abs/2607.11267)
- 综合评分：3.69

**问题背景与核心方法**：论文提出 FLARE，用用户反馈改进 primary RAG。系统离线把用户反馈和对话上下文合并、分类成 factual corrections 与 stylistic/structural guidelines，并写入向量索引；在线查询时，primary RAG 与 Feedback-RAG 并行运行，找到相似历史反馈后，把事实修正注入上下文，把风格指南注入系统指令。

**创新点**：它的启发点在于把 human feedback 变成可检索、可注入的长期修正层，而不是只做一次性评分。对 Search Agent 来说，这相当于把用户纠错转成持续改进的 retrieval memory。

**实验与 benchmark**：评估包含 Fictional Persona Dialogs、Trivia、Wireless Technical Documentation 三个数据集，使用 LLM-as-a-Judge。FPD 上 FLARE 几乎全部达到 5 分；Trivia 包含 70/30 train/test 的 2024-2025 current events；Wireless benchmark 来自内部平台一年真实反馈，包含 80 个训练 QA 和 210 条 textual feedback。论文报告 custom domain 中 feedback 几乎都能检索到，但生成模块只在 61% 答案中有效利用，暴露 retrieval-utilization gap。

**可迁移价值与限制**：适合做 RAG/Search Agent 的用户反馈记忆层。限制是实验依赖 LLM-as-a-Judge，且模板拼接式注入容易让反馈被模型忽略；它更像系统原型报告，方法严谨度弱于前几篇。

### 9. NGM-RAG: Neural Graph Matching based Retrieval-Augmented Generation

- 类别：Search Agent / Graph RAG / Multi-hop Retrieval
- 链接：[2607.11159](https://arxiv.org/abs/2607.11159)
- 综合评分：3.57

**问题背景与核心方法**：传统 RAG 主要依赖文本相似度检索，遇到多跳问题时难以显式利用关系结构。NGM-RAG 把 graph construction、graph matching 和 answer generation 放进统一流程，用 text-based matching 与 GNN matching 组合，再通过 adaptive weighting 选择最相关的 contextual node information。

**创新点**：它的迁移价值在于把 retrieval 从 flat chunks 改成关系图匹配，适合需要多跳证据、实体关系和长上下文摘要的 Search Agent。与 GraphRAG/LightRAG 相比，它强调 neural graph matching，而不是只构建社区摘要或图索引。

**实验与 benchmark**：摘要与正文显示评估覆盖 multi-hop question answering 和 long-context summarization，并与 NaiveRAG、GraphRAG、LightRAG 等 graph-enhanced 方法比较。由于 PDF 表格抽取噪声较大，本日报没有摘录无法稳定核对的具体数值。

**可迁移价值与限制**：可作为 agentic retrieval 的 graph matching 模块参考。限制是候选池里相似 RAG 论文很多，这篇更像组件级改进；若目标是 post-training/tool-use reasoning，它不如 TALRanker 和 Boolean Search 直接。

### 10. SVD-RAG: Efficient Tree-Organized Retrieval-Augmented Generation via Singular Value Decomposition

- 类别：Search Agent / Efficient RAG / Hierarchical Retrieval
- 链接：[2607.10316](https://arxiv.org/abs/2607.10316)
- 综合评分：3.53

**问题背景与核心方法**：RAPTOR 类层级 RAG 需要在每个内部节点做 LLM abstractive summarization，构树成本高。SVD-RAG 用 dense sentence embedding matrix 上的 SVD 做 extractive summarization，挑选 principal components 能量贡献最高的句子作为内部节点摘要。它是 deterministic、无需额外 API 调用，并可用 energy-ratio threshold 根据内容复杂度自适应压缩。

**创新点**：它对 Search Agent 的价值在工程侧：把层级索引构建从 LLM summarization 改成可复现、低成本的线性代数过程。对于需要频繁更新知识库或长文档 memory 的 agent，这种 deterministic compression 可能比黑盒摘要更好审计。

**实验与 benchmark**：摘要报告，在 identical corpora、clustering 和 beam search 的 head-to-head 中，SVD-RAG 与 RAPTOR LLM summarization 检索质量差距在 1-5% 内，MRR 0.867 vs. 0.875，Recall@1 0.483 vs. 0.458，同时构树 317x 更快（0.1s vs. 31.7s）。在 205 chunks、100 queries、20 topic variations 的 scaled multi-topic benchmark 上，相对 flat embedding retrieval，Recall@1 提升 4.2x，MRR 提升 3.1x。

**可迁移价值与限制**：适合做 agent memory 或文档 RAG 的低成本树组织模块。限制是方法本身不解决 query planning、tool-use policy 或答案验证，只是更便宜地组织上下文。

## 今日重点趋势

1. Agent runtime 正在变成第一等研究对象。`Compile, Then Page` 说明 prompt 里的规则不够，状态、cursor、active frame、audit trail 应该由 runtime 维护。
2. Search Agent 的关键训练信号正在从“会不会调用搜索”变成“什么时候值得调用搜索”。`TALRanker` 的 cost-aware reward 很适合作为后训练样本设计参考。
3. Code Agent 的输出需要可验证证据。`Mizzle` 提醒我们，bug report 如果不能被测试、proof 或 trace 检查，就很难进入高信任闭环。
4. Multi-agent 架构的收益来自结构化分歧和角色化审计，而不是简单多跑几次。`Gauntlet` 和 `Robo-Reporters` 都支持这一点。

## 潜在研究 idea

- 为 tool-use agent 构建 `procedure-to-runtime` 数据集：把 SOP、API precondition、拒绝规则和 verifier recipe 编译成可执行 frame，并训练模型在 active frame 下行动。
- 做 Search Agent 的 tool-use gating RL：reward 同时包含答案质量、工具调用成本、未查导致的事实错误，以及证据覆盖率。
- 在 Code Agent bug finding 中加入 proof/test obligation：模型每报告一个 bug，必须给出可运行 repro、symbolic trace 或机器可检查 proof，否则 reward 为低。
- 给 browser agent 做 ambient workflow mining：从真实用户轨迹中发现可程序化重复流程，再把流程固化成 deterministic scripts/skills，而不是每次让 LLM 端到端操作。
