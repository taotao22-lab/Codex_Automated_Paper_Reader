今天的候选与当前 Agent 研究方向高度匹配。主要信号集中在三类问题：Agent runtime 如何从纯文本上下文迁移到可验证、可持久化的执行环境；Agent evaluation 如何从静态 benchmark 走向真实长链路流程；Search/Code Agent 如何把检索、执行反馈、测试和治理机制纳入闭环。

## General Agent

1. [CaveAgent: Transforming LLMs into Stateful Runtime Operators](https://openreview.net/forum?id=p3dlOhpqKD)
   - 核心方法：把 Agent 从“文本生成器”转为“运行时操作者”，用语义流负责轻量推理，用持久 Python runtime 保存对象、状态和执行产物。
   - 迁移价值：贴近 DeepSeek-V3.2 报告里的 reasoning-in-tool-use 与 agentic task synthesis，可作为构建可验证工具使用环境的参考。
   - 可转化实验：为本地 Agent 增加持久 runtime state，记录 DataFrame、API 连接、文件句柄等对象，并把执行结果转成 reward/verifier 信号。
   - 局限：摘要显示评测集中在 Tau2-bench 和 BFCL，真实企业工具链、多权限环境和安全隔离还需要更强验证。
   - 证据来源：abstract-only

2. [LACUNA: Safe Agents as Recursive Program Holes](https://arxiv.org/abs/2605.28617)
   - 核心方法：把 Agent 动作表达为类型化的 `agent[T](task)` 程序洞，由模型生成代码，运行前通过类型检查、工具权限和数据流约束。
   - 迁移价值：把 ReAct、子 Agent、技能和多模型规划纳入普通控制流，适合研究“可编译的 Agent 安全边界”。
   - 可转化实验：为 Code Agent 的每一步工具调用添加静态检查和失败回滚；把编译诊断作为重试反馈。
   - 局限：安全保证依赖类型系统和工具边界设计，对 prompt injection 和外部副作用的覆盖仍需全文确认。
   - 证据来源：abstract-only

3. [Rethinking Memory as Continuously Evolving Connectivity](https://arxiv.org/abs/2605.28773)
   - 核心方法：提出 FluxMem，把 Agent memory 建模为异构图，并通过连接形成、反馈驱动修正、长期巩固三个阶段动态演化。
   - 迁移价值：比静态向量库更适合长期 Agent，尤其适合把成功轨迹蒸馏成可复用 procedural circuits。
   - 可转化实验：在长程任务中记录“任务-工具-结果-反馈”图，定期修剪干扰边、补全缺失边，并评估记忆泛化性。
   - 局限：摘要声称在 LoCoMo、Mind2Web、GAIA 上达到 SOTA，但具体成本、图规模和在线更新稳定性需进一步阅读。
   - 证据来源：abstract-only

## Search Agent

4. [Beyond Similarity: Task-Aligned Retrieval for Language Models](https://arxiv.org/abs/2605.27951)
   - 核心方法：提出 TAG，用“适用性”替代语义相似度，把文档转成可追踪的 condition-action rules，再用成对 LLM judgment 选择适用规则。
   - 迁移价值：对 Search Agent 很关键，说明检索目标不应只是找相似文本，而应找能触发正确动作或约束的依据。
   - 可转化实验：把网页搜索结果转成规则卡片，评估 Agent 是否能选择真正适用于当前任务的规则。
   - 局限：摘要中的任务集中在规则/指令治理场景，开放网页任务中的规则抽取噪声仍需验证。
   - 证据来源：abstract-only

5. [Mixture-of-Experts Knowledge Graph Retrieval-Augmented Generation for Multi-Agent LLM-based Recommendation](https://arxiv.org/abs/2605.28175)
   - 核心方法：MixRAGRec 用多个 Agent 分别负责 KG 检索粒度路由、知识偏好对齐和推荐生成，并用多 Agent policy optimization 联合训练。
   - 迁移价值：虽然是推荐系统场景，但“按查询复杂度选择检索粒度”对 Search Agent 和 RAG Agent 有明显迁移价值。
   - 可转化实验：把搜索任务分为实体级、关系级、子图级检索，由路由 Agent 决定粒度，再比较答案完整性和噪声。
   - 局限：负向信号是 recommender system，日报中不应把它视为通用 Agent 论文，只作为 KG-RAG 路由思路参考。
   - 证据来源：abstract-only

6. [Search for Coverage: Learning Coverage-Aware Retrieval with Augmented Sub-Question Answerability](https://arxiv.org/abs/2605.28522)
   - 核心方法：CoveR 面向长答案 RAG 的 coverage ranking，用 LLM 生成的子问题可回答性信号构造训练数据。
   - 迁移价值：适合 Search Agent 的多跳调研任务，核心不是命中一个相关段落，而是覆盖所有必要事实。
   - 可转化实验：把每日论文检索拆成多个子问题，例如方法、评测、代码、局限，再训练或选择覆盖更完整的检索器。
   - 局限：摘要主要报告 dense retrieval，尚不清楚与 Agent 式迭代搜索结合时的成本和收益。
   - 证据来源：abstract-only

## Code Agent

7. [Benchmarks are Not Enough: RAMP for Runtime Assessing of Agentic Models in Production Systems](https://arxiv.org/abs/2605.27492)
   - 核心方法：RAMP 提出面向生产系统的长链路软件工程 Agent runtime assessment，覆盖串行依赖、工具链交互、恢复机制和多维效用指标。
   - 迁移价值：对 Code Agent 评测很重要，因为它直接指出静态短任务 benchmark 无法反映真实生产工作流。
   - 可转化实验：把本地代码 Agent 任务拆成多阶段流水线，记录阶段成功率、恢复次数、工具调用成本和失败传播。
   - 局限：摘要显示没有模型能完成完整流水线，说明评测很难，但也需要看任务构造是否过窄。
   - 证据来源：abstract-only

8. [Automating Formal Verification with Agent-Guided Tree Search](https://arxiv.org/abs/2605.27485)
   - 核心方法：在 Lean verified-code generation 中比较 agentic loop 与两类 agent-directed tree search，通过 mathlib search 改善证明生成。
   - 迁移价值：适合 Code Agent 的“搜索结构 vs 连续迭代”研究，尤其是形式化验证中如何分配 token budget。
   - 可转化实验：在代码修复任务中比较线性 ReAct、状态树搜索、上下文树搜索，观察中等难度任务和高难任务的差异。
   - 局限：主要面向 Lean 和形式证明，迁移到普通仓库级任务需要改造状态定义和 verifier。
   - 证据来源：abstract-only

9. [Tool Forge: A Validation-Carrying Toolchain for Governed Agentic Execution](https://arxiv.org/abs/2605.28000)
   - 核心方法：把工具封装为带 intent、contract、实现、依赖策略、测试、文档、验证证据和路由元数据的 capsule，并通过 Router 降低上下文负担。
   - 迁移价值：直接对应 Agent 工具治理问题，适合构建安全、可审计、可路由的 MCP/工具目录。
   - 可转化实验：为常用本地工具生成 validation-carrying capsule，并比较全量 schema 暴露与 intent-scoped routing 的 token 和成功率。
   - 局限：摘要称 benchmark 是初始系统评测，不应过度解读为 SOTA。
   - 证据来源：abstract-only

10. [GUI Agents for Continual Game Generation](https://arxiv.org/abs/2605.28258)
    - 核心方法：用 GUI Agent 扮演客观 evaluator 和主观 playtester，把游戏生成变成 coding agent 与 GUI agent 的持续循环。
    - 迁移价值：这是 Code Agent 闭环的好样例：生成代码、运行交互环境、由 GUI Agent 测试、再反馈修复。
    - 可转化实验：把网页应用生成任务接入浏览器 playtester，让 GUI Agent 生成可复现失败轨迹并回传给代码 Agent。
    - 局限：游戏生成是特殊场景，结论迁移到普通业务系统前需要验证任务复杂度和评测鲁棒性。
    - 证据来源：abstract-only

## 今日趋势

- Runtime 正在成为 Agent 的核心抽象：CaveAgent、LACUNA、Tool Forge 都在把工具调用、状态、验证和权限从 prompt 文本中移到可执行系统层。
- Agent 评测从“单题正确率”转向“长链路、可观察、可恢复”：RAMP 和 AgenticVBench 都强调真实 workflow、工具选择、失败传播和 harness 影响。
- Search Agent 的检索目标正在从 similarity 走向 applicability 与 coverage：TAG 和 CoveR 都说明搜索应服务于动作和完整答案，而不是单纯相关性。
- Code Agent 的闭环越来越依赖外部 verifier：Lean、GUI playtesting、REST API metamorphic testing 都体现了“执行反馈即训练/评测信号”。

## 潜在研究 idea

- 构建一个 `runtime-first` Agent 原型：把文件系统、Python 对象、工具结果和验证器都放入持久 runtime，再让 LLM 只生成受控操作。
- 为 Search Agent 设计 applicability-based retrieval benchmark：要求 Agent 找到能触发正确动作的规则，而不是最相似网页。
- 在代码 Agent 中比较三种控制结构：线性 ReAct、树搜索、持久 runtime，并用同一组仓库级 issue 评估成本和成功率。
- 把工具文档自动转成 validation-carrying capsule，测试是否能降低工具误调用和上下文开销。
