## General Agent

### 1. CaveAgent: Transforming LLMs into Stateful Runtime Operators
- 链接：[OpenReview 页面](https://openreview.net/forum?id=p3dlOhpqKD) | [PDF](https://openreview.net/pdf/4713bd00b2f65526ef18a6ba85587520fa080983.pdf)
- 证据来源：PDF
- 核心方法：把 agent 的主工作区从文本上下文切到持久 Python runtime。论文用 dual-stream 结构区分轻量语义推理和 runtime 执行，把复杂对象注入、保存、检索都变成一等能力，而不是每轮序列化成文本。
- 创新点：它不是简单的 “code-as-action” 复述，而是把 stateful runtime management、对象级 memory、skill injection 和可验证反馈绑到同一个执行协议里。对 post-training 很关键的一点是，runtime 可观测状态本身能提供程序化 reward / verification 信号。
- 实验价值：在 Tau^2-bench 和 BFCL 上，11/12 个设置提升，零样本之外还显示出显著 token 节省。这个方向直接对应 “工具调用协议是否本身就是瓶颈” 的问题。
- 可转化实验/模块：可以复现实验中的持久对象槽、runtime object store、skill injection registry、以及由 runtime state 导出的 verifiable reward。
- 局限：当前证据主要集中在函数调用和多轮零售任务，跨浏览器/跨仓库/跨 GUI 的泛化仍需额外验证。
- 阅读建议：优先看 runtime stream、stateful runtime management、token efficiency 和 RLVR 相关讨论。

### 2. LACUNA: Safe Agents as Recursive Program Holes
- 链接：[arXiv 页面](https://arxiv.org/abs/2605.28617) | [HTML](https://arxiv.org/html/2605.28617v1)
- 证据来源：paper page
- 核心方法：把 agent action 形式化成 typed hole `agent[T](task)`，由模型在运行时补全代码，再在执行前用类型系统整体检查。被拒绝的 action 不会污染环境，编译错误直接变成 retry feedback。
- 创新点：它把 ReAct、sub-agent、skills、parallel decomposition、多模型规划统一回普通控制流，而不是堆在外层 runtime 上写很多 ad hoc harness。对安全性来说，类型约束同时限制工具可用范围和数据流。
- 实验价值：在 BrowseComp-Plus 和 Tau^2-bench 上验证了“先拒绝危险生成，再重试”的路线可行；对 prompt injection 的能力边界也做了针对性评测。
- 可转化实验/模块：typed tool capability、compile-before-execute、diagnostic-driven retry、静态控制数据流的 agent IR。
- 局限：语言和实现栈偏强类型系统路线，工程接入门槛高；在开放世界 web 任务里的效率与表达力 tradeoff 还要继续看。
- 阅读建议：如果你在做 agent safety 或 tool-use IR，这篇值得当作结构性 related work，而不是只看 benchmark 分数。

### 3. AndroidDaily: A Verifiable Benchmark for Mobile GUI Agents on Real-World Closed-Source Applications
- 链接：[arXiv 页面](https://arxiv.org/abs/2605.27761) | [HTML](https://arxiv.org/html/2605.27761v1)
- 证据来源：paper page
- 核心方法：构建 350 个真实日常 Android 任务，覆盖 94 个闭源 App，并提出 GRADE 评估器，用可观察的外部 guideline 而不是内部状态做自动诊断。
- 创新点：闭源 App 无法读取内部状态，这是 GUI agent 评测长期缺口。GRADE 用操作义务、输出质量、负约束三层 guideline，把“不可见内部状态”的任务转成可验证过程评测。
- 实验价值：GRADE 与人工评审 87.37% 一致；最强模型仅 62.0% 成功率，且在多约束和跨 App 任务上明显掉点。这类诊断型 benchmark 对 agentic post-training 很有价值，因为失败轨迹可直接转成训练样本。
- 可转化实验/模块：外部可观察 verifier、step-level diagnostics、closed-source GUI trajectory logging。
- 局限：仍局限于 Android 生态，且 benchmark 代表的是特定任务分布，不等于通用 computer-use 全覆盖。
- 阅读建议：看 benchmark task design 和 evaluator design，两者都比单纯榜单分数更有迁移价值。

### 4. AgenticVBench: Can AI Agents Complete Real-World Post-Production Tasks?
- 链接：[arXiv 页面](https://arxiv.org/abs/2605.27705) | [HTML](https://arxiv.org/html/2605.27705v1)
- 证据来源：paper page
- 核心方法：围绕真实视频后期流程构建 100 个 agentic 任务，混合程序化 verifier 与专家 rubric，同时对 vendor-native harness 和开源 harness 做对比评测。
- 创新点：它强调 harness 不是中性外壳，而是直接影响工具调用模式和失败形态的变量。这对你做 agent 系统研究很重要，因为很多能力差异可能来自 scaffold，而不只是 base model。
- 实验价值：最佳 agent stack 仅略高于 30%，远低于人类专家；不同 harness 对同一模型表现影响显著。
- 可转化实验/模块：benchmark 中的 harness ablation、task family 拆解、基于真实工作流的 rubric 设计。
- 局限：任务域集中在视频后期，多模态能力要求重，不一定直接映射到通用 office/browser agent。
- 阅读建议：把它当作“任务设计和 harness 设计论文”来读，比把它当单纯视频应用 benchmark 更值。

## Search Agent

### 5. Search for Coverage: Learning Coverage-Aware Retrieval with Augmented Sub-Question Answerability
- 链接：[arXiv 页面](https://arxiv.org/abs/2605.28522) | [HTML](https://arxiv.org/html/2605.28522v1)
- 证据来源：paper page
- 核心方法：提出 CoveR，用 coverage-based contrastive/distillation objectives 训练 dense retriever，不再只看单文档 relevance，而是看一组文档是否共同覆盖信息需求。
- 创新点：训练信号不是人工标注 coverage，而是通过 sub-question answerability 合成。SCOPE 数据集 90K 训练对这点尤其值得看，因为它提供了从长答案需求反推检索监督的实操范式。
- 实验价值：在 nugget coverage 上比强基线高 10%，同时不牺牲 relevance。对需要“多证据、多方面覆盖”的 search agent 来说，这比普通 top-k similarity 更接近真实目标。
- 可转化实验/模块：coverage-aware reward、sub-question decomposition、answerability-based synthetic supervision。
- 局限：当前主要是 long-form RAG 设定，还没直接进入 browser agent 的交互式检索闭环。
- 阅读建议：重点看数据构造与 loss 设计，它们最容易迁移到 search trace 训练。

### 6. Beyond Similarity: Task-Aligned Retrieval for Language Models
- 链接：[arXiv 页面](https://arxiv.org/abs/2605.27951) | [HTML](https://arxiv.org/html/2605.27951v1)
- 证据来源：paper page
- 核心方法：提出 TAG，把文档先转成 traceable condition-action rules，再让模型判断“哪条规则适用”，而不是按 embedding similarity 取 top-k。
- 创新点：这篇点中了一个很实用的 search/agent 问题：很多任务要找的不是“语义最像的上下文”，而是“真正可执行的规则或约束”。这对工具使用、代码规范、政策问答都很关键。
- 实验价值：在 RuleArena、HumanEval + PEP8、Wikipedia NPOV rewriting 上都优于标准 RAG，并显著缩小检索上下文。
- 可转化实验/模块：rule extraction、applicability judgment、低冗余上下文构造。
- 局限：依赖规则化表示的质量；对高度开放、无明显规范边界的检索任务未必同样有效。
- 阅读建议：如果你在做 query planning 或 answer verification，这篇比传统 reranker 论文更值得做 related work。

## Code Agent

### 7. Benchmarks are Not Enough: RAMP for Runtime Assessing of Agentic Models in Production Systems
- 链接：[arXiv 页面](https://arxiv.org/abs/2605.27492) | [HTML](https://arxiv.org/html/2605.27492v1)
- 证据来源：paper page
- 核心方法：构建 production-grounded runtime assessment infrastructure，用统一执行接口、串行依赖任务和 staged recovery 观察 agent 在真实工作流里的失败传播。
- 创新点：把 outcome quality 和 process efficiency 同时纳入评测，把运行过程本身当作一等分析对象，而不是只看最终 pass/fail。
- 实验价值：15 个模型在长串行工作流里普遍崩塌，前期 100% 完成率到末阶段掉到 20%，没有模型完成整条 pipeline。这非常像真实 SWE agent 部署时的问题。
- 可转化实验/模块：runtime telemetry、failure propagation analysis、recovery-stage curriculum、utility-oriented metrics。
- 局限：当前 workload 以编译器构建为核心，任务多样性仍需扩展到 DevOps、ML engineering、web service 等场景。
- 阅读建议：如果你做 code agent 训练闭环，这篇应该列入最高优先级。

### 8. Tool Forge: A Validation-Carrying Toolchain for Governed Agentic Execution
- 链接：[arXiv 页面](https://arxiv.org/abs/2605.28000) | [HTML](https://arxiv.org/html/2605.28000v1)
- 证据来源：paper page
- 核心方法：把 tool 从“函数签名”提升为“带 contract、implementation、dependency policy、tests、validation evidence、routing metadata 的 capsule”，并用 Router 只暴露 intent-scoped tool session。
- 创新点：一方面降低了模型看到的工具上下文，另一方面把验证和治理前移到了 tool 生成阶段。对 MCP/tool-use agent 来说，这是非常直接的系统层启发。
- 实验价值：Router 在 83 个基准案例上 micro-F1 0.901，同时估计减少 99.2% 工具上下文；25 个端到端 local-tool probe 中 23 个通过 live sandbox validation。
- 可转化实验/模块：validation-carrying tool artifact、tool lifecycle state、intent-scoped routing、sandbox-backed acceptance checks。
- 局限：论文自己也承认 benchmark 仍是初始系统 benchmark，不是 SOTA 声明；对真实外部 API grounding 和 adversarial routing 还要继续看。
- 阅读建议：这篇很适合作为“工具供应链”方向的系统论文起点。

### 9. GUI Agents for Continual Game Generation
- 链接：[arXiv 页面](https://arxiv.org/abs/2605.28258) | [HTML](https://arxiv.org/html/2605.28258v1)
- 证据来源：paper page
- 核心方法：提出 PlaytestArena 和 Play2Code。前者用 GUI agent 当自动 playtester，后者让 code agent 和 GUI playtester 在 shared memory 中持续闭环。
- 创新点：这篇最有价值的不是“做游戏”，而是提出一个非常干净的 execute-test-fix 训练/评测模板。GUI agent 给出的反馈比人工报告更可追踪，适合做自动闭环监督。
- 实验价值：Play2Code 达到 66.8% rubric pass-rate，显著优于单次生成和普通 agentic coding baseline。
- 可转化实验/模块：shared-memory multi-agent loop、GUI-based verifier、interactive artifact repair。
- 局限：当前环境是浏览器游戏，任务反馈密度高于很多真实软件工程问题；迁移到通用 repo 还需要额外工程。
- 阅读建议：把它当“交互式测试代理”论文看，价值会比看“游戏生成”更高。

### 10. Agentic Separation Logic Specification Synthesis
- 链接：[arXiv 页面](https://arxiv.org/abs/2605.27531) | [HTML](https://arxiv.org/html/2605.27531v1)
- 证据来源：paper page
- 核心方法：Spec-Agent 结合静态分析、runtime heap tracing、fuzz harness 扩展和 counterexample-guided refinement，为大型 C++ 仓库生成不同层级的形式化规格。
- 创新点：它把 repo-level 理解、formal spec synthesis、动态验证和 agentic refinement 串成闭环，而且目标语言覆盖到 separation logic，这对 memory-intensive systems code 很关键。
- 实验价值：在数百万行 C++ 代码上为 85% 目标函数合成有效规格，没有发现 fuzzing 下的假阳性，并宣称比 Claude Code Opus 4.6 低 10x token 成本。
- 可转化实验/模块：repo slicing、spec language routing、heap-trace-guided reasoning、counterexample-driven repair。
- 局限：主要针对 C++ 和规格合成，不是通用编码助手；但其“静态分析 + 动态验证 + LLM refinement”配方很可迁移。
- 阅读建议：如果你关心 verifiable code agent，这篇值得精读 method 和 evaluation。

## 今日重点趋势

1. 持久 runtime 正在取代“把一切塞回上下文”的 agent 设计。CaveAgent 是最直接代表。
2. agent safety 开始从 prompt-level guardrail 转向 typed / validated execution boundary。LACUNA 和 Tool Forge 都属于这个脉络。
3. 评测正在从静态 benchmark 转向 process-observable benchmark。RAMP、AndroidDaily、AgenticVBench、PlaytestArena 都强调轨迹、恢复、失败模式，而不是单个最终答案。
4. search agent 训练信号开始更贴近真实任务目标。CoveR 优化 coverage，TAG 优化 applicability，都不是传统 similarity retrieval 视角。
5. code agent 最有潜力的闭环还是“生成 -> 执行 -> 验证 -> 修复”。GUI Agents for Continual Game Generation 和 Spec-Agent 都给了高质量实现样板。

## 潜在研究 Idea

1. 用 CaveAgent 式持久 runtime 承载 Search Agent 的检索状态，把 query plan、evidence set、citation graph 作为对象而不是文本缓存。
2. 把 TAG 的 rule applicability 与 Tool Forge 的 tool capsule 结合，做“先判定哪个工具 contract 适用，再调用工具”的 retrieval-to-tool routing。
3. 用 RAMP / AndroidDaily 风格的 step-level diagnostics 训练 code agent recovery policy，而不只训练最终成功率。
4. 借鉴 CoveR 的 sub-question answerability 构造 search trajectory reward，把“覆盖更多有效证据”直接变成 RL/RLVR 信号。
5. 把 Spec-Agent 的 counterexample-guided refinement 迁移到通用 agent planning：计划失败后不是重写整条链，而是只修复被 verifier 指出的局部 hole。
