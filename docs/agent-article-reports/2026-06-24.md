## General Agent

### 1. Are We Ready For An Agent-Native Memory System?

- 来源：[arXiv](https://arxiv.org/abs/2606.24775) / [PDF](https://arxiv.org/pdf/2606.24775)
- 问题背景：很多 Agent memory 工作仍把记忆当成 RAG 插件，用最终任务分数判断好坏。论文指出这会遮住真正影响长期 Agent 的系统问题：记忆如何表示、如何抽取、如何路由检索、如何更新删除、如何控制成本。
- 核心方法：作者把 agent memory system 拆成四个模块：memory representation and storage、memory extraction、retrieval and routing、maintenance。这个拆法很有价值，因为它把“记忆效果不好”拆成可定位的工程问题，例如是抽取丢了事实，还是路由没找回证据，还是维护模块留下了过期事实。
- 创新点：它不是提出又一个 memory prompt，而是用数据管理视角评估 12 个代表性 memory systems 和两个 baseline。论文强调没有单一架构支配所有场景，结构化系统在某些长记忆 QA 上强，图方法适合单跳事实召回，但时间推理和动态更新会暴露弱点。
- 实验与 benchmark：覆盖 5 类 benchmark workloads、11 个数据集，报告 task effectiveness、retrieval fidelity、dynamic update robustness、long-horizon stability 和 operational cost。文中提到 LoCoMo、LongMemEval、DB-Bench 等工作负载；例如 Zep 在 LongMemEval 的 LLMJudge Accuracy 达到 48.0，MemOS 在 LoCoMo exact match 上达到 11.5，MemoChat 在 DB-Bench Task Success Rate 上达到 55.40。
- 可迁移价值：非常适合放进 Agent post-training/系统设计 related work。对代码执行或浏览器 Agent，可以把每条轨迹中的 observation、tool result、失败修复和用户偏好都按四模块拆开评测，而不是只看最终任务成功率。
- 局限：论文结论偏系统评测，尚未直接给出一个统一最优 memory runtime；不同 benchmark 的绝对分数仍偏低，说明当前 memory 系统离稳定长期代理还有距离。
- 阅读建议：优先读 taxonomy、RQ1-RQ5 和 cost-performance trade-off。它能直接启发“Agent 记忆模块消融实验”怎么设计。

### 2. MANGO: Automated Multi-Agent Test Oracle Generation for Vision-Language-Action Models

- 来源：[arXiv](https://arxiv.org/abs/2606.24815) / [PDF](https://arxiv.org/pdf/2606.24815)
- 问题背景：VLA 机器人任务的传统测试 oracle 往往只看最终状态，且需要人工为每个环境写符号规则。这种 oracle 很难定位中间步骤失败，也难迁移到新任务。
- 核心方法：MANGO 把复杂自然语言任务拆成可复用 atomic tasks，再为 atomic task 生成 simulator-grounded oracle definitions，最后把复杂指令分解成有序 atomic actions 和对应 oracle。系统用 Generator、Assessor、Judge 三类 agent 形成结构化反馈循环，生成、审查、裁决并迭代修正 oracle。
- 创新点：它把“评测 VLA agent 是否完成任务”从最终成功/失败，变成中间步骤可执行 oracle 库。对 Agent 训练尤其重要，因为奖励和诊断信号可以从 episode 末端前移到每一步行动。
- 实验与 benchmark：评估在 LIBERO_10 和 RoboCasa Humanoid Tabletop。论文设置 RQ1 生成能力、RQ2 与传统 symbolic oracle 的失败检测和定位比较、RQ3 组件消融、RQ4 初始复杂任务集合能否缩减。结果表明 MANGO 能生成可执行细粒度 oracle，失败检测数量接近 symbolic oracle，同时能定位失败步骤并提供更丰富诊断信息。
- 可迁移价值：可直接迁移到 Browser/Code Agent。把“点击按钮”“运行测试”“修改文件”都拆成 atomic actions，然后让 Generator/Assessor/Judge 生成可执行检查器，可以形成比最终 pass/fail 更密的 RL 信号。
- 局限：领域是机器人仿真，oracle 依赖 simulator state。迁移到 Web 或代码仓库时，需要定义可观测状态和动作谓词，否则 oracle 仍会退化成弱文本判断。
- 阅读建议：重点读 Module 1-3 的输入输出样例、RQ 设置和 ablation。

### 3. Automated Synthesis of RL Environments and Reward functions from Policy Narratives via Grammar-Constrained LLM Refinement

- 来源：[OpenReview](https://openreview.net/forum?id=YikG1amX3O) / [PDF](https://openreview.net/pdf/e4c7884b2d5bf44383d0b2460894172f8edb8e1e.pdf)
- 问题背景：把自然语言政策文本变成 RL 环境，需要定义 typed state space、action set、reward function 和 dynamics，通常依赖人工工程。对 Agent 训练来说，这正是“如何从任务说明自动生成可执行训练环境”的核心问题。
- 核心方法：NarraRL 使用 grammar-constrained LLM extraction，把政策条款映射到 clause-traceable 的 RLSpec schema，再用模板脚手架合成 Gymnasium 环境和奖励函数。关键不是让 LLM 直接写完整环境，而是先让输出受语法和 schema 约束，再用模板保证可执行性。
- 创新点：论文清楚地区分了 reward extraction 和 environment dynamics。实验显示，只抽出奖励还不够，必须把 LLM 抽取出的 action intent 接入 dynamics，环境才真正可学习。
- 实验与 benchmark：覆盖 17 个 UN SDG domain，使用 1.5B 到 3.8B 的小型开源 LLM。grammar-constrained extraction 的 Clause-Reward Overlap 比 unconstrained prompting 高 0.052，Wilcoxon p=1.4e-26。153 个条件下环境合成 pass rate 达 99%；模板脚手架在相同条件下 100% 通过，而 LLM-direct codegen 仅 2.4%。SDG 3 case study 中，加入 extracted action intent 和 opportunity-cost term 后，DQN 在 100% seeds 中满足至少 3/7 health targets。
- 可迁移价值：非常适合 Agent sandbox 数据生成。可以把 issue、网页任务、数据分析需求先抽成 typed spec，再由模板生成可验证环境、reward 和测试器。
- 局限：政策文本结构相对规范；迁移到真实网页或代码仓库时，状态空间和动作空间会更开放，需要更强的 schema 设计。
- 阅读建议：作为“环境合成 + reward 可信性 + executable validation”的范例精读。

### 4. "Zooming In" on Agentic Web Browsers as Assistive Technologies

- 来源：[arXiv](https://arxiv.org/abs/2606.24870) / [PDF](https://arxiv.org/pdf/2606.24870)
- 问题背景：Agentic Web Browsers 逐渐从效率工具变成替用户浏览和操作网页的系统。论文从低视力用户辅助技术角度观察 AWB，但更广泛的问题是：浏览器 Agent 的交互体验、控制权和失败恢复如何被用户感知。
- 核心方法：论文是一个低视力技术专家 case study，观察 AWB 如何通过自然语言理解网页内容、导航和执行浏览器操作。它没有提出训练算法，而是给出人机交互层面的需求和失败模式。
- 创新点：直接把 AWB 放到辅助技术语境中，强调流畅对话导航的潜力，也暴露了当前 agentic UX 的限制。对 Browser Agent 研究来说，它提醒我们评测不应只看 task success，还要看用户是否能理解 agent 当前状态、纠错和接管。
- 实验与 benchmark：非大规模 benchmark，主要是 case study。论文引用 A11y-CUA Dataset、Task Mode、Computer Use Agents 等相关工作，但本身没有提供标准量化指标。
- 可迁移价值：适合启发 Browser Agent 的人机边界设计，例如把页面状态摘要、动作计划、可撤销操作和用户确认机制变成评测维度。
- 局限：样本极小，证据强度弱于系统论文；因此排名低于 memory、oracle 和环境合成类工作。
- 阅读建议：作为 Browser Agent UX 和 accessibility 场景的设计参考，而不是算法基准。

## Search Agent

### 5. ChartWalker: Benchmarking the Cross-Chart RAG Task with Hierarchical Knowledge Graphs

- 来源：[arXiv](https://arxiv.org/abs/2606.23997) / [PDF](https://arxiv.org/pdf/2606.23997)
- 问题背景：很多 RAG benchmark 处理的是文本或表格，而现实搜索任务常要跨多个图表找证据、做比较、求和、趋势分析和多跳推理。现有 chart RAG 数据常靠 key-point 拼接，容易出现词面重合但推理链不成立。
- 核心方法：ChartWalker 先为图表构建 hierarchical knowledge graph，把实体和关系按粒度组织起来，再用 structure-aware sampling 合成语义连续的多跳 reasoning paths，最后生成带显式证据链的 QA。它不是简单随机游走，而是控制 query granularity、hop 和 source chart 数量。
- 创新点：它把跨图表 RAG 的数据构造从“检索相似文本”推进到“图结构约束的证据路径采样”。同时给出 ChartWalker-Agent，一个 VLM-based search agent，沿知识图谱迭代获取证据并做逐步推理。
- 实验与 benchmark：ChartWalker-Bench 包含 564 个 multi-hop QA，4 类 query：Fact Check、Manipulation、Analysis、Complex Reasoning。平均每题使用 2.30 个证据源、2.81 条 reasoning branches、4.52 hops；Complex Reasoning 平均 3.12 个 source charts 和 5.14 hops。实验显示最佳模型总体正确率约 64%，复杂推理多数设置低于 30%；静态 RAG 即使用 10 个 retrieved charts，最强模型也只有约 51% answer accuracy。
- 可迁移价值：很适合 Search Agent 训练数据构造。可把网页、表格、图表、PDF 中的证据转成层级 KG，再让 agent 学习何时继续检索、何时聚合、何时终止。
- 局限：图表实体抽取和 KG 构建本身可能引入错误；benchmark 规模中等，仍需更开放网页环境验证。
- 阅读建议：优先读数据构造流程、Table 2 统计和 ChartWalker-Agent action space。

### 6. Scaling Dense Retrieval with LLM-Annotated Training Data

- 来源：[arXiv](https://arxiv.org/abs/2606.23911) / [PDF](https://arxiv.org/pdf/2606.23911)
- 问题背景：生产搜索中点击日志有位置偏差，尾部 query 稀疏，人工标注又无法覆盖数亿 query-item 对。Search Agent 的检索模块如果只靠弱点击信号，很难学到稳定可迁移的相关性。
- 核心方法：论文用三路生产 retrieval systems 的分歧构造训练信号：系统都同意的是 easy positive，词法系统才能找到的是 hard positive，只骗过一个系统的是 hard negative。随后用 184M cross-encoder、LoRA 2B LLM、LoRA 8B LLM 组成 calibrated annotation cascade，生成 graded relevance 标签，再用 BCE、MNR、Triplet 三阶段 progressive curriculum 训练 two-tower BERT。
- 创新点：把“检索系统之间的不一致”转化为可规模化的难例采样机制，并用 LLM 标注级联替代一次性人工标注。对 Search Agent 来说，这是构造 retrieval reward/model 数据的实用蓝图。
- 实验与 benchmark：训练数据超过 240M examples，五级难度；LLM 标注级联与训练好的人类标注者达到 89.1% agreement。Walmart sponsored search 线上线下评估显示：NDCG@10 相对点击训练生产 baseline 提升 5.1%，tail queries 提升 6.8%；rating 0 的 embarrassing retrievals 从 8.7% 降到 3.5%；两周 A/B test 中 ad spend +2.80%、CTR +1.4%、eCPM +2.8%、click conversion +2.9%。
- 可迁移价值：可用于 Agent 搜索轨迹训练：从 BM25、dense、reranker、浏览器点击轨迹的分歧里挖 hard positives/negatives，再用强模型离线标注，训练轻量检索器或 reward model。
- 局限：强生产场景论文，数据不可复现；领域是电商广告搜索，迁移到开放 Web 需要重新定义 relevance scale。
- 阅读建议：重点看 structured mining 和 curriculum design。

### 7. Dialogue to Discovery: Attribute-Aware Preference Elicitation for Conversational Product Search Assistants

- 来源：[arXiv](https://arxiv.org/abs/2606.24194) / [PDF](https://arxiv.org/pdf/2606.24194)
- 问题背景：对话式搜索不能一直问问题，也不能过早推荐。用户耐心会被无关问题和差推荐消耗，而屏幕空间有限，不能把所有候选都展示出来。
- 核心方法：D2D 用产品属性结构决定下一步是继续问偏好，还是推荐候选。它用 attribute-value preference、attribute uncertainty 和 overlap-aware recommendation timing 来平衡信息增益与用户耐心。用户仿真中还显式建模 patience：高相关推荐和有信息量的问题增加耐心，无关或重复交互降低耐心。
- 创新点：把 conversational search 的策略问题写成“何时问、问哪个属性、何时推荐”的决策流程，而不是把完整任务交给 LLM 自由聊天。
- 实验与 benchmark：从 Amazon Reviews 构造 3 个域：Electronics 2009 items/112 attributes，Home and Kitchen 2001/143，Sports and Outdoors 2015/104。模拟对话中 D2D 相对强 baseline target-finding accuracy 提升 22.2-29.9%，abandonment 降低 6.6-16.1%，平均对话长度缩短 27.5%；还做了用户研究验证满意度和效率。
- 可迁移价值：对 Search Agent 的 query planning 很有启发。可以把网页调查任务里的“澄清问题、检索、呈现答案”也建模成带 patience/cost 的策略，而不是无限扩展搜索。
- 局限：电商商品属性结构清晰，开放域 research search 的属性空间更难枚举。
- 阅读建议：读 attribute selection 和 patience update，会启发多轮搜索 agent 的停止条件。

### 8. EvidenceLens: A Claim-Evidence Matrix for Auditing Financial Question Answering

- 来源：[arXiv](https://arxiv.org/abs/2606.23724) / [PDF](https://arxiv.org/pdf/2606.23724)
- 问题背景：LLM 金融问答会把有证据的事实、弱综合和无支持解释混在一段流畅答案里。普通引用链接只能说明“哪里可能相关”，不能说明“哪一句 claim 被什么证据支持或反驳”。
- 核心方法：EvidenceLens 把答案分解成 atomic claims，并把每个 claim 对齐到 text spans、table cells/rows、chart marks/time intervals。核心界面是 multimodal claim-evidence matrix，用矩阵展示支持覆盖、矛盾、模态失衡和 confidence-support gap。
- 创新点：它把可信 QA 从 retrieval problem 推进到 representation problem：即使证据已经找回，用户仍需要一种结构化表示来比较 claim 和证据关系。
- 实验与 benchmark：这是 IEEE VIS 2026 conditionally accepted short paper，主要是 prototype 和代表性 audit scenarios，没有完整大规模用户实验。论文提出后续评估方案：12-16 个 report questions，任务包括找 unsupported claims、定位 contradictory evidence、判断答案接受/修改/拒绝，并比较 answer text only、answer plus citations、claim list、full matrix 四种条件。
- 可迁移价值：非常适合 Search Agent 的 citation verifier 和 answer auditor。可以把 agent 输出先拆成 claims，再要求每个 claim 绑定证据、模态、支持/矛盾状态和可复核 source anchor。
- 局限：上游 claim extraction 和 evidence alignment 可靠性仍是瓶颈；长答案超过 20 个 claims 时矩阵需要聚类或折叠。
- 阅读建议：作为“答案审计 UI + 证据结构化 schema”的 related work。

## Code Agent

### 9. Automated Summarization of Software Documents: An LLM-based Multi-Agent Approach

- 来源：[arXiv](https://arxiv.org/abs/2606.24689) / [PDF](https://arxiv.org/pdf/2606.24689)
- 问题背景：软件文档、需求和应用描述经常冗长、结构不统一，Code Agent 在仓库理解时需要把这类材料压缩成可靠摘要。单模型 summarization 容易遗漏关键信息或产生泛化描述。
- 核心方法：Metagente 是 LLM-based multi-agent summarization system，使用 Teacher-Student 架构，多 Agent 协作提升摘要相关性和精度。论文扩展了 dynamic iteration strategy，对低潜力样本提前停止，降低计算成本；也扩展到多种 LLM 推荐引擎，并评估 HTML-based application descriptions 的摘要。
- 创新点：它的价值不在“又用 LLM 做摘要”，而在把摘要过程拆成可迭代评审的多 Agent 流水线，并加入动态停止。这对代码仓库 Agent 的 readme、issue、API docs、需求文档摘要都有工程参考。
- 实验与 benchmark：论文报告在 curated datasets 上，Metagente 有较强 semantic alignment 和有竞争力的 ROUGE 分数，并且需要更少训练迭代。PDF 中摘要层面没有给出足够清晰可直接摘录的全部数据集细节，因此这里不编造具体数值。
- 可迁移价值：可作为 Code Agent 的“文档压缩器”模块，把 teacher critic、student summarizer 和动态停止接入长上下文仓库理解。
- 局限：任务仍是 summarization，离代码执行闭环和 issue resolution 还有距离；实验可复现细节需要进一步查 replication artifact。
- 阅读建议：读 architecture 和 dynamic iteration，比看 ROUGE 表更有价值。

### 10. SemChunk-C: Semantic Segmentation for C Code

- 来源：[arXiv](https://arxiv.org/abs/2606.23697) / [PDF](https://arxiv.org/pdf/2606.23697)
- 问题背景：C/C++/C# 文件有宏、嵌套定义和不规则结构，固定窗口或纯语法切分容易把功能单元切碎，影响代码检索、RAG 和后续代码生成。
- 核心方法：论文先定义代码 chunk categories，再训练 LLM-based classifier 来识别 chunk boundaries 并分配功能属性。最终提出 SemChunk-C，一组基于 Ettin encoder 的轻量模型，参数量 17M、32M、68M、150M，用于 C-family 文件语义切分。
- 创新点：它把代码 chunking 从 token 长度规则改成“功能单元 + 语义类别”的监督任务，且用小模型降低本地部署成本。
- 实验与 benchmark：论文称在多个数据集上测试，boundary accuracy 和 semantic coherence 能匹配或超过更大的 code-oriented LLM chunkers，并在若干 curated downstream benchmarks 上验证后续任务收益。PDF 前 7 页没有足够可摘录的完整 benchmark 表细节，因此本日报只引用摘要和正文中可核验的描述。
- 可迁移价值：对 Code Agent 的仓库级检索非常直接。先用 SemChunk 类模型把 C/C++ 仓库切成 interface block、data structure、function unit，再做 embedding 和 bug localization，通常比固定窗口更适合长仓库。
- 局限：目前专注 C-family；对 Python/TypeScript 的 AST 与语义边界需要重新定义标签。
- 阅读建议：适合作为 Code Agent retrieval preprocessing 的相关工作，和长上下文 chunking/索引策略一起读。

## 今日重点趋势

1. Agent memory 正在从“把历史塞进向量库”转向完整生命周期系统：表示、抽取、检索路由、维护和成本都需要单独评测。
2. 可执行环境与 oracle 合成是 Agent post-training 的高价值方向。NarraRL 和 MANGO 都说明 schema、模板、validator、结构化反馈比直接让 LLM 自由生成更稳定。
3. Search Agent benchmark 正在变难：ChartWalker 要求跨图表、多跳、多源证据聚合，EvidenceLens 要求 claim-level 可审计证据，而不是粗粒度 citation。
4. 生产检索数据构造开始把系统分歧、LLM 标注和 curriculum 结合，值得迁移到 Web search trajectory 数据生成。

## 潜在研究 idea

- 把 Agent trajectory memory 按 Agent-Native Memory 的四模块拆开，分别评测 retrieval fidelity、update correctness、long-horizon stability 和 token/latency cost。
- 用 NarraRL 的 grammar-constrained schema + MANGO 的 Generator/Assessor/Judge，为 Browser/Code Agent 自动生成 step-level oracle。
- 用 ChartWalker 的 hierarchical KG path sampling 构造 multi-source web research benchmark，让 Search Agent 学习动态取证而不是一次性 top-k。
- 把 EvidenceLens 的 claim-evidence matrix 改成自动 evaluator：每个 final answer claim 必须绑定 source span、table cell、chart mark 或 execution log。
- 用 SemChunk-C 思路为仓库级 Code Agent 训练小型 semantic chunker，减少固定窗口检索带来的上下文污染。
