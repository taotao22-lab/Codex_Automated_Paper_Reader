## General Agent

### 1. CaveAgent: Transforming LLMs into Stateful Runtime Operators
- 链接：[OpenReview](https://openreview.net/forum?id=p3dlOhpqKD) | [PDF](https://openreview.net/pdf/4713bd00b2f65526ef18a6ba85587520fa080983.pdf)
- 证据来源：PDF
- 相关性判断：今天最值得看的论文。它直接讨论 tool use、persistent runtime、stateful object management、skill injection、verifiable runtime feedback，几乎正对 General Agent 和 Code Agent 的交叉地带。
- 核心方法：把传统 “LLM 产出 JSON / 文本，再由外部框架执行” 改成 “LLM 产出作用于持久 Python runtime 的代码”。论文的双流结构里，semantic stream 负责轻量推理，runtime stream 负责对象状态、工具执行和跨轮保持。关键不是“会写代码”本身，而是把 `DataFrame`、数据库连接等复杂对象直接放进长期存在的运行时，而不是反复序列化回上下文。
- 创新点：一是把 stateful runtime 当成 agent 的主工作区，而不是把文本上下文当主工作区。二是把 skill 管理和 runtime 注入绑定，允许工具能力以可执行对象而不是纯提示词形式进入系统。三是把“可程序验证的运行时状态”明确当作未来 RLVR 的反馈源，这一点对 post-training 非常重要。
- 实验与结果：Tau2-bench 上 12 组设置里 11 组优于原生 function calling；Qwen3-Coder 30B 在 Retail 任务上提升到 `+13.5%`。BFCL 上，3 个开源模型在 CaveAgent 范式下达到 `94.0-94.7%`，接近 Claude Sonnet 4.5 和 Gemini 3 Pro，并超过论文中报告的 GPT-5.1 原生 function-calling 结果。论文还报告总 token 消耗下降 `28.4%`，数据密集任务最高下降 `51%`。
- 可转化实验 / 模块：持久对象寄存区、runtime object registry、skill runtime injection、基于运行时状态的 verifier / reward 抽取、从 JSON 协议迁移到 code-action 协议的 A/B 测试。
- 局限：论文页面显示 `Withdrawn by Authors`，因此应把它当成高价值技术线索，而不是稳定结论。实验也主要集中在 Tau2 / BFCL 与数据查询类任务，离真实浏览器、多仓库代码修复、GUI agent 的开放环境还有距离。
- 阅读建议：如果你只精读一篇，就读这篇。优先看 introduction 对 JSON function calling 的三个限制、方法部分的 Stateful Runtime Management、以及实验里 BFCL 和 token efficiency 的论证。

### 2. The Embodiment Gap in Robot Foundation Models
- 链接：[OpenReview](https://openreview.net/forum?id=D0XcH9Cso4) | [PDF](https://openreview.net/pdf/227fc57825d5883dc3f39593ff9a9b75169364f2.pdf)
- 证据来源：PDF
- 相关性判断：不是 Agent 主线论文，但它提出的“shared structure 与 target-side residual burden”分析框架，对 General Agent 的系统设计和评测报告写法有迁移价值。
- 核心方法：这是一篇 lens-driven survey，不是新模型。作者把跨机器人泛化问题拆成两个轴：`shareability level` 和 `dominant residual locus`，再围绕 semantic scale、physical-data scale、embodiment correspondence scale 三条扩展方向组织文献。
- 可借鉴点：它要求论文报告目标机体上的 residual adaptation burden，包括 target data、model update、calibration/setup、real-robot operation、safety/recovery。这套“把残余成本显式报出来”的思路，可以迁移到 computer-use agent、browser agent、code agent 的部署与评测中。
- 局限：本质上是机器人基础模型综述，没有直接提供 Search Agent / Code Agent 训练数据、工具轨迹或验证器设计。
- 阅读建议：适合当“评测与报告框架”参考，不适合作为今天的主推荐。

## 边缘候选说明

### 3. The Impact of Semantic Pairs on Self-Supervised Representation Learning
- 链接：[OpenReview](https://openreview.net/forum?id=7RgiccEWpN) | [PDF](https://openreview.net/pdf/fe1db61347752b39e8410c1eb78d316740e62086.pdf)
- 证据来源：PDF
- 核心内容：作者手工构造 `187` 个类别、每类 `157` 对的 semantic pairs 数据集，与完全匹配规模的 augmented-pair 基线对比，在 ResNet50、200 epoch 预训练、相同超参下比较 SimCLR、BYOL、DINO、VICReg 等方法。结果显示 semantic-pair 预训练在 transfer learning 与 VOC 检测上稳定优于 augmented-pair 预训练，其中 SimCLR 相对受益最大。
- 为什么分数低：它能启发“正样本构造影响泛化”的数据工程思路，但与 Agent 训练、工具使用、搜索轨迹、代码执行或 verifier 设计没有直接连接。
- 唯一可迁移启发：如果未来要做 tool-use / code / browser trajectory 数据筛选，可以借鉴它的 matched-control 思路，严格隔离“数据配对方式”这个变量，而不是把多种清洗策略混在一起比较。

## 今日重点趋势

1. 真正值得关注的不是“多一个工具接口”，而是把运行时状态、技能注入和验证反馈做成统一协议层。`CaveAgent` 在这点上很强。
2. 即使候选池很小，仍然能看出一个研究判断标准：没有 execution loop、没有中间态可验证性、没有工程接口设计的论文，不该因为关键词沾边就拿高分。
3. 综述型工作如果能把 residual burden、失败归因、部署成本写成显式报告框架，依然对 Agent 系统论文有方法论价值。

## 潜在研究 Idea

1. 把 `CaveAgent` 的 persistent runtime 与代码代理的 patch-test-fix 闭环结合，直接把测试产物、AST、coverage、failing traces 存成长期对象，而不是来回塞进上下文。
2. 把论文中的 “programmatically verifiable runtime state” 扩展成 Search Agent / Code Agent 的训练反馈层，用运行时对象一致性、证据覆盖率、测试状态变化来生成 reward。
3. 借鉴 `Embodiment Gap` 的 residual-burden 报告方式，为 computer-use / browser / code agent 设计一套“目标环境残余成本”卡片：额外人工修复、环境重置、登录/权限、失败恢复、工具适配成本。
