## 方向索引

### General Agent

AgentTether、LogicHunter、PatchOptic 和 UI2App 都在回答同一个工程问题：真实 Agent 系统不能只依赖最终回答或最终 patch，要把中间状态、工具调用、依赖关系、输入输出契约和失败传播显式建模。AgentTether 偏运行时修复，LogicHunter 偏 Agent 框架测试，PatchOptic 偏共享状态更新，UI2App 偏可执行 Web 应用生成与视觉交互推断。

### Search Agent

DynaKRAG、PORTS 和 ParaSet 是今日 Search/RAG 方向主线。它们分别处理多跳检索中的证据动作选择、大工具集合里的工具检索，以及多跳问答里的证据集合兼容性。共同点是把“检索”从独立文档打分推进到动作级、集合级或偏好优化级决策，更接近 Search Agent 的轨迹训练信号。

### Code Agent

LogicHunter、TraceProbe、RuBench 和 Agents That Teach 构成 Code Agent 方向。前两篇最值得精读：LogicHunter 面向 LangChain/LlamaIndex/CrewAI 这类基础设施做主动测试和 Agentic Oracle；TraceProbe 把 SWE-bench 轨迹标准化，解释 resolve rate 背后的搜索、阅读、编辑和验证行为。RuBench 补足非英语仓库级任务，Agents That Teach 则提醒 coding agent 交互设计不能只优化自动完成率。

### 1. AgentTether: Graph-Guided Diagnosis and Runtime Intervention for Reliable LLM Agent Operation

- 类别：General Agent / Runtime Repair
- 链接：[2607.06273](https://arxiv.org/abs/2607.06273)
- 综合评分：4.9

**问题背景与核心方法**：这篇论文解决的是多步、有状态、带工具调用的 LLM Agent 失败后如何自动定位和修复。普通重试不知道错在哪里，最终结果反馈只告诉你失败了，自反思又容易缺少运行证据。AgentTether 把一次运行拆成 Transition Units，再用 Critical Transition Graph 表示步骤之间的数据依赖和因果传播；诊断时不是按日志顺序猜错，而是沿着依赖图定位最可能导致后续失败的关键子轨迹。修复时，它不修改底层 Agent 或环境，而是在工具返回或合成用户消息里注入最小干预，并用 evidence grounding、cooldown 和 minimal-intervention guard 避免过度控制。

**创新点**：它把 Agent repair 从“重新问模型”变成了“图结构诊断 + 运行时受控干预”。论文还把成功轨迹中的工具调用、状态证据和依赖边映射成统一 telemetry schema，用 TerminalBench、SWE-smith 等外部成功轨迹训练结构视图，这对构造可迁移的 tool-use 轨迹奖励很有启发。

**实验与 benchmark**：正文显示评测使用 τ-bench，覆盖 Retail、Airline、Banking 三个域共 261 个任务，主干模型包括 Qwen3.7-max，并在 Banking 上用 GPT-5.4 交叉验证。τ-bench 测的是真实服务流程里的工具-agent-用户交互，强调过程约束而不只是最终回答。论文还比较 blind retry、反馈修复和 AgentTether 组件消融；具体表格数值本次只从 PDF 文本片段核实到“能修复相当比例初始失败”，未逐格摘录。

**可迁移价值与局限**：最可迁移的是 Transition Unit + dependency graph 表示，可用于 Code Agent 的失败归因、Search Agent 的证据链修复，以及 RL 后训练中的过程奖励。局限是它依赖外部 analyst/verifier、结构化日志和可注入干预点；真实产品 Agent 若工具状态不可回放，落地成本会更高。

**阅读建议**：优先读 abstract、诊断图构造、repair protocol 和 τ-bench 实验设置。证据来源：PDF 抽取正文。

### 2. LogicHunter: Testing LLM Agent Frameworks with an Agentic Oracle

- 类别：Code Agent / Agent Framework Testing
- 链接：[2607.06195](https://arxiv.org/abs/2607.06195)
- 综合评分：4.8

**问题背景与核心方法**：LangChain、LlamaIndex、CrewAI 已经是生产 Agent 的关键基础设施，但传统测试很难覆盖这类框架。原因是错误常表现为普通异常、静默语义失败、Pydantic schema 约束冲突或外部服务 mock 组合，而不是简单 crash。LogicHunter 用 specification-aware generation 先生成更贴近真实 API 契约的测试输入，再用 Agentic Oracle 主动诊断失败：检索文档、检查运行时状态、执行目标化验证，并通过多次高置信共识降低 LLM oracle 的随机误报。

**创新点**：它的核心不是“让 LLM 写测试”，而是把测试生成和故障判定都嵌入框架规范、文档检索和运行时检查中。这样能把“这个异常是用户误用还是框架 bug”的模糊判断转化为可审计诊断流程。

**实验与 benchmark**：评测对象是 CrewAI、LangChain、LlamaIndex，正文片段列出 CrewAI 46 个 API、LangChain core 302 个 API、LlamaIndex core 264 个 API；对照 Pynguin、Hypothesis、Atheris、OSS-Fuzz-Gen 等工具。Oracle 评测构造了 1,000 个 failed test case 语料，来自 LangChain 400、LlamaIndex 400、CrewAI 200，并由开发者标注。论文报告发现 40 个此前未知 bug，其中 30 个被确认、26 个修复；Agentic Oracle precision 为 91.17%，比基线高 61 个百分点，基线发现 0 个 bug。

**可迁移价值与局限**：这篇非常适合转化为 Code Agent 训练数据：输入是 API 规范、测试轨迹、失败栈、文档证据和 oracle 判定，输出可以是 bug 分类、最小复现或修复建议。局限是成本较高，且对文档质量、mock 设计和外部服务行为建模有依赖。

**阅读建议**：优先读 Section 3 Approach、4.1 evaluation setup、oracle corpus 构造和 bug taxonomy。证据来源：PDF 抽取正文。

### 3. What Resolve Rate Hides: Trajectory Structure Diagnostics for Coding Agents

- 类别：Code Agent / Trajectory Diagnostics
- 链接：[2607.06184](https://arxiv.org/abs/2607.06184)
- 综合评分：4.71

**问题背景与核心方法**：SWE-bench 类评测通常只看 resolve rate，但同样通过的两个 Agent 可能一个很快定位函数并验证，另一个绕了大量无效搜索；同样失败的轨迹也可能败在搜索、定位、编辑、验证或回滚。TraceProbe 把原始轨迹标准化成九类动作和确定性 effect label，再用 Insight 识别单轨迹反模式，例如搜索循环、跳过验证；用 Converge 对齐成败轨迹，定位行为分歧发生在文件、函数、验证还是补丁范围。

**创新点**：它把 Code Agent 评测从最终 patch 二值标签扩展到过程诊断。这个转变对训练很重要：resolve rate 只能提供稀疏奖励，轨迹结构能给出“哪里开始偏离”的密集监督。

**实验与 benchmark**：论文应用于 SWE-Bench Verified 上 5 个生产设置的 2,500 条轨迹，并做了 SWE-Bench Pro 的跨 benchmark 迁移检查。正文片段显示它关注 searches、reads、edits、tool calls、validation、reversions 等动作，并指出 file choice 太粗，function selection 和 completion behavior 更能定位成败差异。

**可迁移价值与局限**：可直接迁移为 Code Agent 训练数据清洗模块：对成功轨迹也打过程质量分，对失败轨迹生成定位标签。局限是规则和阈值需要针对不同执行环境校准；如果 Agent 工具日志粒度不足，九类动作映射会丢信息。

**阅读建议**：读 taxonomy、Insight/Converge 两个模块和 SWE-Bench Verified 分析。证据来源：PDF 抽取正文。

### 4. PatchOptic for Shared-State LLM Workflows with Projected Views and Verified Structured Updates

- 类别：General Agent / Shared State Runtime
- 链接：[2607.05483](https://arxiv.org/abs/2607.05483)
- 综合评分：4.52

**问题背景与核心方法**：Agentic workflow 往往共享结构化状态，但每次 LLM 调用只能看到局部 view。常见做法用 grep 或 RAG 拼上下文，问题是模型可能误改不可见状态，或输出无法可靠合并回全局状态。PatchOptic 把共享状态访问拆成 projected views 和 verified structured updates：模型只看到当前步骤需要的投影视图，更新必须通过结构化补丁形式提交，并经过验证后才能写回全局状态。

**创新点**：它把 progressive disclosure 从“少给点上下文”提升为“有类型、有验证、有写回协议”的 runtime 机制。这对多 Agent 协作、长流程任务和可审计状态更新特别重要。

**实验与 benchmark**：PDF 正文确认论文讨论 shared-state LLM workflows、projected views、verified structured updates，并把 grep-like keyword search/RAG 视为现有上下文构造方式。抽取文本没有完整呈现具体 benchmark 表格，因此不摘录未核实数值。

**可迁移价值与局限**：可作为 Agent 训练环境的状态层：把每一步 action 约束为对局部 view 的结构化 patch，执行结果天然可验证。局限是需要工程侧定义 schema、projection 和 validator；开放域网页/代码任务中状态 schema 可能动态变化。

**阅读建议**：重点读 view 构造、patch schema、验证规则和失败处理。证据来源：PDF 抽取正文。

### 6. DynaKRAG: A Unified Framework for Learnable Evidence Control in Multi-Hop Retrieval-Augmented Generation

- 类别：Search Agent / Multi-Hop RAG
- 链接：[2607.06507](https://arxiv.org/abs/2607.06507)
- 综合评分：4.43

**问题背景与核心方法**：多跳 RAG 不是一次检索就结束，新文档可能暴露缺失事实、桥接实体、查询缺陷或已经足够回答。现有系统把 query reformulation、evidence critique、sufficiency judging 等操作做成分散模块，缺少统一的可学习控制策略。DynaKRAG 把证据控制看作动态决策过程：根据当前证据状态决定继续检索、改写查询、批判证据还是停止回答。

**创新点**：它把 RAG pipeline 中的“检索几次、何时停、怎样改查询”显式建模为 learnable evidence control。这比固定轮数或手写规则更适合成为 Search Agent 轨迹策略。

**实验与 benchmark**：候选摘要确认其面向 multi-hop RAG，并统一 iterative retrieval、query reformulation、evidence critique、sufficiency judging 等操作。PDF 抽取未稳定呈现完整 benchmark 数值；本日报不编造具体数据集成绩。

**可迁移价值与局限**：可迁移为 Search Agent RL 的 action space：retrieve、rewrite、critique、judge sufficiency、answer。局限是训练信号如何定义仍关键，如果只用最终 QA 正确率，可能学到短视停止策略。

**阅读建议**：重点读 evidence state 表示、动作集合和训练/评估协议。证据来源：PDF 抽取正文 + 摘要。

### 7. PORTS: Preference-Optimized Retrievers for Tool Selection with Large Language Models

- 类别：General Agent / Tool Retrieval
- 链接：[2607.05441](https://arxiv.org/abs/2607.05441)
- 综合评分：4.37

**问题背景与核心方法**：工具集合变大后，把所有工具塞进 prompt 会拖慢、变贵，也会干扰选择。PORTS 把工具选择前置为 retrieval 问题，并用 preference optimization 训练 retriever，让它不仅按文本相似度找工具，还按 LLM 使用偏好和任务成功信号排序工具。

**创新点**：它把 tool selection 的瓶颈从 prompt engineering 转成可训练 retriever。对 Agent 后训练而言，这提供了一个很清楚的数据接口：任务、工具描述、候选工具、偏好对、最终执行结果。

**实验与 benchmark**：PDF 正文确认论文围绕 large tool collections 的 retrieval-based pre-selection，目标是降低输入长度和延迟，并用 preference optimization 改进工具检索。抽取文本未稳定呈现完整 benchmark 名称和数值，因此只记录已核实机制。

**可迁移价值与局限**：可用于构造工具路由器或作为 Agent policy 的第一阶段。局限是如果工具效果依赖状态和组合调用，单步工具偏好可能不足，需要扩展到工具序列偏好。

**阅读建议**：重点读偏好数据构造、retriever 训练目标和工具集合规模实验。证据来源：PDF 抽取正文。

### 9. UI2App: Benchmarking Visual Interaction Inference in Executable Web Application Generation

- 类别：General Agent / GUI-Code Agent
- 链接：[2607.06306](https://arxiv.org/abs/2607.06306)
- 综合评分：4.1

**问题背景与核心方法**：图像驱动的网页生成比文本 prompt 更能表达布局和跨页视觉一致性，但真正的应用不只是静态截图，还包含交互、状态变化和页面间行为。UI2App 关注 Visual Interaction Inference in Executable Web Application Generation：从 UI 截图或视觉信息推断可执行 Web App 的交互逻辑。

**创新点**：它把 Web 生成评测从“看起来像不像”推进到“能不能运行、能不能响应交互”。这对 Computer-use/GUI Agent 很重要，因为 Agent 需要理解控件意图、状态转移和可执行反馈。

**实验与 benchmark**：候选摘要确认其 benchmark 视觉交互推断和 executable web application generation。PDF 抽取文本显示其讨论 text-driven 与 image-driven approaches 的局限，但本次未稳定核实完整 benchmark 表格和指标，故不写具体成绩。

**可迁移价值与局限**：可迁移为 GUI Agent 的环境合成和验证任务：从视觉界面推断动作语义，再用浏览器运行结果验证。局限是偏前端生成，和通用工具调用 Agent 之间还需要动作空间对齐。

**阅读建议**：重点读 benchmark 任务定义、交互标签、执行验证方式。证据来源：PDF 抽取正文 + 摘要。

### 10. Retrieving a Set, Not Independent Passages: Set-Level Compatibility Learning for Efficient Set Exploration

- 类别：Search Agent / Evidence Set Retrieval
- 链接：[2607.05712](https://arxiv.org/abs/2607.05712)
- 综合评分：4.1

**问题背景与核心方法**：多跳问答需要的是一组互相配合的证据，而不是若干独立高分 passage。独立打分会错过“单篇看不强，但和另一篇组合后能回答问题”的证据。论文把检索建模为 query-set compatibility scoring，提出 ParaSet 作为轻量 late-interaction scorer，用预计算 bi-encoder embedding 上的 self-attention 快速探索候选集合；再用 SetCE 做 cross-encoder rerank，二者都用集合级目标训练。

**创新点**：它将检索单位从 passage 提升到 evidence set，并强调效率：相比 LLM-based set selector，正文片段显示 ParaSet+SetCE 竞争力接近或更强，同时约快 17.5 倍。

**实验与 benchmark**：正文片段列出 HotpotQA、2WikiMultihopQA、MuSiQue 等多跳 QA benchmark，指标包括下游 QA EM/F1、检索效果以及 memory/latency。HotpotQA 测维基多跳证据组合，2WikiMultihopQA 强调跨实体多跳推理，MuSiQue 强调组合式多跳问题。

**可迁移价值与局限**：可用于 Search Agent 的证据选择模块，尤其适合把搜索轨迹训练成“选证据集合”而不是“选下一篇文档”。局限是它仍以 QA gold evidence 为中心，开放网页搜索里的噪声、时效性和引用可信度需要额外建模。

**阅读建议**：重点读 query-set compatibility、ParaSet/SetCE 训练目标和 HotpotQA/2Wiki/MuSiQue 实验。证据来源：PDF 抽取正文。

## 今日重点趋势

1. Agent 评测从 outcome-only 转向 trajectory-aware。AgentTether 和 TraceProbe 都说明最终成功率不够，必须保留动作、依赖、验证和回滚信息。
2. Agent 基础设施本身成为测试对象。LogicHunter 把 LangChain、LlamaIndex、CrewAI 当成被测软件，这对生产 Agent 生态很关键。
3. Search/RAG 正在从相似度检索升级为策略控制。DynaKRAG、PORTS、ParaSet 分别把证据动作、工具选择和证据集合变成可学习对象。
4. Benchmark 的边界在扩展：WebRetriever、RuBench、UI2App 分别补网页任务规模、多语言仓库任务、可执行 GUI 交互。

## 潜在研究 idea

- 用 TraceProbe 式 action taxonomy 清洗 Code Agent 轨迹，再把 AgentTether 式依赖图作为失败归因标签，形成“失败根因定位 + 修复动作选择”的后训练数据。
- 把 PORTS 的工具偏好检索扩展成多步工具序列偏好：不仅选第一个工具，还预测后续工具链和停止条件。
- 用 ParaSet 的 evidence-set 目标训练 Search Agent，让模型直接优化证据组合完整性，而不是单文档相似度。
- 在 PatchOptic 的 structured update 上做 RL：只有通过 validator 的状态 patch 才给奖励，减少长流程 Agent 的隐性状态污染。
