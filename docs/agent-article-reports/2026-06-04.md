## General Agent

### 1. Self-Reflective APIs: Structure Beats Verbosity for AI Agent Recovery
- 链接：[arXiv](https://arxiv.org/abs/2606.05037) | [PDF](https://arxiv.org/pdf/2606.05037)
- 证据来源：PDF
- 核心方法：把 API 报错从“给人看的自然语言诊断”改成“给 agent 直接修复下一步请求的结构化恢复建议”。论文的关键不是更长的报错文案，而是 `recovery_feedback.suggestions[]` 这类机器可执行反馈接口。
- 创新点：这篇真正击中了工具使用闭环里最常见却经常被忽略的问题。很多 agent 失败并不是不会规划，而是在 tool call 失败后只能靠先验乱猜。作者用 API 结构设计而不是模型升级来解决这个恢复环节。
- 关键发现：在 30 个对抗任务、3 个模型的 leak-audited 试验里，结构化恢复建议相对 plain-English diagnosis 带来约 `+36.7` 到 `+40.0` 个百分点的任务完成率提升，并且 token-per-success 更优。
- 可转化实验：可以直接在 browser/code agent 工具接口里加入 `repair_suggestions`、`constraint_origin`、`retry_patch` 三类字段，对比同一模型在有无结构化恢复反馈下的重试成功率和 token 消耗。
- 局限：实验规模还不算大，且收益在 `gpt-4o-mini` 上不显著；更像高价值接口论文，而不是大规模 benchmark。
- 阅读建议：如果你要做 tool-use reasoning、computer-use harness 或 agent runtime，这篇应该优先于单纯提升模型能力的论文。

### 2. Token Budgets: An Empirical Catalog of 63 LLM-Agent Budget-Overrun Incidents, with an Affine-Typed Rust Mitigation as a Case Study
- 链接：[arXiv](https://arxiv.org/abs/2606.04056) | [PDF](https://arxiv.org/pdf/2606.04056)
- 证据来源：PDF
- 核心方法：先做 63 起真实 agent 成本失控事故的经验目录和八类故障分类，再用 affine-typed Rust budget primitive 把“预算不能重复委托、不能双花、不能越权复用”变成编译期约束。
- 创新点：这篇的价值不只是“加一个花费上限”，而是把成本治理从 runtime wrapper 上升到程序语义层。对多 agent 委托来说，这比事后 callback 或日志告警更硬。
- 关键发现：单 agent 情况下简陋 Python counter 也能守住 cap，但多 agent delegation fanout race 是分水岭；论文给出的 Rust 设计能在编译期拒绝那类模式，而 asyncio 版本在对应实验里 `30/30` 超支。
- 可转化实验：可以把 budget token、API quota、tool-call 次数、外部写权限都抽象成 affine resource，在 agent runtime 中统一做 capability 设计。
- 局限：二进制层面的 cap-soundness 还没被完全形式化证明；此外 reasoning-token 计费模型仍部分依赖 provider 侧控制。
- 阅读建议：适合和权限治理、审批流、sandbox 设计一起读，是很强的“agent 生产事故 -> 类型化约束”范例。

### 3. Proof-Carrying Agent Actions: Model-Agnostic Runtime Governance for Heterogeneous Agent Systems
- 链接：[arXiv](https://arxiv.org/abs/2606.04104) | [PDF](https://arxiv.org/pdf/2606.04104)
- 证据来源：PDF
- 核心方法：把治理中心从厂商 runtime session record 挪到可移植的 `action certificate` 上，用 route-review-prove 流程在异构运行时之间维持统一的审批、外部性描述和可回放证据。
- 创新点：论文抓到一个很现实的问题：同一个高风险动作在 shell、SDK tool call、hosted agent session 里表面形态不同，但治理问题相同。它提供的是跨 runtime 的稳定治理对象，而不是某一家平台的 guardrail 配置。
- 可转化实验：可以把 coding agent 的 git push、browser agent 的外发动作、search agent 的外部发布统一编码成 action certificate，看不同 runtime 下的审批保真与审计回放差异。
- 局限：实验基于受保护 benchmark 和参考实现，离通用行业标准还有距离；论文更偏系统治理而不是大样本 benchmark。

### 4. Provably Auditable and Safe LLM Agents from Human-Authored Ontologies
- 链接：[arXiv](https://arxiv.org/abs/2606.04903) | [PDF](https://arxiv.org/pdf/2606.04903)
- 证据来源：PDF
- 核心方法：提出 `Agentic Redux`，用 meta-agent 约束全局状态转移必须保持不变量，并把全部决策线性写入 append-only ledger，再结合 Ontology-First Agent Design 让人工本体定义先于代理角色生成。
- 创新点：这篇把 agent architecture 当作可证明对象，而不是把安全全留给运行时 guardrail。它对“先定义可证明状态机，再让 LLM 在边界内工作”的方向有明显启发。
- 可转化实验：可把仓库修改、审批、工具调用都表示成带不变量的 typed action system，对比“LLM as judge”与“typed meta-agent adjudication”的错误模式。
- 局限：目前更像强理论框架和领域化案例，缺少广泛 benchmark 对比；很多现实问题仍需 defense-in-depth 辅助层。

## Search Agent

### 5. QO-Bench: Diagnosing Query-Operator-Preserving Retrieval over Typed Event Tuples
- 链接：[arXiv](https://arxiv.org/abs/2606.04646) | [PDF](https://arxiv.org/pdf/2606.04646)
- 证据来源：PDF
- 核心方法：把跨文档问答重新定义为 `自然语言表达的数据库算子执行`，用 typed event tuples 构造 gold answer，并按 operator 级别诊断 filter / join / intersection / counting 等失败点。
- 创新点：最有启发的是它把评价目标从“是否抓到相关段落”转成“是否保留了算子执行所需的 typed value”。这对 Search Agent 比传统 RAG benchmark 更接近真实任务。
- 关键发现：不同范式在不同 operator 上排名会反转；相似度检索在 filter/project 上更强，但 IE-to-SQL 在 intersection/counting 更强。长上下文 oracle 也远未饱和，说明瓶颈不只在 answer model。
- 可转化实验：可以直接把 query-operator-preserving 指标迁到网页搜索、企业情报检索、代码依赖检索，测试 agent 在结构化约束问题上的证据保真能力。
- 局限：当前数据底座是企业事件新闻，外推到其他领域要看 schema 迁移成本。

### 6. Context-as-AI-Service: Surfacing Cross-File Dependency Chains for LLM-Generated Developer Documentation
- 链接：[arXiv](https://arxiv.org/abs/2606.04397) | [PDF](https://arxiv.org/pdf/2606.04397)
- 证据来源：PDF
- 核心方法：给代码 agent 叠加一层专门的 retrieval layer，让它能通过 keyword + semantic search 追踪 `跨文件依赖链`，而不是只靠局部 file reads 和 symbol navigation。
- 创新点：论文不是单纯说“多给上下文”，而是强调 documentation correctness 常依赖不显眼的 dependency chain，这和代码审查、bug 定位、repo QA 的证据寻源完全同构。
- 关键发现：在 API 注释审查和教程验证两个 case study 中，CAIS 不只复现 baseline 能找到的问题，还额外找到跨文件事实错误、缺少前置条件和一个可执行 bug，并降低了 wall-clock time 与 token 用量。
- 可转化实验：可直接迁到 code agent 的 issue resolution 流程，让 agent 在 patch 前后主动追索 tests、examples、internal utilities 和 upstream docs。
- 局限：证据目前来自两个生产 SDK case study，不是大规模 benchmark；但工程迁移价值很高。

### 7. Cascading Hallucination in Agentic RAG: The CHARM Framework for Detection and Mitigation
- 链接：[arXiv](https://arxiv.org/abs/2606.04435) | [PDF](https://arxiv.org/pdf/2606.04435)
- 证据来源：PDF
- 核心方法：把 agentic RAG 的失败从单点 hallucination 拆成跨阶段传播的 cascade，并用 stage-level fact verification、cross-stage consistency、confidence propagation monitoring、resolution triggering 四个模块在轨迹上持续监测。
- 创新点：它不把最终答案当唯一检测对象，而是把 `trajectory corruption` 当成主要风险源。这比输出级 hallucination detector 更贴近多步 Search Agent。
- 关键发现：论文报告 cascade detection rate `89.4%`、false positive `5.3%`、每 stage 额外延迟约 `215ms`，并显著优于只盯最终输出的检测器。
- 可转化实验：可以把同样的 cross-stage consistency 机制放进多跳检索、citation verification 或 browser agent 的工具轨迹里。
- 局限：实验依赖作者搭建的 LangChain + GPT-4o trajectory wrapper，外部复现和真实生产泛化还需要再看。

### 8. Cartridges at Scale: Training Modular KV Caches over Large Document Collections
- 链接：[arXiv](https://arxiv.org/abs/2606.04557) | [PDF](https://arxiv.org/pdf/2606.04557)
- 证据来源：PDF
- 核心方法：把长文档集合蒸馏为可复用 modular KV caches，并通过 joint multi-cartridge training、dynamic distractor mixing 和 budget manager 解决“独立训练 cartridge 混用就崩”的问题。
- 创新点：这篇对 Search Agent / long-context agent 的价值在于，它不再把 memory 只看成文本摘要，而是看成可索引、可加载、可组合的缓存工件。
- 关键发现：在多个长上下文数据集上，CAS 比单体 cartridge 高 `10-31` 点，并能在检索选择 cartridge 时以更少 prompt tokens 接近或超过传统 RAG。
- 可转化实验：适合迁到 repo memory、浏览轨迹 memory、长期个人知识库 memory，比较 `全文拼接`、`chunk RAG`、`retrieved cartridge` 三种范式。
- 局限：对信息密度极高的数据仍可能输给 text RAG；并且训练/服务复杂度高于标准向量检索。

## Code Agent

### 9. Synthetic Hallucinations, Real Gains: Hard Negatives from Frontier Models for FIM Hallucination Mitigation
- 链接：[OpenReview](https://openreview.net/forum?id=OoYmjSlQap) | [PDF](https://openreview.net/pdf?id=OoYmjSlQap)
- 证据来源：PDF
- 核心方法：让 frontier code models 专门生成“看起来合理但项目里并不存在”的 FIM hard negatives，再用这些 `chosen / rejected` 对监督训练小型开源 code model，针对 autocomplete 场景降低代码幻觉。
- 创新点：这篇最强的是数据构造 recipe。它不是依赖执行沙箱或昂贵偏好数据，而是把大模型当成错例生成器，为小模型蒸出极高价值负样本。
- 关键发现：Qwen2.5-Coder-7B-Instruct 在 Delulu 上 exact match 提升 `+18.8`，3B 版本也有 `+12.8` 的提升，而且作者做了按语言、错误类型、难度和训练目标的系统消融。
- 可转化实验：可以把同样思路扩展到 repo-level patching、tool-call repair、test-fix proposal，专门合成“局部合理但全局不存在”的 hard negatives。
- 局限：目标仍是 mid-keystroke FIM，不是完整 issue resolution；但对 Code Agent 后训练数据设计非常有启发。

### 10. TeleSWEBench: A Commit-Driven Benchmark for Evaluating LLM-Powered Software Engineering in Telecommunications
- 链接：[arXiv](https://arxiv.org/abs/2606.05001) | [PDF](https://arxiv.org/pdf/2606.05001)
- 证据来源：PDF
- 核心方法：从真实 srsRAN 5G commits 挖出 734 个结构化测试案例，区分 scope localization 与 functional correctness，并引入 TeleJudge 做文件级 patch 评审聚合。
- 创新点：它把通用 SWE-bench 难以覆盖的 `高状态性 + 强领域约束 + 多文件协议栈` 场景补上了，对“真实 code agent 到底在哪一步掉链子”很有帮助。
- 关键发现：最强工具也只有约 `25%` 的 shippable changes，说明 domain-specific repo reasoning 仍远没到可用阶段。
- 可转化实验：适合拿来对比 agent 在普通仓库与高专用仓库上的定位失败、上下文耗尽、只会输出解释不产出 patch 等模式。
- 局限：领域很窄，更多是在 benchmark 维度给 Code Agent 研究施压，而不是直接给出通用方法。

## 今日重点趋势

1. 本批最值得关注的主线是 `把 agent 失败恢复做成接口的一部分`。`Self-Reflective APIs` 处理 tool error recovery，`Token Budgets` 处理资源约束恢复，`PCAA` 处理高风险动作治理，`CHARM` 处理多阶段轨迹纠错。
2. Search Agent 的评价标准正在从“检索到相关文本”转向“是否保留了后续执行真正需要的结构化证据”。`QO-Bench`、`CAIS`、`CAS` 都在强化这个方向。
3. Code Agent 这批里最强信号不是更强模型，而是更好的训练数据与更苛刻 benchmark。`Synthetic Hallucinations` 和 `TeleSWEBench` 分别对应这两条线。

## 潜在研究 Idea

1. 为代码/浏览器工具统一设计 `self-reflective tool schema`，把错误诊断、可修复参数、最小重试 patch 一起暴露给 agent。
2. 用 `Token Budgets + PCAA` 的组合做一个“资源治理 + 审批治理”联合 runtime，测试多 agent delegation 中的越权、超支和审计回放。
3. 基于 `QO-Bench` 思路构建 `repo-operator benchmark`，把代码仓库里的 symbol join、cross-file filter、counting、temporal diff 查询显式结构化。
4. 借鉴 `Synthetic Hallucinations` 生成 repo-level hard negatives，把错误 import、错误 API、错误 patch、错误 citation 都系统化为拒绝样本。
