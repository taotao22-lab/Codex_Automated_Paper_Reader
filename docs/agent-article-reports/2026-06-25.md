## General Agent

### 1. Beyond Function Calling: Benchmarking Tool-Using Agents under Tool-Environment Unreliability

- 来源：[arXiv](https://arxiv.org/abs/2606.25819) / [PDF](https://arxiv.org/pdf/2606.25819)
- 问题背景：很多 tool-use benchmark 默认工具说明准确、调用稳定、输出可信。但真实 Agent 系统里的工具会改接口、报错、返回格式漂移，甚至多个来源互相冲突。只测 function call 是否填对参数，会高估 Agent 的真实可用性。
- 核心方法：论文提出 ToolBench-X，把 tool-use 看成一个带不确定转移的序列决策过程。它从干净工具环境出发，注入 5 类可恢复 hazard：Specification Drift、Invocation Error、Execution Failure、Output Drift、Cross-source Conflict。关键点是每个被注入的任务仍至少有一条可行恢复路径，例如 retry、fallback、verification 或 cross-checking，因此模型失败不能归咎于任务无解。
- 创新点：它把评测焦点从“函数调用正确”推进到“工具环境不可靠时仍完成任务”。这对 Agent post-training 很重要，因为训练信号可以从单步 API accuracy 变成 hazard diagnosis、recovery action、evidence verification 和 final answer consistency。
- 实验与 benchmark：ToolBench-X 包含 1106 个任务、4956 个工具，覆盖 sequential、parallel 和 mixed workflows。论文与 BFCL v3、T-EVAL、UltraTool、AgentNoiseBench 对比，强调后者多数不同时覆盖五类可靠性 hazard 和混合工作流。实验显示所有被测模型在 hazard 环境下成功率低于 60%；失败更多来自 hazard 诊断和恢复无效，而不是调用次数或推理预算不足。targeted recovery hints 能恢复不少失败任务，test-time scaling 的收益相对有限。
- 可迁移价值：可以直接用作工具使用 RL 的任务模板。对于浏览器、代码执行、检索工具，可把真实失败日志归为五类 hazard，再构造“可恢复但不干净”的训练 episode，让模型学习何时重试、何时换工具、何时交叉验证。
- 局限：PDF 片段中没有完整列出所有模型的逐项数字，日报只引用已核实的总体结论；benchmark 的工具是否足够接近真实生产系统，还需要看代码和任务分布。
- 阅读建议：优先读 benchmark construction、hazard taxonomy 和 recovery analysis。它很适合作为 tool-use robustness 的核心 related work。

### 2. Memory Makes the Difference: Evaluating How Different Memory Roles Shape Conversational Agents

- 来源：[arXiv](https://arxiv.org/abs/2606.25361) / [PDF](https://arxiv.org/pdf/2606.25361)
- 问题背景：很多 memory/RAG 工作关注“怎么存、怎么检索”，但较少区分被检索记忆在回答中的功能角色。对长期 Agent 来说，同样是 retrieved memory，有的补充约束，有的澄清意图，有的看似相关但会误导模型。
- 核心方法：论文提出细粒度 conversational memory taxonomy，把记忆按功能区分为 answer、clarifying、enriching、distracting、irrelevant 等类型，并设计 user-centric evaluation proxy。它不只用 reference answer 做 F1/ROUGE，而是模拟用户视角评估相关性、信息量、约束意识和个性化。
- 创新点：这篇论文把 memory 从“召回片段集合”变成“对 Agent 决策有不同角色的证据”。这种视角比简单扩大检索 k 更适合长期 Agent，因为它可以解释为什么更多上下文有时反而降低 grounding。
- 实验与 benchmark：实验使用两个长期、多会话对话数据集和三个 frontier LLM。论文讨论 LoCoMo、LongMemEval、MT-Bench-Plus、InfiniteBench 等长期对话/记忆评测背景。结果显示 clarifying memory 会提升事实准确性和 constraint awareness，irrelevant memory 会降低 topic relevance 和 constraint awareness；随着 retrieval size 增加，模型并不一定更愿意 grounding 到检索内容。
- 可迁移价值：适合用于 Agent memory 数据标注。可以把轨迹中的 observation、user preference、tool result、failure note 标成不同 memory roles，再训练检索路由器判断哪些记忆应该进入上下文，哪些应压缩或隔离。
- 局限：主要是 conversational agent，不是工具执行 Agent；结论需要迁移到代码、搜索、GUI 场景验证。
- 阅读建议：重点看 taxonomy 和 evaluation proxy。它能启发“记忆片段角色标注 + 检索路由”实验。

## Search Agent

### 3. Is GraphRAG Needed? From Basic RAG to Graph-/Agentic Solutions with Context Optimization

- 来源：[arXiv](https://arxiv.org/abs/2606.25656) / [PDF](https://arxiv.org/pdf/2606.25656)
- 问题背景：GraphRAG、Modular RAG、Agentic RAG 都声称能提升复杂知识任务，但实际什么时候该用图、什么时候该用 agentic multi-step planning 并不清楚。很多系统只扩大检索范围，却没有证明 generation quality 同步提升。
- 核心方法：论文构造 9 个标准化 RAG 场景，从 basic document RAG 到 text-graph hybrid retrieval、predefined/computed KG、agentic multi-step planning 和 agent-graph integration。它还提出 context optimization：relation-grouped graph representation、graph retrieval deduplication、content-aware document deduplication，以及 hybrid ReAct-ReWOO 风格的 batch agentic retrieval。
- 创新点：最有价值的是把 GraphRAG/Agentic RAG 的收益拆成“检索结构”“上下文成本”“生成质量”三件事。论文指出存在 retrieval-generation gap：更多或更复杂的检索不一定按比例提高最终答案质量，传统 retrieval-oriented metrics 可能高估高级 RAG 的收益。
- 实验与 benchmark：使用 STaRK-Prime，一个 precision medicine 半结构化知识库，约 129K 个 disease/drug/protein/gene 等实体和 8.1M 条 PrimeKG 关系。评测使用官方 human-generated queries，指标包括 Hit@1、Hit@5、Recall@20 和 MRR。context optimization 在 GraphRAG/Agentic RAG 中带来 19%-53% token usage reduction。
- 可迁移价值：对 Search Agent 很直接。可以把网页、代码库、论文 PDF 的 text 和 graph evidence 分开管理，让 agent 一次规划多个互补 query，再合并去重的 evidence memory，而不是每轮 ReAct 只打一条 query。
- 局限：实验域是 precision medicine，迁移到开放网页搜索时 KG 质量和实体链接会更难；PDF 片段没有核实完整表格数字，因此日报只引用摘要和方法页可验证结果。
- 阅读建议：优先读 9 scenarios、context optimization 和 retrieval-generation gap 分析。它适合作为 Agentic RAG 系统设计参考。

### 4. Tracing Target Answers in Poisoned Retrieval Corpora via Token Influence Attribution

- 来源：[arXiv](https://arxiv.org/abs/2606.25721) / [PDF](https://arxiv.org/pdf/2606.25721)
- 问题背景：RAG 系统容易被 corpus poisoning 攻击操纵答案。常见防御依赖额外分类器或 LLM verifier，成本高，而且不一定能解释攻击目标答案如何进入生成。
- 核心方法：TRACE 用 token influence attribution 在 retrieved documents 中寻找对目标答案有高影响的 recurrent keywords，再做二次验证确认这些 token 对模型预测的影响。直观地说，它不是再请一个模型判断“文档可不可信”，而是追踪答案相关 token 在检索上下文中的来源和影响路径。
- 创新点：它把 RAG 安全检测放在生成前的 retrieval-stage，不需要外部监督或辅助模型，适合做低成本在线防御。对 Search Agent 来说，这类影响归因也可用于解释“为什么 agent 相信这个来源”。
- 实验与 benchmark：论文报告在 3 个 QA benchmark 和 6 个 LLM 上测试，展示了较强检测性能，并能同时揭示 attacker-specified target answers。PDF 片段未完整显示具体数据集名称和数值表格，因此此处不编造未核实数字。
- 可迁移价值：可以作为 Search Agent 的 evidence validation 模块。训练搜索轨迹时，不只记录 source URL，还记录哪些 token/句子真正影响最终答案，从而做引用可信度和 poisoning 检测。
- 局限：当前描述集中在 QA/RAG poisoning；对复杂多跳网页搜索、代码检索和动态网页内容的效果需要进一步验证。
- 阅读建议：读 method 的两阶段 detection workflow 和 experiments 表格。若要做可靠搜索 Agent，这篇适合作为引用归因和攻击检测的候补模块。

## Code Agent

### 5. Unlocking Model Potentials Through Adaptive Multi-Agent Scaffolding for Efficient Issue Resolution

- 来源：[arXiv](https://arxiv.org/abs/2606.25514) / [PDF](https://arxiv.org/pdf/2606.25514)
- 问题背景：GitHub issue resolution 是长程任务：Agent 要读 issue、定位代码、复现失败、改 patch、跑测试和验证。单 Agent 把全部状态塞进共享上下文，容易 context degradation、context poisoning、test overfitting 和 patch overfitting。
- 核心方法：icat-agent 是一个去中心化多 Agent scaffold，由 Explorer、Patch Editor、Validator 组成。它先用 rubric 判断 issue quality：高质量 issue 直接并行 patching 和 validation，低质量 issue 先让 Explorer 补足仓库结构、依赖和相关代码上下文。不同 Agent 不共享全局上下文，而是通过同步 event-based message passing 交换必要信息。
- 创新点：它不是换一个更强模型，而是证明 scaffold 能释放同一 backbone 的潜力。把 patch 生成和验证隔离，可以减少自我验证导致的过拟合；按 issue 质量自适应选择路径，可以在模糊问题上先探索，在明确问题上节省成本。
- 实验与 benchmark：评估覆盖 SWE-bench Verified 500 个实例和 SWE-bench Pro 731 个实例，对比 SWE-agent、mini-SWE-agent、Claude Code，并使用 MiniMax M2.5、GPT-5.4-xhigh、GPT-5-mini、Claude Sonnet 4.5 等 backbone。论文报告同模型下 SWE-bench Verified 提升 3.6-8.4%，SWE-bench Pro 提升 6.3-18.5%；icat-agent+GPT-5.4-xhigh 在 SWE-bench Pro 解决率 67.4%，比 mini-SWE-agent+GPT-5.4-xhigh 的 59.10% 高 8.3 个百分点；相比 Claude Code multi-agent baseline 平均每实例成本低约 $1.18。
- 可迁移价值：非常适合 Code Agent 后训练。可以把 Explorer/Patch Editor/Validator 的事件流作为多角色轨迹，把“定位-修改-验证”拆成可监督步骤，并把 validation agent 的反馈转成 reward 或偏好数据。
- 局限：结果很强，但依赖具体模型和 scaffold 实现；需要看开源代码确认事件协议、上下文截断和验证策略是否可复现。
- 阅读建议：精读 issue quality rubric、event message passing 和 ablation。它是今天最值得加入 Code Agent related work 的论文之一。

### 6. IntentTester: Intent-Driven Multi-agent Framework for Cross-Library Test Migration

- 来源：[arXiv](https://arxiv.org/abs/2606.25588) / [PDF](https://arxiv.org/pdf/2606.25588)
- 问题背景：开源项目里有大量高质量单元测试，但测试知识被锁在具体库和 API 里。传统迁移依赖 API 签名和结构映射，遇到跨语言、不同设计风格或缺少一一对应 API 时容易生成不可执行测试。
- 核心方法：IntentTester 不直接翻译测试代码，而是先抽象成语言无关的 Test Description Language，再用 repository graph 对齐语义相关实体和依赖，最后由 LLM-guided multi-agent reasoning 合成目标库可执行测试，并通过迭代验证修复。
- 创新点：它把测试复用的单位从“代码结构”换成“测试意图”。这对 Code Agent 很有启发：Agent 可以先把已有测试提炼成行为规范，再跨仓库生成新的 oracle，而不是机械模仿 API 调用。
- 实验与 benchmark：评估覆盖 JSON、HTML、Time 三个领域，9 个开源项目，Java/Python 两种语言。IntentTester 生成 2776 个语法正确测试，正确率 85%，两个 baseline 为 51% 和 43%；其中 2410 个测试成功执行，effectiveness rate 74%。论文还报告发现 25 个真实缺陷，包括 stack overflow、null dereference、parsing inconsistency，部分已被维护者确认或修复。
- 可迁移价值：可用于构造 Code Agent 的测试生成训练数据。特别适合“从相关项目借测试意图 -> 生成目标仓库测试 -> 执行反馈 -> 修正”的闭环。
- 局限：当前覆盖的库类型相对有限；TDL 的表达能力和 repository graph 构建质量会决定迁移上限。
- 阅读建议：重点看 TDL、repository graph alignment 和 validation loop。它能补足 SWE-bench 类任务中测试 oracle 稀缺的问题。

### 7. Evaluating LLMs on Real-World Software Performance Optimization

- 来源：[arXiv](https://arxiv.org/abs/2606.25530) / [PDF](https://arxiv.org/pdf/2606.25530)
- 问题背景：现有代码 benchmark 多关注功能正确性，或只测单函数性能。真实性能优化要在仓库级代码中权衡 runtime、peak memory、输入规模、噪声测量和执行条件，难度远高于“让测试通过”。
- 核心方法：论文提出 SWE-Pro，从开源项目中抽取 102 个专家写的优化任务，并为每个任务配 parameterized tests。评测同时看 runtime、peak memory 和 Time-Weighted Memory Usage，在不同输入和噪声条件下比较模型 patch 与专家 patch。
- 创新点：它把 Code Agent 的目标从 functional correctness 推向 engineering quality。尤其重要的是 memory footprint 和 TWMU，因为很多模型能写出通过测试的代码，却不能做真正的系统级优化。
- 实验与 benchmark：SWE-Pro 的专家实现总体 speedup 15.5x、peak memory reduction 171.3x；专家在 91.2% 任务上改善 runtime，在 65.7% 任务上改善 peak memory。当前 LLM 的 runtime gain 很小，memory optimization 几乎不存在，暴露出与专家级工程能力的明显差距。
- 可迁移价值：可作为 Code Agent RL 的高级奖励来源。除了 pass/fail，还可以把 wall time、memory、TWMU 和噪声鲁棒性纳入 reward，训练模型避免“能跑但很慢/很耗内存”的 patch。
- 局限：性能测量天然受硬件和环境影响，复现需要严格执行环境；102 个任务规模不算大，但质量高。
- 阅读建议：适合和 SWE-bench 一起读，用来扩展 Code Agent 评测维度。

### 8. LibEvoBench: Probing Temporal Knowledge Stratification in Code Generation Models

- 来源：[arXiv](https://arxiv.org/abs/2606.25402) / [PDF](https://arxiv.org/pdf/2606.25402)
- 问题背景：真实代码库常固定在旧版本库上，而 LLM 训练语料混合了多个时间点的 API。模型很容易写出“在另一个版本有效”的代码，这类 anachronistic error 很难被普通代码生成 benchmark 捕捉。
- 核心方法：LibEvoBench 跨多个 Python 库版本构造三类任务：API Calling、API Identification、Signature Recall，并提出 Software Evolution Understanding Score。它用 API lifecycle analysis 跟踪每个 public API 的存在和签名变化。
- 创新点：论文把代码生成的知识问题从“知道 API 吗”细化为“知道目标版本的 API 吗”。这对仓库级 Code Agent 极关键，因为修复旧项目时，最新文档往往会误导模型。
- 实验与 benchmark：目标库包括 PyTorch、NumPy、SciPy，覆盖 29 个库版本、超过 16K 个 API、超过 125K 个评测实例。论文评估 proprietary frontier models 和 Qwen-3.5 系列。结果显示模型对 evolving APIs 有稳定性能下降，最强模型在 stable 与 evolving API 上仍有 7-10 个百分点 gap；明确给出版本约束几乎无帮助，而提供相关文档可带来 10-20 个百分点 API-calling accuracy 提升。
- 可迁移价值：可以直接转成 repo-level Agent 的 retrieval policy：每次改代码前先识别依赖版本，再检索匹配版本文档，而不是默认使用最新 API。
- 局限：主要覆盖 Python 科学生态；其他语言和框架仍需类似数据构造。
- 阅读建议：重点读 task definition、SEUS 和 documentation ablation。它很适合作为“代码 Agent 必须版本感知”的证据。

### 9. CodeChat-Eval: Evaluating Large Language Models in Multi-Turn Code Refinement Dialogues

- 来源：[arXiv](https://arxiv.org/abs/2606.25747) / [PDF](https://arxiv.org/pdf/2606.25747)
- 问题背景：开发者与 coding assistant 的真实交互通常是多轮 refinement：改风格、重构、添加逻辑、换实现策略，同时保持原功能正确。现有 HumanEval、MBPP、SWE-bench 更偏单轮或单任务，不测 follow-up 修改后的功能回归。
- 核心方法：CodeChat-Eval 从多轮代码修订对话构造 evaluation sessions，使用 dynamic instruction selection algorithm 选择适用于当前代码版本的后续指令，并用原任务测试套件检查 refined code 是否保持 functional correctness。
- 创新点：它明确测 Pass-to-Fail regression：模型先写对了，但在后续“看似无害”的 refinement 中改坏。这个评测比单轮 pass@k 更贴近日常代码 Agent 的持续编辑风险。
- 实验与 benchmark：论文构造 10-turn evaluation sessions，评估 Llama、Qwen、DeepSeek、GPT 四个家族共 8 个开源和闭源模型。结果显示 functional correctness 在多轮 refinement 后显著下降，下降幅度从 GPT-5 Nano 的 19.2% 到 Llama 3.1 8B 的 69.2%；Semantic 指令导致约 21% 下降，Add 指令约 17% 下降，是最容易触发回归的类型。
- 可迁移价值：适合用于训练“编辑后必须运行测试/保持行为”的 Code Agent。可以把每轮修订后的测试结果作为 dense feedback，训练模型识别高风险 refinement。
- 局限：仍以测试套件作为正确性代理；如果测试不完备，模型可能保持测试通过但引入隐藏行为变化。
- 阅读建议：适合作为多轮 coding assistant 评测基准，特别是与 issue resolution agent 的长轨迹训练结合。

### 10. Representation Matters: An Empirical Study of Program Representations for LLM Vulnerability Reasoning

- 来源：[arXiv](https://arxiv.org/abs/2606.25356) / [PDF](https://arxiv.org/pdf/2606.25356)
- 问题背景：很多 LLM 漏洞检测方法直接把源代码塞进 prompt，默认更多源码上下文会更好。但安全推理往往依赖控制流、数据依赖和调用关系，原始源码可能稀释关键证据。
- 核心方法：RepBench 把真实 C/C++ 漏洞样例转换为多种表示：raw source、AST、CFG、PDG、组合表示，以及由 VulChecker/Hector 生成的 enriched PDG。任务固定为给定 target function 和 CWE，判断 VULNERABLE 或 SAFE。
- 创新点：论文系统证明“上下文越多越好”并不成立。结构化静态分析表示能把漏洞相关证据压缩出来，反而比长源码更利于 LLM 推理。
- 实验与 benchmark：主评测来自 PrimeVul-derived corpus，包含 107 个 Joern-based C/C++ testcases，覆盖 CWE-122、CWE-190、CWE-191、CWE-415、CWE-416；辅助轨道包含 19 个 ePDG cases。最强 AST+PDG 达到 83.2% accuracy，raw source 为 53.5%，提升 29.7 个百分点。graph-only prompts 整体优于 source-only 和 source-plus-graph，同时 prompt overhead 更低。
- 可迁移价值：对 Code Agent 检索和上下文压缩很实用。修漏洞或做代码审查时，不应只拼接文件内容，而应把 AST/PDG/CFG 作为可检索证据层，让模型围绕缺陷相关路径推理。
- 局限：样本规模较小，且集中在 C/C++ 和五类 CWE；对 Python/JavaScript 业务漏洞和跨文件数据流还要扩展。
- 阅读建议：作为“程序表示选择影响 LLM 推理”的实证证据阅读，可与仓库级上下文压缩工作结合。

## 候补与 Related Work

- [The Hitchhiker's Guide to Agentic AI](https://arxiv.org/abs/2606.24937)：603 页实践型综述，覆盖 LLM substrate、RLHF/PPO/DPO/GRPO、trajectory-based RL、Agentic RAG、memory、MCP、A2A、多 Agent 拓扑、评测和部署。不是新算法论文，但适合作为系统性参考书。
- [How Do Developers Maintain and Evolve Their Agents' Instructions?](https://arxiv.org/abs/2606.25257)：研究 Agent Context Files 如何演化，计划用仓库挖掘、ACF change taxonomy 和代码质量指标分析 coding agent 指令维护。偏 study plan，方法结果尚未充分落地，但对 agent instruction governance 很相关。
- [TheoremGraph](https://arxiv.org/abs/2606.25363)：构建 informal/formal mathematics statement-level dependency graph，包含 11.7M theorem-like environments、18.3M candidate dependencies、LeanGraph 388,105 declaration nodes 和 11.3M typed edges，并提供 HTTP API 与 MCP interface。数学域较窄，但对可追踪检索、引用和形式化 RAG 有启发。
- [Adaptive Re-Ranking](https://arxiv.org/abs/2606.25249)：用 utility-based labeling 做 query-level reranking routing，在 BM25、MiniLM-L6-v2、BGE-v2-m3 间选择，报告相对 BGE 有 1.15-53x lower median latency、1.11-5.22x lower mean latency，nDCG@10 变化 -17.5% 到 +4.0%。可用于 Search Agent 的成本感知检索策略。
- [The Web4 Agent Economy](https://arxiv.org/abs/2606.25876)：大规模分析 Web4 agent 生态，包括 99,448 个 multi-chain identity registrations、317,596,323 transaction logs、341 个 MCP projects、349 个 GitHub issues。偏生态研究，但对 MCP、agent identity、payment interoperability 和跨环境授权有工程参考。

## 今日重点趋势

- Tool-use 评测从 API 调用正确率转向不可靠环境下的恢复能力。ToolBench-X 的五类 hazard 很适合转成后训练数据标签。
- Code Agent scaffold 的价值正在超过单纯换模型。icat-agent 说明上下文隔离、独立验证和 issue quality routing 能显著改变同一 backbone 的解题率和成本。
- Code Agent benchmark 正在补齐真实工程维度：性能优化、版本感知、多轮 refinement、静态分析表示，都是单轮代码生成测不到的能力。
- Search Agent/RAG 的下一步不是盲目扩大上下文，而是做结构化检索、去重、token budget 管理和证据可信度归因。

## 潜在研究 idea

- 用 ToolBench-X 式 hazard taxonomy 重放真实 Agent 失败日志：把浏览器、代码执行、搜索 API 的失败归类为 spec drift、execution failure、output drift 等，构造 recoverable trajectory 数据。
- 为 Code Agent 建一个“版本感知修复”训练集：issue + dependency lockfile + matched-version docs + failing tests，让模型学习先识别版本再改代码。
- 把 IntentTester 的 intent abstraction 迁移到 SWE-bench：从已有项目测试抽象出行为意图，给 issue resolution agent 自动生成更强验证器，降低 patch overfitting。
- 给 Agent memory 做 role-aware retrieval：把轨迹记忆标成 answer/clarifying/enriching/distracting/irrelevant，让检索器不只学相似度，还学“这条记忆对当前决策扮演什么角色”。
- 在 Agentic RAG 中加入 TRACE 式 influence attribution：最终回答必须能回溯到影响 token 和来源片段，用于防 poisoning、引用可信度和 reward shaping。
