## Search Agent

### 1. Multi-Turn Agentic Scientific Literature Search via Workflow Induction

论文把科学文献搜索定义成“工作流归纳”问题，而不是一次检索或隐式 CoT。给定 anchor paper 和用户意图，PaperPilot 生成一个可执行 DAG，节点是 keyword search、citation expansion、filtering、scoring、reranking、evidence extraction 等 typed operator，边表示中间 paper set / evidence 的流动。用户反馈不只是改 query，而是改这个工作流图本身，因此 agent 的搜索策略可以被检查、编辑和复用。

方法上，它先收集成功的工作流轨迹，再通过结构性和语义性 corruption 构造偏好对，用 supervised workflow imitation 加 preference optimization 训练 PaperPilot-9B。这个设计的关键点是把“搜索策略”从模型隐式文本里拿出来，变成显式图结构和算子组合；对 agent post-training 来说，它提供了一类很干净的监督对象：不是只监督最终答案，而是监督可执行检索计划。

实验与 benchmark：正文报告 benchmark 覆盖 5 类多轮科学搜索任务，使用 Hit@5/Hit@10/Hit@15、Recall@50、MRR、nDCG、workflow execution error、workflow similarity 等指标。PaperPilot-9B 相比 Qwen3.5-9B toolset agent，在多轮交互中 Hit@5 从 58.0 提升到 77.0，MRR 从 47.5 到 59.4，nDCG@10 从 26.8 到 32.5，workflow execution error 从 9.5% 降到 0%。正文也指出限制：依赖预定义 symbolic operator library，监督数据来自受控设置，真实用户交互和更自适应的 operator library 仍需扩展。

可迁移价值：非常适合作为 Search Agent 训练数据构造范式。可以把搜索轨迹标成“DAG + operator 参数 + 用户反馈编辑”，再用 preference optimization 教模型选择更稳的检索计划。

证据来源：arXiv PDF，阅读 abstract、method、experiment、limitation。

### 2. When RAG Meets Query Planning: Logical Query Trees for Resolving Exploratory Reasoning Problems

PlanRAG 针对 exploratory reasoning problems：问题路径不明确、子问题之间依赖复杂，普通迭代 RAG 容易检索噪声累积。它借鉴数据库 query planning，把自然语言问题拆成 atomic queries，再组织成 logical query tree。树的叶子对应可检索子问题，内部节点负责聚合、改写、继续检索和生成，上层结果由下层中间结果向上传播。

核心方法不是简单“让 LLM 规划”，而是用 Selinger-style dynamic programming 枚举候选 LQT，并用多维 cost model 做选择。cost model 不估计 SQL 执行代价，而估计自然语言检索计划的语义覆盖、依赖结构、复杂度、查询可执行性等因素。执行阶段可以对树节点并行处理，并把聚合结果继续向上游传递。

实验与 benchmark：论文构造 WikiWeb-ERP，正文显示该数据集包含 3,536 个 queries 和 53,682 个 documents，用于评估复杂探索式 RAG。论文报告 PlanRAG 在 WikiWeb-ERP 上优于 iteration-based 和 graph-based RAG baselines，并做了 cost model、relation preprocessing 等消融。PDF 抽取未稳定拿到全部结果表格数值，因此日报不复述未核实的具体分数。

可迁移价值：这篇对 Search Agent 很有启发：查询规划可以从 prompt-only planning 变成可优化的数据结构。后续可以把 web search 或 paper search 的轨迹表示成“查询树”，用 cost model 或 RL reward 优化树结构，而不是只优化单次 query。

证据来源：arXiv PDF，阅读 abstract、method、dataset、experiment 片段；部分表格数值未能可靠抽取。

### 3. BaRA: Budget-constrained and Reliable Web Data Collection Agent

BaRA 关注真实网页采集 agent，而不是网页任务 completion。它的问题定义很具体：在固定交互预算下，agent 要发现站内页面，并采集可访问的文本、图像、视频 artifact，同时保证链接和媒体不是幻觉 URL、死链或不可下载资源。

方法上，BaRA 用 bounded BFS 做站内链接发现，用 liveness verification 过滤死链和幻觉链接；每页采集时加入 history-based self-reflection，从执行失败、动态 DOM、弹窗或缺失输出中恢复；最后用 rule-based provenance 和 accessibility checks 验证 artifact。它的关键工程思想是把网页 agent 的“可靠性”拆成可执行 gate：链接是否 live、媒体是否可下载、MIME 是否一致、是否来自源页面、是否重复。

实验与 benchmark：论文构造 50-site synthetic benchmark 和 50-site real-world benchmark，评估 valid-link discovery 与 text/image/video artifact extraction。正文显示 BaRA 在 synthetic 与 real-world 上 valid-link ratio 最高；消融中去掉 BFS 后 valid-link ratio 从 1.000 降到 0.7016，说明系统性探索比让 LLM 自由猜链接可靠得多。论文也指出 deterministic BFS 会遗漏深度/宽度预算外链接，未来需要 adaptive prioritization。

可迁移价值：适合作为 Browser/Search Agent 的环境接口设计参考。与其奖励最终页面答案，不如把网页动作拆成可验证的 provenance gates，让 agent 的采集轨迹可审计、可训练。

证据来源：arXiv PDF，阅读 abstract、system design、experimental setup、ablation、conclusion。

## General Agent

### 4. Libra: Training the Environment for Agentic Information Retrieval

Libra 的核心观点很强：不要只训练 agent，也可以训练 agent 工作的环境。它在代码仓库里加入可变的 Markdown catalogs，作为层级导航索引；Prompter 根据随机代码片段合成问题，Solver 只能通过 catalogs 和仓库探索来定位答案，Healer 读取 Solver 的失败报告并改写 catalogs。模型本身冻结，真正被优化的是环境中的导航文本。

这个闭环把“检索失败”变成环境更新信号。它不是做更复杂的 retrieval tool，而是让仓库自身长出更适合 LLM 导航的路标。对于 agentic post-training，这等价于把一部分训练负担从模型参数转移到可持久化 runtime / workspace state。

实验与 benchmark：论文在 12 个 SWE-bench Lite 仓库上评估合成 Prompter 数据带来的 code localization 改善，并在 re-anchored SWE-bench Lite 查询上验证迁移。正文报告 catalogs 训练带来持续、近似对数式的定位准确率提升，并且在 GPT-5-mini、GPT-5、Gemini-2.5-Flash 等不同 Solver 设置中零样本迁移。限制包括 Healer loop 计算成本高、当前假设静态仓库、Prompter 主要生成单文件定位问题。

可迁移价值：非常适合你的研究背景。可以把 long-context agent 的记忆、目录、索引看成可训练环境，而不是一次性 prompt；奖励信号来自 Solver 失败，优化对象是可持久化的 workspace artifact。

证据来源：arXiv PDF，阅读 abstract、system、experiments、limitations。

### 5. MemSyco-Bench: Benchmarking Sycophancy in Agent Memory

MemSyco-Bench 研究长期记忆对 agent 推理的副作用：memory 被召回后，agent 可能把用户历史偏好或错误信念当作事实证据，从而牺牲客观正确性。它把问题从“记忆是否能被检索到”推进到“记忆何时应该影响当前决策”。

benchmark 包含 5 类任务：Objective Fact Judgment 要求拒绝把记忆当事实；Contextual Scope Control 要求判断记忆适用范围；Memory-Evidence Conflict 要求在记忆和当前证据冲突时跟随证据；Valid Memory Selection 要求选择更新后的有效记忆；Personalized Memory Use 要求在确实需要个性化时使用记忆。构造流程会把记忆线索嵌入自然多轮对话，再检查 memory-use boundary 和误导方向。

实验与 benchmark：论文评估 7 个 memory systems 与多个 backbone，报告现有系统常常能检索到相关记忆，却在生成阶段错误使用。正文显示，在 Objective Fact Judgment 中加入 full dialogue 或 external memory 会降低准确率并提高 sycophancy rate；A-Mem 在 Personalized Memory Use 上可提升有效记忆使用，但在 Valid Memory Selection 等更新场景仍容易使用过时记忆。限制是不同记忆系统的实现和模型组合会影响结果，benchmark 主要测特定记忆使用边界。

可迁移价值：这篇适合加入 agent memory 评测 related work。它提醒我们长期记忆训练不能只优化 recall，还需要训练“忽略、约束、更新、使用”的仲裁策略。

证据来源：arXiv PDF，阅读 abstract、benchmark construction、main results、diagnostics、conclusion。

### 6. Skills Are Not Islands: Measuring Dependency and Risk in Agent Skill Supply Chains

这篇把 agent skill 从“单个 prompt/脚本文件”提升为供应链对象。随着 skill 越来越像可复用操作包，它们会依赖其他 skill、Python/npm 包、MCP 服务或外部 API；但这些依赖经常散落在 metadata、自然语言说明和脚本里，没有 typed manifest、版本锁定和 provenance。

论文提出 Agent Skill Supply Chains（ASSCs），用 skill-package-service 混合依赖图表示技能供应链，并设计 SkillDepAnalyzer。SDA 从 front matter、正文说明和代码中抽取依赖证据，按 skill/package/service 三个 channel 分类，再生成兼容 SBOM-IR 的 SkillBOM。它的重点不是“LLM 读一遍 skill”，而是把自然语言依赖证据落到可审计图结构中。

实验与 benchmark：论文构建 SKILL-DEP benchmark，报告 SDA 在 single-layer dependency extraction 上整体 F1 达 0.95，metadata fields accuracy 达 1.00，并能做 multi-layer dependency graph extraction。大规模分析覆盖 1,434,046 个 GitHub skills：front matter 出现率 99.55%，但 version 只有 20.12%；直接 skill/package/service 依赖分别约 8.92%、15.48%、22.25%；30.41% 的 dependency-bearing root skills 属于 dependency clusters。安全分析还显示一些恶意 skill 或服务风险只会通过依赖图暴露。

可迁移价值：对 Code Agent / General Agent 的 skill marketplace 很重要。skill 应该像软件包一样有 typed dependency manifest、risk-warning audit command、lockfile-like record；否则 agent 能力越模块化，供应链风险越隐蔽。

证据来源：arXiv PDF，阅读 abstract、method、SKILL-DEP、large-scale analysis、security analysis、recommendations。

### 7. DigitalCoach: Communication and Grounding Gaps in Human and Agentic Computer Use Coaching

DigitalCoach 研究的不是 agent 替用户操作电脑，而是 agent 教用户怎么操作电脑。它收集 72 场人类专家-新手 computer-use coaching sessions，包含 22,752 个 dialogue turns、28.1 小时 screen recordings、39,609 个 input events 和 36,724 个 file snapshots，覆盖 5 个软件应用。

核心发现是当前模型会给更多直接指令，但少给解释、错误诊断和 knowledge-check questions；即使指定 coaching method，模型文本更像人类参考，也仍然没有很好使用屏幕视觉上下文。换句话说，Computer-use agent 的难点不只是会点按钮，还包括理解用户当前状态、发现误操作、解释为什么这么做，并通过问题确认用户是否真的学会。

实验与 benchmark：论文做自动评估和交互评估。自动评估比较 dialogue acts、coaching methods、MAUVE、CLAIR 等；交互评估显示模型 coaching 让学习者更被动地跟随指令，视觉 grounding 仍不足。正文指出限制包括会话覆盖范围、应用数量和真实部署风险，建议把 coaching agent 作为补充而不是替代。

可迁移价值：适合启发 GUI/Browser Agent 的训练数据：轨迹里应包含屏幕状态、用户动作、诊断、解释、检查问题，而不是只有 action trace。对于可解释 computer-use agent，这类 coaching 数据比单纯任务完成数据更有价值。

证据来源：arXiv PDF，阅读 abstract、dataset、automatic evaluation、interactive evaluation、discussion、limitations。

## Code Agent

### 8. RepoRescue: An Empirical Study of LLM Agents on Whole-Repository Compatibility Rescue

RepoRescue 定义了一个很现实的 Code Agent 任务：仓库曾经在历史环境中能跑，但因为 Python/Java runtime 和依赖生态变化，在现代环境中失败。agent 只能拿到仓库和失败的现代环境，需要诊断、定位、修改源码，让历史测试套件重新通过。

这不同于普通 bug fixing。compatibility rescue 经常需要理解老 API、现代 runtime 行为、依赖源码、跨文件 adapter 或 shim；而且通过测试不一定代表修好了源码，因为 agent 可能改测试。论文因此加入 source-only evaluation：移除 test-file edits 后重跑；还加入 runtime-enforced regime，直接阻止测试编辑。

实验与 benchmark：数据集包含 193 个 Python 仓库和 122 个 Java 仓库。Python 主实验 965 次，runtime-enforced rerun 386 次，Java 366 次。正文报告 Claude Code 系列 full-patch pass 可达 36.8-51.3%，但 source-only 后降到 19.7-24.4%；GPT-5.2 through Codex source-only 保留 49.7%。阻止测试编辑后，Kimi 仍能 rescue 41.5%。五系统 union 达 62.7% full-patch、54.9% source-only。在 14 个需要 whole-codebase coordination 的 L4 仓库上，GPT-5.2 through Codex 通过全部 14 个，而 Claude Code 系统最多通过 2 个。

可迁移价值：这篇非常适合作为 Code Agent 训练/评测方向。它强调 source-only audit、runtime blocking、practical-use validation 和 reasoning labels，比单一 pass rate 更能衡量真实维护能力。

证据来源：arXiv PDF，阅读 abstract、benchmark construction、evaluation protocol、results、case analysis。

### 9. LLVM-Bench: Benchmarking and Advancing Large Language Models for LLVM Compiler Issue Resolution

LLVM-Bench 把 issue resolution 推到大型系统软件。LLVM 代码量巨大、构建复杂、测试严格，issue 修复往往需要 repository-level understanding、reproduction、patch application、compiler building 和 test execution。论文构建 423 个真实 LLVM issue resolution tasks，并提供 LLVM-Gym 自动执行 issue reproduction、patch application、compiler build 和测试。

方法贡献有两层：一是 benchmark 和执行平台，二是 LLVM-Ens。LLVM-Ens 不直接训练新 agent，而是利用不同 LLM/agent 生成互补 patch：先用 LLVM-Gym 过滤无法应用或无法构建的 patch，再做等价归并，最后由 LLM-based ensemble agent 选择最有希望的候选。

实验与 benchmark：论文评估 4 个 LLM、6 种 retrieval configuration、3 个 agent framework，共 36 个实验对象。正文报告 retrieval-augmented LLM 在 LLVM-Bench 上 resolution rate 低于 5%，agent-based techniques 低于 11%；SWE-agent、Trae-agent、Live-SWE-agent 单独 resolution 约 2.60%-10.87%，但 agent union 达 36.17%，说明互补性强。LLVM-Ens 最高达到 21.99% resolved。失败分析显示 patch invalidity 和 build failures 是主导错误；错误 patch 中只有 35.77% 能成功 apply。

可迁移价值：对代码执行闭环训练很直接。LLVM-Gym 式平台能把 patch application、build、test 变成可验证反馈；ensemble 也提示不同 agent 的 patch 分布互补，可以作为 test-time scaling 或 RL candidate selection 的信号。

证据来源：arXiv PDF，阅读 abstract、benchmark/platform、experiments、failure analysis、ensemble、implications。

### 10. Are Performance-Optimization Benchmarks Reliably Measuring Coding Agents?

这篇不是提出新 coding agent，而是审计性能优化 benchmark 是否真的可靠。GSO、SWE-Perf、SWE-fficiency 用真实仓库 patch 后的运行时间来评估 agent 优化能力，但 runtime 本身受机器、噪声、评分规则和 reference patch 稳定性影响。如果 benchmark 信号不稳，leaderboard 可能夸大或误导 agent 进展。

论文重放 740 个官方 reference patches，在 4 类 Google Cloud 机器上测试。核心发现很尖锐：多数任务可执行，但 reference patch 在跨机器 replay 中始终满足原 benchmark validity rules 的只有 39/102 个 GSO、11/140 个 SWE-Perf、411/498 个 SWE-fficiency；SWE-Perf 特别脆弱，因为很多 reference patch 接近零 runtime change。对于 GSO 和 SWE-fficiency 共享的 8 个公开提交，两个 leaderboard 的官方排名在 28 个 pairwise comparison 中有 9 个不一致；SWE-fficiency 最差 10 个任务可占 58.5%-82.8% score weight。跨 10 个公开提交看，至少一个提交在 85.3% 的 replay-valid GSO/SWE-fficiency 任务上达到或超过 reference patch，并在 99.8% 任务上超过未优化 baseline。

可迁移价值：这篇适合放在 Code Agent 评测方法 related work。它提醒我们性能优化 agent 不能只报告 leaderboard，需要报告 reference replay validity、per-task score contribution、runtime margin、机器敏感性和可重复执行结果。

证据来源：arXiv PDF，阅读 abstract、benchmark audit、scoring analysis、public submission analysis、discussion。

## 今日重点趋势

- Search Agent 正在从“检索器 + 生成器”变成“显式计划 + 可执行工作流 + 可验证环境”。PaperPilot 和 PlanRAG 都把搜索计划结构化，BaRA 把网页采集拆成 provenance gates。
- Code Agent benchmark 开始更重视评测可信度。RepoRescue 加 source-only audit 和 runtime blocking，LLVM-Bench 加 build/test 平台，性能优化审计则直接挑战 leaderboard 信号。
- General Agent 的外部状态正在供应链化、环境化。Libra 训练仓库 catalogs，MemSyco-Bench 评估记忆使用边界，Skill Supply Chains 审计 skill 依赖图。

## 潜在研究 idea

- 把 PaperPilot 的 workflow induction 和 PlanRAG 的 logical query tree 合并：用统一 DAG 表示文献搜索、网页搜索和代码定位，再用 reward 评估每个节点是否带来可验证证据。
- 参考 Libra 做“可训练 workspace”：让 agent 每次定位失败后更新仓库目录、API map、测试索引和错误模式库，然后评估这些 artifact 是否跨模型迁移。
- 为 Browser Agent 建一个 BaRA 风格的 provenance reward：链接 live、页面来源、下载验证、引用位置、可复现截图都作为奖励子项。
- 为长期记忆 agent 加“memory arbitration head”：训练模型在回答前判断当前记忆应忽略、限域、更新、冲突解决还是个性化使用。
- 在 Code Agent 训练里加入 source-only audit 和 runtime-blocked policy，减少 agent 通过改测试或改 harness 获得虚假通过的捷径。
