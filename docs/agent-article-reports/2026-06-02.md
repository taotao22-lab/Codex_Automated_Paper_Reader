## General Agent

### 1. HLL: Can Agents Cross Humanity's Last Line of Verification?
- 链接：[arXiv](https://arxiv.org/abs/2606.02449) | [PDF](https://arxiv.org/pdf/2606.02449)
- 证据来源：PDF
- 核心方法：把 CAPTCHA 从“识别题”改写成 `GUI agent` 的闭环评测，要求模型在网页环境里完成定位、点击、状态追踪和过程一致性验证，而不是只交最终答案。
- 创新点：一是把 `trace-conditioned validation` 放进评测，专门区分“答对了”与“动作轨迹真的成立”；二是加入 cluttered webpages、harder variants 这类现实噪声，逼近真实服务的自动化防线。
- 关键发现：论文结论很明确，当前前沿多模态代理在这个人机边界上仍然脆弱，性能会随验证码类型、界面复杂度和轨迹验证要求急剧下降。
- 可转化实验：可以把 HLL 的思路迁到 browser agent / computer-use post-training，额外记录 `final answer correctness`、`action trace validity`、`state recovery` 三类 reward。
- 局限：这是一个高价值 benchmark，不是训练方法论文；而且对象是 CAPTCHA 边界，不直接覆盖普通 WebArena 式任务。

### 2. ACRL: Adaptive Control of Training-Inference Discrepancy for Stable Reinforcement Learning
- 链接：[OpenReview](https://openreview.net/forum?id=8sBZSgAkAH) | [PDF](https://openreview.net/pdf/7da20f22f50e7e2cc577af346ad2abdfce36600e.pdf)
- 证据来源：PDF
- 核心方法：显式监控 RL 训练引擎和推理/rollout 引擎之间的 discrepancy，并用自适应控制器把偏差维持在可训练区间，而不是等训练崩坏后再靠重要性采样补救。
- 创新点：论文把 `训练-推理引擎分离 + 推理侧低精度量化` 当成一类系统问题来处理，这比只改 RL 损失更接近真实 LLM post-training 栈。
- 关键发现：作者报告在 FP8 推理条件下，ACRL 能稳定 RL 训练，同时保持接近 BF16 基线的精度，并优于文中比较的 IS 修复。
- 可转化实验：对 tool-use / code-agent rollout 增加 `policy discrepancy -> reward variance -> success rate` 联动监控，验证能否减少长轨迹训练时的坍塌。
- 局限：实验仍偏数学/推理任务，还没有直接证明它会改善复杂交互式 Agent 的工具使用。

### 3. WorldPack: Dynamic Frame Compression for Long-context Video World Modeling
- 链接：[OpenReview](https://openreview.net/forum?id=zJuiG3PiNJ) | [PDF](https://openreview.net/pdf/cce708ca8b3a98fe9dc8e57519799dd89a636ce2.pdf)
- 证据来源：PDF
- 核心方法：提出 spatially-aware compressed memory，不按时间邻近度统一压缩历史，而是根据 3D spatial relevance 动态决定哪些帧低压缩、哪些帧高压缩。
- 创新点：把 `trajectory packing` 和 `geometric selection` 绑定起来，同时扩大保留历史长度和保持与当前视角最相关的记忆精度。
- 关键发现：论文报告有效上下文从 4 帧扩到 22 帧，推理开销只增加约 16%，并在 LoopNav 与 RECON 上持续优于 Oasis、Mineworld、DIAMOND、NWM，尤其在需要回忆远处观察的空间推理任务上更明显。
- 可转化实验：这非常适合迁到 Agent 长上下文系统里，尤其是 browser trajectory / GUI screenshot memory / code navigation history 的压缩策略。
- 局限：对象是视频 world model，不是文本 Agent；但“按空间/任务相关性分配上下文预算”的思想非常可迁移。

### 4. Distilling Neuro-Symbolic Programs into 3D Multi-modal LLMs
- 链接：[arXiv](https://arxiv.org/abs/2606.01215) | [PDF](https://arxiv.org/pdf/2606.01215)
- 证据来源：PDF
- 核心方法：APEIRIA 通过三阶段课程，把 neuro-symbolic program traces 蒸馏到 3D MLLM 里：先做 3D perception alignment，再做 CoT-SFT 教分解与逐步验证，最后做 CoT-RL 扩展到开放概念和更复杂指令。
- 创新点：重点不是蒸馏具体知识，而是蒸馏 `reasoning pattern`，因此既保留解释性，又把闭集 symbolic learner 的结构迁到开放词汇 MLLM。
- 关键发现：论文在 ScanRefer、Multi3DRefer、SQA3D、Scan2Cap 上评估，结论是优于之前的 NS3D 方法，并能匹配强 3D MLLM。
- 可转化实验：对通用 Agent 来说，这篇最有价值的是“先从可执行/可验证程序轨迹蒸馏推理形状，再用 RL 放宽到开放场景”这一训练路线。
- 局限：场景仍是 3D 空间推理，离搜索和代码闭环还有一层映射。

### 5. Quality-Diversity Evolution for Discovering Diverse Vulnerabilities in LLM Safety
- 链接：[arXiv](https://arxiv.org/abs/2606.00801) | [PDF](https://arxiv.org/pdf/2606.00801)
- 证据来源：PDF
- 核心方法：不是在 token 空间做不可解释的梯度攻击，而是在语义层进化攻击策略，用 MAP-Elites 在 `strategy type / encoding method / prompt length` 行为空间里维持多样化攻击档案。
- 创新点：把红队从“找一个最强攻击”改成“覆盖更多失效模式”，而且输出的攻击仍可解释，适合安全团队复盘。
- 关键发现：不同模型暴露出明显不同的脆弱组合，例如 GPT-4o-mini、Gemini 2.0 Flash、Claude 3.5 Sonnet 的弱点模式并不一样；论文结论之一是 refusal style 也会影响进化攻击的爬升难度。
- 可转化实验：可直接迁到 agent safety 和 tool-use red teaming，把行为维度扩展为 `tool misuse / citation fabrication / browser escalation / code execution bypass`。
- 局限：目前还是单轮或有限轮的安全测试，不是完整 agent-level environment。

## Search Agent

### 6. TimeLogic Challenge @ CVPR 2026: Strong MLLMs Meet Evidence-Seeking Agents for Temporal-Logic Video Question Answering
- 链接：[arXiv](https://arxiv.org/abs/2606.01631) | [PDF](https://arxiv.org/pdf/2606.01631)
- 证据来源：PDF
- 核心方法：把 temporal-logic VideoQA 视作主动搜证问题，代理按照 `Think-Act-Observe` 循环，用 global overview、temporal scan、segment skim、dense focus 等多粒度采样工具去找关键证据帧。
- 创新点：每个 observation 都带绝对时间戳，使 before/after/until/overlap 等关系能被还原成共享时间轴上的数值比较；并按问题类别路由到不同 prompt、budget 和迭代深度。
- 关键发现：这是一个 training-free 系统，论文报告用 Gemini 3.1 Pro 作为骨干，在官方测试集上达到 77.13 AvgAcc。
- 可转化实验：Search Agent 可以直接借鉴它的 `粗搜 -> 聚焦 -> 验证` 采样结构，把帧换成网页段落、文档 chunk 或搜索结果页。
- 局限：目标是视频时序推理，不是网页搜索；迁移价值主要在证据搜集策略。

### 7. ODTQA-FoRe: An Open-Domain Tabular Question Answering Dataset for Future Data Forecasting and Reasoning
- 链接：[arXiv](https://arxiv.org/abs/2606.02433) | [PDF](https://arxiv.org/pdf/2606.02433)
- 证据来源：PDF
- 核心方法：提出一个带未来预测的 tabular QA 任务，并用 TimeFore 把系统拆成 `Retriever -> Forecaster -> Analyzer` 三个协作角色：先生成 SQL 取历史表，再调用时序模型，再综合成标准化回答。
- 创新点：它不是让单个 LLM 硬做未来数值预测，而是把检索、预测和答案合成拆开，属于比较典型的 agentic decomposition。
- 关键发现：论文把数据集和方法一起给出，强调传统 open-domain QA 与未来导向预测之间的能力缺口，并报告 TimeFore 有效。
- 可转化实验：对 Search Agent 很有启发的一点是，把“查询规划”和“外部专用模型调用”作为一等公民，而不是都塞回同一个 LLM prompt。
- 局限：数据域是房地产表格，泛化到开放网页搜索仍需验证；实验摘要里也没有给出很细的跨模型消融数字。

## Code Agent

### 8. Type-Error Ablation and AI Coding Agents
- 链接：[arXiv](https://arxiv.org/abs/2606.01522) | [PDF](https://arxiv.org/pdf/2606.01522)
- 证据来源：PDF
- 核心方法：构造单一类型错误程序集，系统比较四种反馈信号对 AI coding agent 修复能力的影响：完整 unification stack、近邻错误位置、最小 type error、以及仅动态测试失败。
- 创新点：这篇不再把编译器报错默认视为给人看的产物，而是把它当作 `agent-facing interface` 重新评估，问题设定非常对路。
- 关键发现：论文结论很清楚，给代理更多、更详细的类型错误上下文，修复成功率更高；而且静态类型系统提供的结构化反馈通常比单纯测试失败更有帮助。
- 可转化实验：可以把 IDE / 编译器 / linter / static analyzer 的错误输出专门重新设计成 agent 友好格式，再比较 patch 质量和修复轮次。
- 局限：实验语言是 Shplait，规模相对受控；仓库级多文件 issue resolution 还没有覆盖。

### 9. AI-PROPELLER: Warehouse-Scale Interprocedural Code Layout Optimization with AlphaEvolve
- 链接：[arXiv](https://arxiv.org/abs/2606.00131) | [PDF](https://arxiv.org/pdf/2606.00131)
- 证据来源：PDF
- 核心方法：借助 Magellan，把 AlphaEvolve 用于演化 Propeller 的 interprocedural code layout heuristic，再用黑盒调参器调超参，奖励直接来自真实硬件执行计数器，而不是静态成本模型。
- 创新点：真正有价值的是 `生成多个变体 -> 上真机跑 -> 用低方差性能计数器做 reward` 的闭环，这和高质量 Code Agent 的“提出补丁 -> 执行 -> 测量 -> 再修”结构几乎同构。
- 关键发现：论文报告在多个 large warehouse-scale 应用上，相对现有 FDO/PLO 额外提升 0.23% 到 1.6%，并把运行方差压到 0.05% 以下，让进化学习能稳定工作。
- 可转化实验：可以把这套范式迁到性能优化型 code agent，尤其是 benchmark-driven patch search，而不是只看是否通过单测。
- 局限：这是编译/二进制布局优化问题，任务面比一般软件修复更窄。

### 10. SEMBridge: Tagless-Final Program Semantics with Weakest-Precondition and Bounded-Checking Interpretations
- 链接：[arXiv](https://arxiv.org/abs/2606.00220) | [PDF](https://arxiv.org/pdf/2606.00220)
- 证据来源：PDF
- 核心方法：用 tagless-final EDSL 让同一份 object program 同时生成可读代码、具体执行、weakest precondition、bounded counterexample search 等多种解释结果，减少“实现”和“语义工件”漂移。
- 创新点：它不是又做一个大而全验证器，而是强调一种轻量的软件工程结构，把 formal methods artifact 和工程代码绑在同一可执行接口上。
- 关键发现：论文在 5 个例子上展示，同一程序定义可以稳定地产生 state transformer 和 verification condition，并在最多 729 个状态域上通过 bounded checking。
- 可转化实验：如果做代码代理的 verifier，这篇很适合用来设计 `代码补丁 -> 语义约束 -> 有界反例搜索` 的共享中间层。
- 局限：原型语言是 loop-free imperative core，离真实仓库语言和复杂副作用还比较远。

## 今日重点趋势

1. 今天最强的论文几乎都在强调“结构化反馈优于单次生成”。`Type-Error Ablation` 依赖更详细的错误上下文，`AI-PROPELLER` 依赖真实执行计数器，`HLL` 依赖可验证动作轨迹，`ACRL` 依赖训练-推理差异反馈。
2. `memory / context budgeting` 是第二条主线。`WorldPack` 用空间相关性压缩历史，`TimeLogic` 用多粒度搜证预算找关键帧，这两篇都在说明“不是上下文越大越好，而是预算分配越对越好”。
3. Search Agent 直系供给偏弱，但“搜证式推理”在扩张。今天最有价值的不是传统网页检索论文，而是把检索、验证、规划拆成显式步骤的方法。

## 潜在研究 Idea

1. 设计一个 `agent-facing diagnostics benchmark`：同一个任务，系统比较编译器错误、测试失败、静态分析、执行 trace、工具日志等不同反馈组合对修复代理的影响。
2. 把 `WorldPack` 的相关性压缩思路迁到 browser/code agent memory：不是 summarization 一刀切，而是根据后续子任务概率分配保真度。
3. 复用 `HLL` 的 trace-conditioned validation 思路，为 browser/code/search agent 构建“答案正确但过程无效”的反作弊评测。
4. 把 `AI-PROPELLER` 的真实执行 reward 和 `ACRL` 的 discrepancy control 结合，做更稳定的 agentic RL 或 patch-search 训练闭环。
