## General Agent

### 1. Autonomous Incident Resolution at Hyperscale: An Agentic AI Architecture for Network Operations
- 链接：[arXiv](https://arxiv.org/abs/2606.09122) | [PDF](https://arxiv.org/pdf/2606.09122)
- 证据来源：PDF
- 核心方法：把网络事故处置拆成 `Intake / Planning / Execution / Verification` 四层，并进一步落成分工明确的多智能体角色；工具调用通过标准化协议接到运行手册、拓扑、历史记录和执行系统，外层再包一层渐进式自治与闭环验证。
- 创新点：这篇最值得看的不是“多智能体”四个字，而是它把生产事故自动化需要的关键约束都放进架构里了：分层授权、回滚、验证、知识编码、人工接管门槛、自治等级提升路径。它给的是一套可部署的 Agent operating model，而不是 demo 级 workflow。
- 关键结果：论文报告在大云厂商生产网络中，常见事故类别的自治解决率超过 90%；对熟悉故障模式，平均修复时间相对人工流程提升了两个数量级；自动回滚只触发在小比例执行尝试上，且都能恢复。
- 可转化实验/模块：可以直接迁移到通用 Agent post-training：把工具调用顺序、验证器、回滚策略、授权边界和“何时升级人工”都编码成可学习决策对象，再用任务成功率、成本和安全事件做 reward。
- 局限：覆盖面仍以高频、已知故障为主；跨域事故联动、多系统状态一致性、合规审计仍然是瓶颈。
- 阅读建议：本批优先级最高，尤其适合拿来设计你自己的“真实工具环境中的安全 Agent 闭环”。

### 2. Understanding How Enterprises Adopt the Model Context Protocol for LLM-Driven Software Engineering
- 链接：[arXiv](https://arxiv.org/abs/2606.09182) | [PDF](https://arxiv.org/pdf/2606.09182)
- 证据来源：PDF
- 核心方法：不是提出新协议，而是对 8 家公司、20 位从业者做半结构化访谈，系统梳理企业里 MCP 的价值、部署阻力、安全担忧和期待功能。
- 创新点：过去 MCP 讨论多数停留在“怎么接工具”，这篇补的是工程现实层面的证据。它把生态碎片化、跨组件协调、分布式状态管理、故障诊断这些真正会卡住 Agent 落地的问题明确说透。
- 关键结果：文中给出 `100%` 受访者认为 MCP 对工作关键，`80%` 认为它显著提升 LLM 连接外部工具的效率，`75%` 强调其对跨系统协作和知识复用的重要性；同时也指出互联网和金融场景对风险关注点不同，后者更看重审计、访问控制与合规。
- 可转化实验/模块：很适合拿来定义 Agent infra 研究问题，而不是只做 prompt。比如：跨 server 状态同步、工具失败定位、低代码 server 组装、协议层 observability、权限与审计抽象。
- 局限：样本量不大，且主要来自互联网/金融企业；它更像工程研究和需求梳理，不是可直接复现的 benchmark。
- 阅读建议：如果你在考虑 Agent system design、MCP-based tool-use、企业可运维性，这篇应该进 related work。

## Search Agent

### 3. When Should Queries Be Decomposed? A Stage-Aware Study of Query Decomposition for Multi-Condition Retrieval
- 链接：[arXiv](https://arxiv.org/abs/2606.08577) | [PDF](https://arxiv.org/pdf/2606.08577)
- 证据来源：PDF
- 核心方法：论文直接挑战“复杂查询先拆再搜”这个默认直觉。作者发现 decomposition 在初始召回阶段会因为 semantic dilution 拉低 recall，但在 reranking 阶段能提升细粒度条件验证，因此提出保留原查询做初检索、只在重排阶段用子查询的 stage-aware 框架。
- 创新点：贡献不只是一个新 reranker，而是把 query decomposition 放回 pipeline 位置上看，指出它不是“是否使用”的问题，而是“在哪一层使用”的问题。
- 关键结果：在 MultiConIR 和 SSRB 上，作者报告 stage-aware 方案能稳定优于把 decomposition 用在全流程的做法；PDF 结论部分明确指出，初始检索阶段的分解会造成严重召回失败，而 reranking 里的分解能更好过滤“几乎相关但不满足所有条件”的干扰项。
- 可转化实验/模块：对 Search Agent 很直接。可以把“规划子查询”从 first-hop retrieval 挪到 evidence verification / reranking / citation checking 阶段，减少 agent 过早把查询拆碎导致的证据丢失。
- 局限：重点是多条件检索，不是开放网页浏览；迁移到浏览器 agent 时，还要补网页跳转、证据串联和引用可信度建模。
- 阅读建议：如果你在做 search trajectories、query planning 或 RAG reranking，这篇是当天最值得复现的小而硬结果。

### 4. ToolRec: Calibrated Preference Alignment for Query Recommendation in On-Device Assistants
- 链接：[arXiv](https://arxiv.org/abs/2606.08466) | [PDF](https://arxiv.org/pdf/2606.08466)
- 证据来源：PDF
- 核心方法：面向端侧助手的 query recommendation，不再把用户意图当普通聊天请求，而是围绕“触发系统工具”的可执行查询做对齐。作者先构建含 708 个系统工具的 `SysToolKit`，再通过双层 calibration 清洗点击日志，最后用 sample-weighted KTO 对齐模型。
- 创新点：论文抓住了一个很实用的 agent 问题：当最终目标是触发工具时，偏好数据不应把所有点击一视同仁，而应放大真正具有执行价值的查询。
- 关键结果：在线 A/B 测试运行在 OPPO 小布，覆盖超过 1.5 亿月活；作者报告 CTR 和总点击量都优于强基线，同时保持查询相关性。
- 可转化实验/模块：适合迁移到 Search Agent / Assistant Agent 的“tool-invoking query generation”任务，把轨迹数据按执行价值和用户噪声重新加权，而不是只做离线 top-k 推荐优化。
- 局限：核心任务是 query recommendation，不是开放域搜索问答；对引用可信度、网页证据保真帮助有限。
- 阅读建议：偏产业化，但在“搜索入口就是工具入口”的场景里很有启发。

### 5. Decoupling Semantics and Logic: A Training-Free Coarse-to-Fine Pipeline for Video Retrieval-Augmented Generation
- 链接：[arXiv](https://arxiv.org/abs/2606.07924) | [PDF](https://arxiv.org/pdf/2606.07924)
- 证据来源：PDF
- 核心方法：提出一个 training-free 的两阶段 Video RAG。第一阶段用高召回语义预取，只保留高保真视觉摘要和全局描述；第二阶段引入 `A.I.R.` 推理过滤 agent，把 OCR / ASR 等噪声模态重新纳回上下文做逻辑级 rerank；最后再用 Prompt Sculpting 约束生成 JSON 和 chunk-level citation。
- 创新点：它明确把“语义相似”与“逻辑相关”解耦。这个抽象对 Search Agent 很有价值，因为很多 agent 检索失败不是没召回，而是无法从语义接近的候选里剔除逻辑上不满足约束的伪证据。
- 关键结果：实验在 MAGMaR 2026 Full RAG track 上进行，背景库大约 11 万个多语言事件视频，检索评估用 nDCG/Recall，生成评估用 MiRAGE + CLUE。文中主张该流程在 retrieval precision 和 persona-constrained generation 上都更稳。
- 可转化实验/模块：可以把同样结构迁移到网页 Search Agent：先保持召回宽度，再让 reasoning-based verifier 在候选页上做 persona/constraint/citation 级过滤。
- 局限：任务是视频 RAG，不是网页搜索；论文更像系统描述，结果粒度与可复现性不如纯 IR 论文细。
- 阅读建议：适合借鉴架构，不建议把其 leaderboard 结果直接当通用 Search Agent 结论。

## Code Agent

### 6. Empirical Study for Structured Output Control in LLMs for Software Engineering
- 链接：[arXiv](https://arxiv.org/abs/2606.09395) | [PDF](https://arxiv.org/pdf/2606.09395)
- 证据来源：PDF
- 核心方法：系统评估 LLM 在 4 类软件工程任务中的结构化输出可靠性，把失败分成 syntax、structural、semantic 三类，再比较 grammar-constrained decoding、regex validation 和模板严格匹配生成 `TTMG`。
- 创新点：这篇很对 Code Agent 痛点。很多代码 agent 不是“答案错”，而是“输出结构不合法导致工具链直接崩”。论文把这个问题从 prompt 技巧提升成了一个独立评测维度。
- 关键结果：作者指出 `TTMG` 几乎消除了 syntax error，但 structural 与 semantic error 仍然明显存在；这说明仅靠约束解码还不足以保证真实工具链可用性，残余错误会继续向下游工作流级联。
- 可转化实验/模块：可以直接改你自己的 Code Agent evaluator，把“能否通过 schema / AST / patch contract / tool invocation contract”单独记分，而不是只看最终任务成功。
- 局限：论文主要做经验评估与错误分类，不提供一个已经解决全部结构问题的新模型。
- 阅读建议：本批第二优先级。对任何依赖 JSON、patch、函数调用、workflow artifact 的 Code Agent 都很关键。

### 7. FASE: Fast Adaptive Semantic Entropy for Code Quality
- 链接：[arXiv](https://arxiv.org/abs/2606.09800) | [PDF](https://arxiv.org/pdf/2606.09800)
- 证据来源：PDF
- 核心方法：用 embedding 空间里的 pairwise distance matrix 和 minimum spanning tree 近似代码语义熵，替代昂贵的 LLM-based equivalence check，再用自适应聚类估计功能等价类。
- 创新点：它把“多样本代码候选到底有多不确定”这个常见 Code Agent 问题变成了低成本图结构估计，而不是继续堆更多 judge-model 调用。
- 关键结果：在 HumanEval 和 BigCodeBench 上，相对基于 LLM entailment 的语义熵，作者报告 Spearman 相关性平均提升 25%，ROCAUC 提升 19%，代价只有传统方法大约 0.3% 的运行成本。
- 可转化实验/模块：非常适合接进 agentic coding workflow 的 self-check、sample selection、retry gating、patch risk estimation。
- 局限：它估计的是不确定性，不直接修复错误；表现依赖 embedding 模型质量与聚类稳定性。
- 阅读建议：如果你关心“什么时候该继续采样/测试/回滚”，这篇比很多纯 codegen 论文更有立即可用价值。

### 8. LongRTL: Graph-Similarity-Guided LLM-driven Long Context RTL Optimization
- 链接：[arXiv](https://arxiv.org/abs/2606.08944) | [PDF](https://arxiv.org/pdf/2606.08944)
- 证据来源：PDF
- 核心方法：将长上下文 RTL 优化拆成三个 agent：`Partition Agent` 用 AST 图相似度切分子树，`Optimization Agent` 用 AST+RTL 的 multimodal RAG 生成子模块优化方案，`Reconstruction Agent` 再按逻辑关系重组，保证全局功能等价。
- 创新点：真正有价值的是“先结构化再优化”的路线，而不是让一个大模型硬吃上千行 RTL。这个思路对 repository-level code agent 同样成立。
- 关键结果：作者声称在长上下文 benchmark 上做到 100% functional equivalence，并带来显著 PPA 改进；消融实验显示，如果去掉基于图相似度的 partition，functional pass rate 会下降 44.4%。
- 可转化实验/模块：可以借来做大型仓库 patching 的分区策略，把 AST/CFG/调用图切块、模板检索、局部生成、全局重组做成 agent pipeline。
- 局限：任务域是硬件 RTL 优化，不是通用软件仓库；图模板和重组约束对传统代码仓库未必同样容易构建。
- 阅读建议：如果你关心长上下文代码修改如何避开“整仓喂给模型”的蛮力方案，这篇值得细看。

### 9. AgentCompile: An LLM-Guided Compiler for Direct CUDA Inference
- 链接：[arXiv](https://arxiv.org/abs/2606.07665) | [PDF](https://arxiv.org/pdf/2606.07665)
- 证据来源：PDF
- 核心方法：把 LLM 放在编译器里的“语义决策点”而不是最终执行点。LLM 只提供 candidate priority、参数提示和风险注释，真正的 kernel 生成、约束检查、验证、benchmark 和 fallback 仍由编译器决定。
- 创新点：这篇体现了一个很成熟的 Agent 设计原则：让模型负责语义启发，让 deterministic system 负责可执行性和验收。
- 关键结果：在端到端自回归推理中，相比 PyTorch eager，论文报告在 Qwen3-1.7B / Qwen3-4B / Llama-3.2-1B-Instruct 上分别达到 `5.66x / 4.05x / 4.26x` 加速。
- 可转化实验/模块：对于 Code Agent，这篇最重要的不是 CUDA，而是“bounded candidate space + deterministic acceptance + empirical validation + fallback”的执行契约。
- 局限：更偏编译器/系统优化；和软件工程 issue resolution 的直接距离比结构化输出、patch 验证要远一些。
- 阅读建议：适合放进“LLM 只做建议，不直接掌控执行”的 safe agent design 参考列表。

### 10. Toward Compiler World Models: Learning Latent Dynamics for Efficient Tensor Program Search
- 链接：[arXiv](https://arxiv.org/abs/2606.09312) | [PDF](https://arxiv.org/pdf/2606.09312)
- 证据来源：PDF
- 核心方法：把调度评估从“看静态代码快照”改成“看动作驱动的程序状态演化”，用 TensorIR encoder + latent state predictor + ranking cost model 在连续潜空间里 rollout schedule trajectory。
- 创新点：它把 world model 思路放进编译器搜索，很像把 Agent 的“动作后果预估”迁移到程序优化轨迹上。
- 关键结果：在 TVM AutoScheduler 中，作者报告在相同 64 次试验预算下，相比 Ansor 在 GPU/CPU 上分别带来 `1.37x / 1.54x` 代表子图时延改进，并能用 10 倍更少的测量逼近大预算基线。
- 可转化实验/模块：可以把相同思想迁到 Code Agent 的 patch search：不是只看某个补丁静态文本，而是显式建模“编辑轨迹导致的仓库状态变化”。
- 局限：编译器 schedule search 与通用仓库修复差异较大；world model 训练本身也需要高质量轨迹数据。
- 阅读建议：偏研究前沿，短期工程可用性不如前四篇，但长期启发很强。

## 今日重点趋势

1. 生产级 Agent 正在从“会调工具”转向“有清晰自治等级、验证层、回滚层和审计边界”。`Autonomous Incident Resolution` 是这一趋势最强证据。
2. Search Agent 里最有价值的进展不是更大模型，而是把检索流程拆细：什么时候保留全局语义，什么时候再做条件级验证。`Stage-Aware Decomposition` 和 `Decoupling Semantics and Logic` 都属于这个方向。
3. Code Agent 的关键瓶颈仍然是“输出契约”和“风险估计”，不是单纯多采样。`Structured Output Control` 和 `FASE` 分别覆盖了这两个缺口。
4. 长上下文代码问题正在从 brute-force context stuffing 转向结构切分、轨迹建模和 bounded search。`LongRTL`、`AgentCompile`、`Compiler World Models` 都是同一条线上的不同落点。

## 潜在研究 Idea

1. 做一个 Agent workflow evaluator，把 `结构合法性`、`工具调用契约满足度`、`验证器通过率`、`回滚率` 分开计分，参考 `Structured Output Control` 与 `Autonomous Incident Resolution`。
2. 为 Search Agent 设计“双阶段查询策略”：第一阶段只保 recall，第二阶段才让 agent 拆查询、做 constraint verification 和 citation pruning。
3. 在 Code Agent 中引入低成本 uncertainty gating：先用类似 `FASE` 的 embedding-based entropy 决定是否继续采样、补测或要求人工复核。
4. 把 `LongRTL` 的结构分区 + 重组范式迁移到 repository-level patching：调用图/AST 分区、模板检索、局部补丁、全局回归验证。
