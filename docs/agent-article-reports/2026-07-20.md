## 方向索引

### General Agent

`PCTD` 是今天最重要的 General Agent 论文。它把多轮移动端工具调用中的任务分解看成会影响工具检索排序的中间动作，并指出直接用 Recall/NDCG 做 RL 奖励会鼓励重复分解、堆关键词等 reward hacking 行为。`MLLM-DataEngine` 和 `Evolutionary Algorithm-Guided LLMs for PINN Design` 虽然不是通用 agent 论文，但都展示了“评测-失败归因-生成增量数据/配置-重新训练或执行”的闭环，对 agentic post-training 数据管线很有启发。`RecGPT-V3` 则提供了生产系统中的长期记忆、语义 ID 和低成本 latent reasoning 设计，可迁移到长期用户/任务记忆。

### Search Agent

Search 方向今天以相邻信号为主。`Scientific Claim-Source Retrieval Revisited` 明确比较了翻译、元数据、style transfer、reranking 和 verification-based reranking 对科学 claim 寻源的贡献，最适合迁移到带引用约束的 Search Agent。`What Do Chinese-Language Generative Search Engines Cite and Surface?` 是大规模生成式搜索审计，价值在于提供 citation-level measurement schema。`LLMs Encode Relevance as a Layer-Wise Cross-Lingual Signal` 研究相关性在 LLM hidden state 中的线性可解码性，可作为 reranker/verifier 诊断工具。`Efficient and Effective In-place Graph-based Vector Index Updates` 更偏系统，但对需要长时间运行、持续更新记忆库的检索 agent 有工程价值。

### Code Agent

当天没有强 Code Agent/SWE-bench/仓库级修复论文。`EduGuard` 是代码教育场景的安全 RAG tutor，直接价值不在自动修 bug，而在“检索 grounding + 独立 verifier + 泄题/过度依赖控制”的安全反馈闭环；它可迁移到代码助手的教学/解释/调试建议约束。`Evolutionary Algorithm-Guided LLMs for PINN Design` 也有弱 Code Agent 信号：LLM 不直接写 Python，而是生成结构化、可审计配置，由固定 runtime 执行和评估。

## Top 10

### 1. PCTD: Preference-Guided Counterfactual Task Decomposition for Agent Tool Retrieval

- 类别：General Agent / Tool Retrieval / RL Reward Design
- 链接：[2607.15696](https://arxiv.org/abs/2607.15696)
- 综合评分：4.71

**问题背景与核心方法**：多工具 agent 常把用户指令先分解成原子子任务，再用这些子任务检索候选工具。问题是，如果训练时直接把 Recall@K 或 NDCG@K 当作分解奖励，模型可能学到投机策略：把同一意图重复拆很多遍、堆砌关键词、让检索器更容易命中，而不是真的生成更清晰的执行计划。PCTD 的做法是把“分解本身带来的边际收益”单独估出来：同一个输入、同一个工具库、同一个 ground truth，一支分支用原始 query 检索，另一支分支先分解再检索，然后用两者 NDCG/coverage 差值作为 counterfactual reward。与此同时，它训练一个 process reward model，从完整性、准确性、指代消解、逻辑连贯性和原子性等维度给分解结构打偏好分。最终训练信号同时约束“分解确实改善工具排序”和“分解结构像人类参考答案”。

**创新点**：这篇的关键不是又做一个工具检索 benchmark，而是把 reward hacking 的因果归因讲清楚。传统检索奖励会把 raw-query 关键词、工具先验和分解质量混在一起，PCTD 用 paired counterfactual baseline 切掉浅层相关性，再用偏好奖励防止模型只追求检索分数。这个思路很适合迁移到 General Agent 的 post-training：凡是中间计划、查询改写、工具选择会影响最终指标的任务，都可以构造“有中间动作 vs 无中间动作”的反事实差分奖励。

**实验与 benchmark**：论文使用 ToolRet 和自建 MTDTool。ToolRet 聚合 35 个工具调用数据集，覆盖 Web、Code、Custom 三域，共 44K tools；MTDTool 面向移动端多轮交互，提供 dialogue context、intent trajectories、decomposition results 等细粒度标注，并把测试集分成 in-domain 和 OOD。指标包括 NDCG@K、Coverage@K 和 decomposition quality。论文报告 PCTD 在 ToolRet OOD 的 NDCG@10 达到 39.82，高于 ToolQP 的 37.77；在两套 benchmark 的平均 N@10/C@10 达到 69.64/90.78。消融中去掉 preference reward 或 counterfactual reward 都会下降，说明两条奖励信号都必要。

**可迁移价值**：可直接启发工具调用训练集和 RL 奖励设计：让模型学会“什么时候分解有实际检索增益”，而不是把任务分解当成固定模板。可以把它迁移到 query planning、browser action planning、代码仓库检索和多工具 API retrieval。

**局限**：任务集中在移动端工具检索，真实 agent 的执行失败、环境状态变化和工具副作用没有被完整建模。论文的奖励仍依赖 ground-truth tool set 和参考分解，真实开放任务里需要先解决可验证反馈来源。

**阅读建议**：优先读第 3.2 节 counterfactual reward 和第 4 节实验消融；它们最适合改造成 agent post-training 奖励。

**证据来源**：arXiv PDF，已抽取 abstract、method、experiments、conclusion。

### 2. MLLM-DataEngine: Closing the Loop of Multimodal Instruction Tuning Data Generation

- 类别：General Agent / Closed-loop Data Generation / Post-training
- 链接：[2607.15299](https://arxiv.org/abs/2607.15299)
- 综合评分：4.45

**问题背景与核心方法**：传统 instruction tuning 数据生成常常和评测脱节：先批量造数据，再训练，再去 benchmark 上看结果。MLLM-DataEngine 把这个过程闭环化：先在 SEED-Bench 等 benchmark 上评测当前 MLLM，找出答错的问题形成 bad case pool；再根据错误类型构造 prompt，从 Visual Genome 等图像来源和 OCR 工具中生成针对性增量数据；随后用这些数据微调模型，再进入下一轮评测。直观说，它不是随机补数据，而是把 benchmark failure 当成下一轮数据生成的导航信号。

**创新点**：对 Agent 训练的启发在于“失败驱动的数据引擎”。Agent 也可以在工具调用、搜索、代码执行任务中收集失败轨迹，按失败类型生成下一轮环境、任务或演示数据。论文还比较 direct answer 与 multiple choice 等数据格式，说明数据形态要服务于目标 benchmark，而不是所有任务共用一种格式。

**实验与 benchmark**：论文在 LLaVA-1.5 和 MiniGPT4-v2 上实验，评测包括 SEED-Bench、MMBench Dev、OK-VQA、VizWiz、VSR、GQA 等。正文报告 LLaVA-1.5 在 SEED-Bench 上提升 2.53%，并在传统 VQA benchmark 上也有收益；补充结果展示了多轮 refinement 中各能力项的变化。数据生成使用 Visual Genome 图像、PaddleOCR 结果和 GPT-4 生成问答/选项。

**可迁移价值**：可以把 bad case pool 改造成 agent failure pool：失败网页任务、失败工具调用、失败代码测试、错误引用等都能作为下一轮任务合成条件。尤其适合做“评测集驱动的训练数据增量生成”。

**局限**：论文主要在多模态 QA，不涉及真实工具副作用和长程环境交互；生成数据仍依赖强模型和已有 benchmark，可能过拟合 benchmark 失败分布。

**阅读建议**：重点读数据引擎流程图、bad case pool 构造和数据格式消融；不用深挖多模态细节。

**证据来源**：arXiv PDF，已抽取 abstract、method、experiments、appendix results。

### 3. Evolutionary Algorithm-Guided LLMs for Physics-Informed Neural Network Design

- 类别：General Agent / Executable Search Loop / Scientific Agent
- 链接：[2607.15560](https://arxiv.org/abs/2607.15560)
- 综合评分：4.21

**问题背景与核心方法**：PINN 配置设计涉及网络结构、激活、损失权重、采样、优化器和约束处理等组合，LLM 可以提出建议，但一次性建议不会从已训练结果中积累经验。论文把 LLM 放进进化算法框架：每代保留已评估配置、父代、指标、lineage、失败变化和当前最好 MSE；LLM 作为 mutation/crossover operator 生成下一代结构化 AlgorithmSpec；固定 PyTorch runtime 执行这些配置并返回真实训练指标。作者明确限制 LLM 不直接生成 Python，而是生成可验证 schema，由 runtime 执行。

**创新点**：它把“LLM 会不会提出好点子”改成“LLM 能不能在可执行评测闭环里逐代改进”。这对 Code/Tool Agent 很重要：让模型输出结构化、受 runtime 约束的动作，而不是自由写代码；让数值结果、失败配置和 lineage 进入下一轮上下文，而不是只靠自然语言反思。

**实验与 benchmark**：实验在 Wave-C benchmark 上进行，两次独立 10 代搜索，共训练 60 个 PINN、600,000 optimizer steps。两次 run 的最佳配置都出现在最后一代；相对第 0 代最好配置，best MSE 分别降低 2.97% 和 95.38%。论文还指出某些配置能降低数据误差但 PDE residual 仍高，说明单一指标会误导搜索。

**可迁移价值**：适合迁移到代码 agent 的 repair loop 或工具 agent 的 policy search：LLM 只提交 schema/action spec，runtime 负责执行，评测指标负责选择，memory 记录成功和失败变化。

**局限**：实验规模小且集中在单一 PDE；不是通用 agent benchmark。数值收益两次 run 差异很大，说明稳定性还需要更多种子和任务验证。

**阅读建议**：读第 3 节架构、cross-generation memory 和 “No generated Python” 设计。

**证据来源**：arXiv PDF，已抽取 abstract、method、experiments、conclusion。

### 4. EduGuard: A Safe RAG-Based LLM Tutor for Programming Education

- 类别：Code Agent / RAG Tutor / Verification
- 链接：[2607.15738](https://arxiv.org/abs/2607.15738)
- 综合评分：3.97

**问题背景与核心方法**：编程教育中的 LLM tutor 既要解释代码和调试，又不能幻觉、违反课程政策或直接泄露完整作业答案。EduGuard 把 tutoring 拆成多个安全模块：query understanding 判断请求类型和风险；hybrid FAISS/BM25 从教师批准材料中检索；pedagogical strategy selection 控制提示方式；Llama-3.1-8B 生成反馈；DeBERTa-v3-large-MNLI 作为独立 verifier 做 claim-level grounding 检查；overreliance control 检测直接答案泄露和被动依赖。

**创新点**：它不是普通 RAG，而是把“答案正确”“证据 grounded”“不泄题”“引导学生思考”四个目标放到同一条推理链中。对 Code Agent 的可迁移点是：当 agent 给代码解释、调试建议或学习型反馈时，可以用独立 verifier 和 policy-aware leakage check 约束输出，不让模型直接给不可验证或不合规答案。

**实验与 benchmark**：作者构造 BILearn-CS，600 个由教师编写、TA 验证的编程支持问题，覆盖概念题、调试、误解、作业支持、Bangla-English code-mixed query 和 adversarial direct-answer prompts；另有 150 个 CS50-style public course-forum 问题和 10 名本科生小规模 pilot。BILearn-CS 上 EduGuard 的 correctness 为 90.1%，grounding 89.4%，rubric alignment 90.8%，hallucination 4.9%，direct-answer leakage 9.8%；CS50-Forum 上 correctness 86.7%，hallucination 6.8%，leakage 8.9%；pilot 中 post-test accuracy 从 GPT-4o-mini Tutor 的 74.2% 对比提升到 EduGuard 的 81.2%。

**可迁移价值**：适合做代码助手的安全输出层、教学模式、rubric-aware generation 和独立 verifier baseline。也可借鉴其 benchmark 构造方式，为代码解释/调试帮助构造泄露与过度依赖测试集。

**局限**：教育场景强，和自动 issue resolution 有距离；pilot 只有 10 人，课程政策标签依赖具体课程语境。

**阅读建议**：读 system architecture、BILearn-CS 构造和 leakage/hallucination 指标。

**证据来源**：arXiv PDF，已抽取 abstract、architecture、results、limitations。

### 5. LLMs Encode Relevance as a Layer-Wise Cross-Lingual Signal

- 类别：Search Agent / Reranker Diagnostics / Representation Analysis
- 链接：[2607.15555](https://arxiv.org/abs/2607.15555)
- 综合评分：3.88

**问题背景与核心方法**：LLM 已经常被当作 relevance judge 或 reranker，但多数研究只看最终输出，不知道模型内部是否真的表征了 query-document relevance。论文从 instruction-tuned LLM 的 residual stream 中抽取各层 activations，训练线性 probe 判断 query-document 是否相关，并比较这种内部信号与模型生成式 relevance judgment 的差异。它还关注跨语言场景，研究相关性信号是否能在不同语言间迁移。

**创新点**：对 Search Agent 来说，它提供了一种诊断工具：不要只看 reranker 输出分数，也可以检查模型内部在什么层形成了 relevance signal。未来可以用它做工具调用前的置信度估计、reranker 校准或 active retrieval gate。

**实验与 benchmark**：论文 PDF 显示实验围绕 query-document relevance、instruction-tuned LLM activations、linear probe 和 cross-lingual transfer 展开。由于正文表格抽取较碎，本次只确认其评测对象是 relevance decoding 与生成式判断对比；具体数据集和数值结果建议二次打开 PDF 表格核对。

**可迁移价值**：可作为 Search Agent 的 verifier/reranker 可解释性模块，尤其适合分析“模型为什么认为这篇文档相关”。

**局限**：它是表征分析论文，不直接提供可部署的 search pipeline；本次抽取未能稳定核对全部表格数值，因此日报不编造具体数值。

**阅读建议**：适合作为 reranker 诊断 related work，不是优先实现对象。

**证据来源**：arXiv PDF，已抽取 abstract 和方法/实验相关段落；部分表格数值未稳定核验。

### 6. Scientific Claim-Source Retrieval Revisited: A Comparative Study of Style Transfer and Re-Ranking

- 类别：Search Agent / Claim-Source Retrieval / Verification Reranking
- 链接：[2607.15875](https://arxiv.org/abs/2607.15875)
- 综合评分：3.88

**问题背景与核心方法**：社交媒体中的科学 claim 常常和源论文标题/摘要风格差异很大，直接检索容易找不到真正来源。论文系统比较多种处理：把多语言 claim 翻译成英语；给 source representation 加入 publication metadata；把 claim 改写成 formal/scientific/abstract/question 风格；用 cross-encoder 或 generative reranker 重排；最后再用 attribution、entity overlap、verification-based reasoning 等结构化信号重排 top candidates。

**创新点**：最值得迁移的是 verification-based reranking。它不只问“文本像不像”，而是让模型考虑 source verification、evidence alignment、context specificity 和 conflicting findings，把候选源是否真正支撑 claim 作为重排信号。这和 Search Agent 的引用可信度、事实核验、答案可追溯性高度相关。

**实验与 benchmark**：实验使用 CheckThat! 2026 Source Retrieval for Scientific Web Claims，指标为 MRR@5。翻译 claim 对所有检索器都有帮助，E5 平均 MRR@5 提升 +0.167，GTR +0.125，GritLM +0.041。加入 publication metadata 对 BM25 提升最大，平均 MRR@5 从 0.354 到 0.382。GritLM + Qwen3-8B reranker 将平均 MRR@5 从 0.633 提到 0.733；verification-based reranking 进一步提升到 0.758，增益 +0.025。

**可迁移价值**：可直接转化为 Search Agent 的 citation verifier：先 broad retrieval，再 rerank，再让 verifier 判断“这条引用是否真正支撑答案中的原子 claim”。

**局限**：任务是 scientific claim-source retrieval，不是开放网页浏览；verification step 使用大模型，成本和延迟需要控制。

**阅读建议**：优先读第 4.5 节和 Table 4/5。

**证据来源**：arXiv PDF，已抽取 method、tables、discussion、conclusion。

### 7. What Do Chinese-Language Generative Search Engines Cite and Surface? A Large-Scale Empirical Study

- 类别：Search Agent / Generative Search Audit / Citation Measurement
- 链接：[2607.15771](https://arxiv.org/abs/2607.15771)
- 综合评分：3.77

**问题背景与核心方法**：生成式搜索把用户注意力从传统排名列表转移到生成答案和引用来源，但中文平台上的 citation 和 entity exposure 仍缺少系统测量。论文对四个主流中文生成式搜索平台的 Web 和 App 接口做控制实验，覆盖 8 个平台接口、614 个 queries、每个 query-platform-interface 三次重复，原始数据 214,119 条，清洗后形成 160,860 条 citation-level master dataset。

**创新点**：它的贡献不是新模型，而是 measurement schema。对 Search Agent 来说，训练和评测不能只看答案对不对，还要看引用来源从哪里来、是否稳定、是否偏向特定域名、同一 query 多次运行是否一致。论文的 citation-level primary key、query layer、platform/interface/run 等字段可以直接借鉴到 agent search trace logging。

**实验与 benchmark**：这是实证审计，不是模型 benchmark。PDF 显示数据框架包括平台、接口、query layer、industry domain、quote_url、quote_index、run、citation-level id 等字段，并分析 cited source 和 generated answer text。它引用既有研究中“生成句子是否被引用支持”“引用是否支持对应句子”等问题，但本日报只采用本论文可核验的数据规模和设计信息。

**可迁移价值**：适合给 Search Agent 建 trace schema 和评价维度：引用覆盖率、来源多样性、跨运行稳定性、平台偏差、citation grounding。

**局限**：偏审计和 GEO，不提供检索/重排训练方法；平台访问和清洗细节可能难复现。

**阅读建议**：适合做 Search Agent evaluation/logging 设计参考。

**证据来源**：arXiv PDF，已抽取 abstract、dataset framework、experimental design。

### 8. RecGPT-V3 Technical Report

- 类别：General Agent / Long-term Memory / Production Multi-agent System
- 链接：[2607.15591](https://arxiv.org/abs/2607.15591)
- 综合评分：3.68

**问题背景与核心方法**：RecGPT 系列把 LLM 用于淘宝推荐中的用户理解和意图推理。V3 主要解决三个生产问题：用户行为建模不能无状态，tag-to-item 存在信息瓶颈，显式 chain-of-thought 在高 QPS 场景成本太高。系统引入 Memory Hub，增量维护用户长期偏好；引入 Semantic ID tokens，让 LLM 能在文本 tag 和离散 item code 之间联合推理；引入 Latent Intent Reasoning，把长解释压缩为低成本隐式推理，必要时再生成解释。

**创新点**：虽然是推荐系统，但它对长期 agent 有两点启发：第一，记忆不是把历史全文塞进上下文，而是增量 curated 的偏好单元；第二，工具/物品空间可以用可学习离散 ID 接入 LLM，突破纯自然语言标签瓶颈。对于长期任务 agent，可以把文件、工具、网页、用户偏好或项目实体编码成类似 SID 的可检索记忆符号。

**实验与 benchmark**：论文报告系统部署在淘宝 “Guess What You Like” feed，与 RecGPT-V2 做大规模在线 A/B test；IPV 提升 +1.28%，CTR 提升 +1.07%，GMV 提升 +1.39%，GPU 消耗降低 52.4%，reasoning token cost 降低约 200x。PDF 还说明 RecGPT-V2 的 global planner 占 agentic intent analysis 约 95% 计算成本。

**可迁移价值**：适合借鉴长期记忆压缩、增量更新、离散实体 token 和 latent reasoning 的工程设计。

**局限**：强领域生产报告，推荐系统目标和工具使用 agent 不同；很多细节依赖淘宝内部数据和基础设施。

**阅读建议**：读 Memory Hub、Hybrid Tokenization、Latent Intent Reasoning 三节。

**证据来源**：arXiv PDF，已抽取 abstract、system overview、memory、tokenization、online A/B results。

### 9. Efficient and Effective In-place Graph-based Vector Index Updates

- 类别：Search Agent / Dynamic Vector Index / Retrieval Infrastructure
- 链接：[2607.15576](https://arxiv.org/abs/2607.15576)
- 综合评分：3.65

**问题背景与核心方法**：长期运行的 RAG/Search Agent 需要不断插入和删除向量，但许多 graph-based ANN index 在动态更新时要么吞吐低，要么搜索质量下降。论文提出 Yi，一个 in-place graph-based vector index update 系统，目标是在大规模替换、插入、删除过程中同时维持更新吞吐、搜索吞吐和 recall。它通过向量级更新机制、任务调度、buffer 管理和局部重连来避免频繁 out-of-place rebuild。

**创新点**：对 Agent 的价值在基础设施层：如果 agent 的 memory store、网页缓存、代码索引或用户知识库持续变化，索引更新不能靠 nightly rebuild。Yi 展示了“在线更新 + 查询不中断 + recall 基本保持”的系统路线。

**实验与 benchmark**：论文在 DEEP100M 上从 100M 初始索引中插入/删除 30M vectors，Yi 更新吞吐约 1.6K QPS，比 OdinANN 高 2.8x；搜索吞吐比 DiskANN、OdinANN、SPFresh 分别最高约 1.7x、1.7x、3.9x，同时 recall 接近 OdinANN 并高于 DiskANN。SIFT800M 上插入/删除 40M vectors，Yi 在 8M 和 20M batch 下平均更新吞吐比 OdinANN 高 2.3x 和 1.5x，搜索吞吐约高 1.8x。

**可迁移价值**：适合关注长期记忆检索、动态代码索引、持续网页抓取的 Agent 系统设计。

**局限**：纯系统论文，不涉及 LLM policy、工具调用训练或 agent evaluation；迁移价值需要通过具体 memory/index 工程体现。

**阅读建议**：需要做大规模动态向量库时读；否则作为 infrastructure related work 即可。

**证据来源**：arXiv PDF，已抽取 method、DEEP100M/SIFT800M results、ablation。

### 10. Adaptive Retrieval Strategies for Biomedical Question Answering

- 类别：Search Agent / Adaptive RAG / Domain QA
- 链接：[2607.15283](https://arxiv.org/abs/2607.15283)
- 综合评分：3.37

**问题背景与核心方法**：不同 QA 类型的信息需求不同：yes/no 需要精确证据，factoid/list 需要实体和候选聚类，summary 需要更广泛证据收集。论文提出面向 BioASQ 的自适应检索框架：query understanding 识别问题类型，PubMed 文档检索用多种关键词和同义词策略扩展候选，再用 jina-reranker-v2-base-multilingual 和 bge-reranker-v2.5-gemma2-lightweight 分阶段重排，最后用 LLM filtering 和问题类型特定的证据聚合生成答案。

**创新点**：它的 Agent 价值在于“检索策略随任务类型切换”。Search Agent 不应对所有问题都用同一个 retriever top-k 和同一个 reranker；应该根据答案形态、召回风险和合成需求切换 evidence acquisition plan。

**实验与 benchmark**：实验基于 BioASQ Task 13B，Phase A 评估 document retrieval 和 snippet extraction，Phase A+ 评估 yes/no、factoid、list、summary 等回答任务。论文报告 Phase A 文档检索和 snippet extraction 四个 batch 均排名第一；list questions 在 Batch 2、3、4 的 recall 排名第一，Batch 4 的 F-measure 排名第一。作者说明 summary questions 依赖人工评估，部分最终成绩仍可能调整。

**可迁移价值**：可改造成 Search Agent 的 retrieval planner：先判定问题类型，再选择 broad/narrow retrieval、entity clustering、freshness preference、reranker depth 和 LLM filtering。

**局限**：领域是 biomedical QA，且系统 heavily tuned for BioASQ；直接迁移到开放网页搜索需要换数据源和 verifier。

**阅读建议**：读 retrieval/reranking pipeline 和 question-type-specific aggregation。

**证据来源**：arXiv PDF，已抽取 abstract、system pipeline、experiments、results。

## 今日重点趋势

1. 工具调用 RL 正在从 outcome metric 转向 causal/process reward。PCTD 的反事实检索增益适合推广到 query planning、task decomposition、tool gating 和代码修复策略选择。
2. 闭环数据生成开始把 benchmark failure 当成训练数据生成器的输入。MLLM-DataEngine 和 PINN 进化搜索都说明，失败归因和可执行反馈比一次性 synthetic data 更重要。
3. Search Agent 的评测需要从“答案正确”扩展到“证据是否可寻源、引用是否支撑、来源是否稳定”。Scientific Claim-Source Retrieval 和中文生成式搜索审计都在补这个层。
4. 长期 agent 系统需要动态记忆基础设施。RecGPT-V3 的增量记忆和 Yi 的动态向量索引分别给出了模型层和系统层方案。

## 潜在研究 idea

- 反事实工具分解奖励：对同一任务比较 raw instruction、LLM decomposition、human decomposition 三种工具检索结果，用差分 NDCG/coverage 训练分解模型，并加入重复子任务惩罚。
- Agent failure data engine：把浏览器 agent、代码 agent、搜索 agent 的失败轨迹按错误类型聚类，自动合成下一轮训练任务和 verifier checks。
- Citation-aware Search Agent：把 claim decomposition、source retrieval、verification reranking 和 citation-level logging 串成闭环，奖励不仅看答案，还看每个原子 claim 是否被可靠来源支撑。
- Runtime-constrained code/search agent：让模型输出 schema 或 action spec，由固定 runtime 执行、验证、回传指标，减少自由生成代码带来的不可控错误。
