## General Agent

### 1. Less Context, More Accuracy: A Bi-Temporal Memory Engine for LLM Agents Where a Lean Retrieved Context Beats the Full History
- 链接：[arXiv](https://arxiv.org/abs/2606.09900) | [PDF](https://arxiv.org/pdf/2606.09900)
- 证据来源：PDF
- 核心方法：提出 `Engram`，把 agent memory 做成双时态系统。写入侧保留 lossless episode；异步整理侧提取 `(subject, predicate, object)` 事实、建立 bi-temporal knowledge graph，并用“失效而非删除”的方式保留 provenance 与 supersession chain；读取侧把 dense、lexical、graph、recency/salience 混合检索，再做 point-in-time 过滤，最后拼成紧凑上下文。
- 创新点：论文最有价值的是明确反对“全历史回放一定更准”。作者证明真正有效的是高保真检索切片，而不是更长上下文；同时他们把 memory benchmark 的评测完整性问题单独拎出来处理，给了可复现 harness 和逐题日志。
- 关键结果：在 500 题 `LongMemEval_S` 上，约 `9.6k` token 的检索切片得到 `83.6%`，高于 `79k` token 全历史基线的 `73.2%`，而且 token 成本约低 `8x`；作者还强调 facts-only 读取会掉 recall，必须让 facts 与原始 chunk 联合工作。
- 可转化实验/模块：非常适合迁移到长期工具使用 Agent。你可以直接复用“事实层 + chunk 层 + 时间过滤 + provenance”这套 memory stack，并把“lean retrieved context 是否优于 full history”作为 post-training 的 reward 或 evaluator。
- 局限：主要验证的是长期记忆问答，不是浏览器或代码执行闭环；如果迁移到代码仓库记忆，还要补 repository state 与工具副作用建模。
- 阅读建议：今天最值得细读的 General Agent 论文之一。

### 2. What makes a harness a harness: necessary and sufficient conditions for an agent harness
- 链接：[arXiv](https://arxiv.org/abs/2606.10106) | [PDF](https://arxiv.org/pdf/2606.10106)
- 证据来源：PDF
- 核心方法：不是提出新系统，而是做概念工程。论文沿着 `test harness -> eval harness -> agent harness` 的谱系，给出“agent harness”的必要且充分条件，并把它 operationalize 成 inclusion / exclusion test。
- 创新点：它把今天已经被混用的 `framework / SDK / IDE plugin / orchestrator / eval harness` 边界重新画清楚。对于 Code Agent 研究，这种定义工作并不花哨，但能直接减少比较对象混乱、benchmark 口径漂移和系统描述偷换概念的问题。
- 关键结果：作者把这一定义应用到 `Claude Code、Codex CLI、Aider、Cline、OpenHands、SWE-agent` 以及若干边界案例上，声称能够稳定地“该进则进、该排则排”，最后还给出按设计张力组织的研究议程。
- 可转化实验/模块：如果你要设计自己的 agent benchmark 或报告体系，这篇可以直接当术语基线，用来拆清楚“环境层 / agent harness / eval harness / tool protocol”的职责。
- 局限：它是概念论文，不提供新模型、新数据或新训练结果；工程启发强，但实验性弱。
- 阅读建议：适合放进 related work 和术语定义章节，不适合作为“算法突破”来读。

### 3. $\tau$-Rec: A Verifiable Benchmark for Agentic Recommender Systems
- 链接：[arXiv](https://arxiv.org/abs/2606.10156) | [PDF](https://arxiv.org/pdf/2606.10156)
- 证据来源：PDF
- 核心方法：面向多轮 agentic 推荐，把传统 `LLM-as-a-judge` 换成可验证奖励，并设计 `reveal-tagged elicitation` 机制，控制约束在对话中的显露方式，再用 `pass^k` 来度量一致性而不是单次“看起来像对了”。
- 创新点：虽然应用域是推荐，但它真正可迁移的点是“把对话 Agent 评测改成 verifiable reward + constraint reveal curriculum”。这和你关心的 tool-use reasoning、交互轨迹验证很接近。
- 关键结果：论文测试了多个模型家族，最好模型在 `pass^1` 只有约 `57%`，到 `pass^4` 只有约 `35%-38%`，显示多轮 agent 在一致性上还有明显断层。
- 可转化实验/模块：可以借鉴其 `reveal-tagged` 设计，为通用 Agent 做逐步暴露约束的训练/评测任务，而不只是一次性给全量 specification。
- 局限：任务空间仍是目录化推荐，对开放网页搜索和代码执行的迁移需要二次设计。
- 阅读建议：更像“评测范式启发”，不是直接拿来当 Search 或 Code Agent 的 SOTA。

## Search Agent

### 4. SkillResolve-Bench: Measuring and Resolving Same-Capability Ambiguity in Agent Skill Retrieval
- 链接：[arXiv](https://arxiv.org/abs/2606.10388) | [PDF](https://arxiv.org/pdf/2606.10388)
- 证据来源：PDF
- 核心方法：论文关注 skill retrieval 里更细的失败模式：检索器命中了正确能力族，却把 query-specific 的“危险兄弟技能”一并暴露出来。作者据此构建 `SkillResolve-Bench`，并提出先解析 active candidate family、再用 utility / contract-profile cues 在族内选代表的方法。
- 创新点：它把 skill retrieval 从“找相关文档”提升成“控制执行风险的程序暴露”。这个角度很对 Agent：错的不只是信息，而是会把 stale resource、错误前提或错误步骤直接带入执行链。
- 关键结果：在 661 组 helpful / risky sibling 上，`SkillResolve` 做到 `Recall@3 = 0.766`、`NDCG@3 = 0.699`，同时把 `HSR@3` 压到 `0`；而没有 representative selection 时，危险兄弟暴露率会明显上升。
- 可转化实验/模块：对 tool library、skill router、MCP server 选择都很有用。你可以直接把 `harmful sibling rate` 作为工具检索层的新指标，而不是只看 Recall。
- 局限：任务输入是 skill library 元数据，不是开放网页；迁移到 Web search 时还要把网页证据可靠性也纳入风险建模。
- 阅读建议：今天 Search Agent 方向最强的一篇，尤其适合“检索错一步，执行全错”的场景。

### 5. Effective Reinforcement Learning for Agentic Search by Recycling Zero-Variance Queries During Training
- 链接：[arXiv](https://arxiv.org/abs/2606.10709) | [PDF](https://arxiv.org/pdf/2606.10709)
- 证据来源：PDF
- 核心方法：针对 outcome-only reward 下常见的 GRPO 训练，论文指出 all-correct / all-incorrect rollout group 会变成零方差样本，白白浪费 rollout。作者提出 `query recycling`，把这些零方差 query 放回可变样本池，让它们在策略变化后重新成为有效学习信号。
- 创新点：价值不在“多收集数据”，而在“训练分布跟着策略共同演化”。这很贴合 agentic search 的真实问题，因为 query 的难度不是静态的，而是 policy-relative。
- 关键结果：一个 `1.7B` 模型只用 synthetic data，就在 7 个 multi-hop QA benchmark 上做到平均 `Pass@1 = 66.0`，可对标甚至超过一些用 benchmark-derived supervision 训练的更大模型；扩到 `4B` 时结果进一步到 `71.3`。
- 可转化实验/模块：你可以把这一招直接迁到 search trajectory RL、tool routing RL，甚至 code agent 的 outcome-only patch training 里，把当前“没梯度的任务”保留到后续阶段再学。
- 局限：论文任务还是 multi-hop QA 风格的 search agent，不是浏览器级真实网页环境；对复杂交互与 citation verification 的帮助需要额外验证。
- 阅读建议：方法简单但很实用，优先级很高。

### 6. ConvMemory v2: A Recall-Preserving Top-10 Evidence Reranker for Conversational Memory Retrieval
- 链接：[arXiv](https://arxiv.org/abs/2606.10842) | [PDF](https://arxiv.org/pdf/2606.10842)
- 证据来源：PDF
- 核心方法：在已有轻量 reranker 的 protected top-10 上再接一个 cross-encoder，只重排而不改召回集合，因此 `Recall@10` 和 `Hit@10` 被“结构性地”锁定不变，优化目标集中在 top-10 内部排序质量。
- 创新点：论文强调的是检索系统 contract design。它不夸大“全面超越”，而是把自己的贡献限定为 recall-preserving cascade，并用 load-bearing ablation 证明收益来自 candidate-specific memory text。
- 关键结果：在 `LoCoMo` 上把 `MRR` 从 `0.5824` 提到 `0.6560`，`H@1` 从 `0.4440` 提到 `0.5474`；而且与更昂贵的 top-500 cross-encoder 相比只落后很小一截。
- 可转化实验/模块：这篇很适合迁到 Search Agent 的 evidence rerank 或 Agent memory retrieval，尤其是你希望保住召回不动，只在高可信候选集合里做精排时。
- 局限：它优化的是会话记忆检索，不是开放域搜索；top-10 保护策略能否迁到网页证据集合，需要看召回层本身是否足够强。
- 阅读建议：属于“工程契约设计比模型更重要”的好例子。

## Code Agent

### 7. DeNovoSWE: Scaling Long-Horizon Environments for Generating Entire Repositories from Scratch
- 链接：[arXiv](https://arxiv.org/abs/2606.10728) | [PDF](https://arxiv.org/pdf/2606.10728)
- 证据来源：PDF
- 核心方法：为 whole-repository generation 构建大规模可验证训练集 `DeNovoSWE`。数据通过 sandboxed agentic workflow 自动构造，核心策略是 `divide and conquer + critic-repair`，再用 difficulty-aware trajectory filtering 平衡数据质量与难度。
- 创新点：这篇最重要的不是“又一个 SWE 数据集”，而是把长程 Code Agent 的训练对象从 issue-level bug fix 推到 doc-to-repo。它覆盖的是架构搭建、模块组织、文件间一致性和长链路实现，而不是单点修补。
- 关键结果：数据集包含 `4,818` 个 whole-repository 实例；在 `BeyondSWE-Doc2Repo` 上，Qwen3-30B-A3B 微调后从 `5.8%` 提到 `47.2%`。这个提升幅度说明长程 repo 训练数据仍然是主要瓶颈。
- 可转化实验/模块：对你最有价值的是数据构造 pipeline：sandbox、critic-repair、轨迹过滤、长程可验证目标，这些都可以迁到更通用的 code-execution 闭环训练。
- 局限：doc-to-repo 与 issue resolution 仍有任务差异；如果做真实生产仓库修复，还要补历史状态、测试回归和增量编辑约束。
- 阅读建议：今天 Code Agent 第一优先级。

### 8. TestMap: Evidence Infrastructure for Foundation-Model-Assisted Test Generation
- 链接：[arXiv](https://arxiv.org/abs/2606.10211) | [PDF](https://arxiv.org/pdf/2606.10211)
- 证据来源：PDF
- 核心方法：把 test generation 看成“生成 + 验证 + 修复 + 归档证据”的基础设施问题。`TestMap` 自动串联 repository analysis、source-test mapping、baseline execution、coverage、mutation、test smell、repair 和 experiment tracking，并保留失败、中间修复和低收益样本。
- 创新点：这篇直接补齐了 Code Agent 论文里常被略过的一环：不是只看最终通过的测试，而是记录候选测试的全生命周期，把中间失败也当作可研究证据。
- 关键结果：它主要贡献是开源基础设施和 evidence model，而不是单一 benchmark 数字；论文明确主张 candidate-level observability 才能让不同模型、提示和策略可比较。
- 可转化实验/模块：很适合迁到代码 agent post-training，把 “生成补丁 / 生成测试 / 运行 / 失败 / 修复 / 再运行” 整条轨迹保存下来做监督或 RL。
- 局限：当前实现聚焦 C#/.NET；跨语言迁移需要重做 runner、mapping 和分析工具链。
- 阅读建议：如果你关心可复现、可审计的 Code Agent 训练与评测，这篇比很多 leaderboard 论文更重要。

### 9. MASTOR: A Multi-Agent Approach to Semantic Test Oracle Generation for RESTful APIs
- 链接：[arXiv](https://arxiv.org/abs/2606.10465) | [PDF](https://arxiv.org/pdf/2606.10465)
- 证据来源：PDF
- 核心方法：围绕 REST API semantic oracle generation，先由 source extraction agent 收集 endpoint 的源码上下文，再并行走 single-operation 与 multi-operation 两条 oracle 生成路径，之后加 challenger agent 审核并触发定向重生，最后做 normalization。
- 创新点：这篇的启发点是把“生成断言”变成 multi-agent critique loop，而不是一次性 prompt 出断言。尤其 multi-operation behavioral consistency 这一路，对状态相关工具调用验证很有迁移价值。
- 关键结果：在 13 个开源 API 项目上，生成 `10,022` 个 oracle，平均 mutation score `75.4%`；在 50 个操作的基线对比里明显优于 direct prompting 与 SATORI。
- 可转化实验/模块：适合迁到 Code Agent 的 validator 设计，尤其是工具调用后状态一致性、跨 API 序列约束和 challenger-style 自我复核。
- 局限：任务域还是 REST API testing，离通用 repo patching 有距离；oracle 的语义质量也仍受源码可读性影响。
- 阅读建议：如果你想做“执行后如何验证结果”的研究，这篇值得保留。

### 10. Multi-task LLMs for Bug Classification: Efficient Inference with Auxiliary Decoding Heads
- 链接：[arXiv](https://arxiv.org/abs/2606.09956) | [PDF](https://arxiv.org/pdf/2606.09956)
- 证据来源：PDF
- 核心方法：用轻量 multi-task LLM 做 line-level bug classification，并通过 token alignment 解决现有方法在线级定位时的 tokenization 对齐问题；推理时整文件只需一次 forward、只生成一个 token。
- 创新点：它对 Code Agent 的意义在于证明“不是所有调试都要 agentic long-chain reasoning”。在某些高频诊断环节，轻量判别模型可以以低成本覆盖长文件上下文，适合做 agent 前置筛查。
- 关键结果：作者声称在 line-level bug localization 上达到同类 setting 的 SOTA，并在 `Defects4J` 与 `PypiBugs` 上以远低于 agentic 方法的推理开销取得可比效果。
- 可转化实验/模块：可作为 code agent 的预检器、bug prior 或 patch search narrowing 模块，先给出高风险行，再把昂贵 agent 预算集中投到局部区域。
- 局限：它是 fault localization，不是完整修复闭环；对跨文件因果错误和执行后验证帮助有限。
- 阅读建议：适合作为 Code Agent 系统中的低成本辅助模型，而非主 Agent 本体。

## 今日重点趋势

1. 长程 Code Agent 的瓶颈越来越明确地落在“可验证训练环境”而不是单纯模型规模。`DeNovoSWE` 和 `TestMap` 都在补环境与轨迹证据。
2. Search / Memory Agent 的价值点在从“召回更多”转向“暴露更安全、排序更稳、上下文更短但更准”。`SkillResolve-Bench`、`Engram` 和 `ConvMemory v2` 是同一趋势的不同切面。
3. Agent 评测正在从主观 judge 转向 verifiable reward、执行风险指标和术语边界清理。`τ-Rec` 与 `What makes a harness a harness` 都属于这条线。
4. 对真实系统来说，轻量辅助模块仍然重要。`Multi-task Bug Classification` 说明判别器、预检器、路由器这类小模块，仍然是 Agent 闭环里很值得训练的角色。

## 潜在研究 Idea

1. 做一个 `agent harness + eval harness + tool protocol` 的分层评测框架，把术语和接口先固定，再测任务成功率。
2. 为 Search Agent 增加 `harmful sibling rate` 或近似指标，专门衡量“检索到了错误但可执行的相邻工具/技能”。
3. 在长期 Agent memory 里加入 `bi-temporal facts + raw chunk` 双通道检索，并把 provenance 直接写进最终上下文。
4. 用 `DeNovoSWE` / `TestMap` 风格的轨迹保存方式，构造 code agent 的补丁-测试-修复全过程训练数据。
